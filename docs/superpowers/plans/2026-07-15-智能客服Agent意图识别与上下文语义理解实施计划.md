# 智能客服 Agent 意图识别与上下文语义理解实施计划

> 编制日期：2026-07-15
> 适用项目：`/Users/qqx/job/code/eladmin-mp`
> 核心范围：用户输入意图识别、上下文语义引用、组合式查询语义、多意图/追问、真实模型评测
> 首要故障样本：`活跃客户数 -> 他们分别还剩多少餐`
> 执行方式：按阶段和复选框推进；每个阶段先补测试，再改实现，最后同步业务/API 文档。

## 1. 背景与结论

当前 Agent 已具备模型优先、规则兜底、受控语义、QueryPlan 编译、工具白名单、权限校验和结果校验。下一阶段的主要短板不在数据查询安全，也不在是否接入大模型，而在“用户一句话究竟想完成什么，以及它与上一轮是什么关系”。

本计划不扩展全渠道、知识库、工单、语音、情绪运营等客服平台能力，只解决以下核心问题：

1. 当前问题是新查询、追问、局部修改、纠错、重试还是集合展开。
2. 本轮是否引用历史会话中的实体、集合、指标结果或查询条件，以及引用哪一个上下文句柄。
3. 用户要对引用对象执行查询、筛选、分组、投影、比较、解释还是条件修改。
4. 一句话包含一个还是多个业务请求，应合并、顺序执行还是澄清。
5. 模型输出的置信度是否真的可信，线上错误如何进入回归集。
6. 识别正确后，系统是否具备完成该语义帧所需的最小受控查询能力。

核心判断：

- 保留现有“LLM 只负责理解，服务端负责规划和授权”的安全架构。
- 禁止实现“某个词直接映射某个行为”的新规则，例如禁止硬编码“他们=上一轮集合”“分别=按客户展开”。
- 将当前单目标 `BusinessQuestionAnalysis` 升级为“单轮会话理解结果 + 一个或多个组合式语义帧”。
- 为历史实体、结果集合、指标和查询条件建立通用上下文句柄；引用解析基于类型兼容、会话焦点、时序和任务状态排序。
- 将用户诉求拆为 `目标对象 + 数据范围 + 度量 + 维度 + 操作 + 输出粒度`，避免为每种说法创建专用意图枚举。
- 首期用“活跃客户数 -> 各客户早餐/午晚餐剩余餐数”验证通用机制，但实现中不出现针对该句式的特殊分支。

## 2. 首要故障样本

### 2.1 实际对话

```text
用户：现在仍有可用餐数的活跃客户数是多少
Agent：当前口径活跃客户数为 15 客户。

用户：他们分别还剩多少餐呢
Agent：请提供客户编号，例如：C10001。
```

附件会话证据显示：

- 第一轮正确识别 `ACTIVE_CUSTOMER_COUNT`，调用 `getActiveCustomerSummary`，结果为 15。
- `lastBusinessQueryContext` 只保存总数、指标和计划指纹，没有保存“这 15 位客户”这一可引用集合。
- 第二轮把“还剩多少餐”落到了单客户 `MEAL_BALANCE` 语义。
- “他们”没有绑定上一轮活跃客户集合。
- 本轮没有被解析出从汇总结果切换到客户维度明细的通用 `PROJECT + GROUP` 操作。
- 当前 `getActiveCustomerSummary` 只返回 `total`，不存在受控的活跃客户余额明细工具。

### 2.2 正确语义：组合式语义帧，而不是句式规则

第一轮完成后，系统应登记一个可引用但不携带客户明细的上下文句柄：

```json
{
  "handleId": "ctx-result-1",
  "kind": "ENTITY_SET",
  "entityType": "CUSTOMER",
  "definitionId": "AGENT_ACTIVE_CUSTOMER_V1",
  "cardinality": 15,
  "sourceTurnId": "turn-1",
  "salience": 1.0,
  "allowedOperations": ["COUNT", "PROJECT", "GROUP", "FILTER"],
  "expiresAt": "2026-07-15T10:27:17+08:00"
}
```

第二轮应被解析为通用语义帧：

```json
{
  "interactionMode": "FOLLOW_UP",
  "referenceTurn": "PREVIOUS_BUSINESS_QUERY",
  "frames": [
    {
      "goal": "QUERY",
      "targetEntity": "CUSTOMER",
      "scope": {
        "type": "CONTEXT_REFERENCE",
        "requiredKind": "ENTITY_SET",
        "requiredEntityType": "CUSTOMER"
      },
      "measures": ["REMAINING_BREAKFAST", "REMAINING_LUNCH_DINNER"],
      "dimensions": ["CUSTOMER"],
      "operations": ["PROJECT", "GROUP"],
      "outputShape": "DETAIL_LIST",
      "constraints": {},
      "missingInformation": []
    }
  ],
  "requiresClarification": false
}
```

模型输出的是“需要什么类型的上下文”，不是可信的 `handleId`。`ContextReferenceResolver` 校验候选后，才把 `ctx-result-1` 绑定到可执行语义帧。

这里“他们”和“分别”只用于帮助模型理解上述结构，服务端不读取这两个词决定行为。下列不同说法应得到相同或等价语义帧：

- `这些客户每个人还有几餐`
- `刚才这批人的餐数明细`
- `把上面统计范围按客户拆开，显示早餐和午晚餐余额`
- `我想看这个统计集合对应的客户余额`

反过来，`他们今天都排餐了吗` 应复用相同集合句柄，但产生不同的 measures、constraints 和 outputShape，而不是命中“他们”的固定处理分支。

### 2.3 正确处理结果

在客服拥有客户及订单查询权限时：

1. 引用解析器在候选上下文句柄中，将本轮范围解析为上一轮 `ENTITY_SET<CUSTOMER>`。
2. 操作解析器将本轮诉求解析为余额度量、客户维度和明细输出。
3. 生成固定 QueryPlan，调用新增的受控明细工具。
4. 返回客户编号/脱敏姓名、剩余早餐数、剩余午晚餐数和合计，不返回手机号、地址或金额。
5. 明细超过上限时分页或截断，并明确说明不是完整列表。

在客服缺少明细权限时：

- 明确提示当前账号只能查看汇总，不能用“找不到客户”掩盖权限问题。

## 6. 设计原则

1. **规则提确定事实，模型理解完整语义。** 客户编号、订单号、标准日期等由规则优先；意图、指代、组合关系由模型处理。
2. **意图和工具分离。** 模型只能选择业务意图、指标、对象范围和交互模式，工具仍由服务端目录映射。
3. **上下文引用必须可证明。** 任何省略、代词或隐式引用都只能绑定已登记的上下文句柄，不能引用模型想象的对象。
4. **上下文保存语义，不保存敏感结果。** 保存指标、范围、集合定义、计划指纹和限长摘要，不保存完整工具响应。
5. **先拒绝错误执行，再追求直接回答率。** 重要歧义不猜测。
6. **置信度以评测校准为准。** 模型自报置信度仅作为特征，不能单独决定执行。
7. **能力缺失要显式反馈。** 识别出正确意图但没有登记工具时返回 `CAPABILITY_NOT_AVAILABLE`，不能退化成错误意图。
8. **禁止表面词直接驱动执行。** 规则层可以把代词、省略、量词等作为“需要上下文理解”的信号，但不得把具体词映射为 scope、dimension、tool 或 QueryPlan。
9. **语义组合优先于意图爆炸。** 不新增 `ACTIVE_CUSTOMER_MEAL_BALANCE_LIST`、`EXPIRING_ORDER_CUSTOMER_LIST` 这类排列组合式意图；使用目标、范围、度量、维度和操作组合表达。

## 7. 目标架构

```text
当前消息 + 确定性槽位 + 最近受控会话摘要 + 当前任务栈
    |
    v
Deterministic Guard
  - RESET / 明确重试 / 安全拒绝
  - 客户、订单、日期、餐次提取
    |
    v
Context Reference Resolver
  - 从会话状态读取 ENTITY / ENTITY_SET / METRIC_RESULT / FILTER_SET / TASK 句柄
  - 按类型兼容、会话焦点、时序、任务关联和权限计算候选分数
  - 候选分差不足时澄清，不靠代词词典直接决定引用
    |
    v
Conversation Understanding
  - interactionMode
  - frames[]
  - target / scope / measures / dimensions
  - operations / constraints / outputShape
  - ambiguities / missingSlots
    |
    v
Understanding Validator + Confidence Policy
  - schema/枚举/领域/权限可见目录
  - 多意图组合规则
  - 置信度校准/拒绝/澄清
    |
    v
MultiIntent Planning
  - 每个 frame 经过能力目录验证后编译固定 QueryPlan
  - 校验依赖、顺序、工具和数据预算
    |
    v
Controlled Execution
  - 单意图执行
  - 同源组合执行
  - 集合明细批量执行
    |

```


新增受控类型：

- `SemanticScope`
  - `EXPLICIT`：本轮显式实体或条件。
  - `SESSION_FOCUS`：当前会话焦点。
  - `CONTEXT_REFERENCE`：绑定通用上下文句柄。

- `SemanticOperation`：`COUNT / LIST / FILTER / PROJECT / GROUP / COMPARE / AGGREGATE / EXPLAIN / REFINE`。
- `SemanticOutputShape`：`SCALAR / SUMMARY / DETAIL / DETAIL_LIST / GROUPED_RESULT / COMPARISON`。
- `SemanticMeasure` 复用或映射到版本化指标目录，例如 `ACTIVE_CUSTOMER_COUNT / REMAINING_BREAKFAST / REMAINING_LUNCH_DINNER`。

这些字段是正交组合关系。新增一个业务能力通常只登记“允许哪些组合”，而不是再新增一个包含对象、指标和展示方式的长意图名称。

## 10. 上下文指代与任务状态

### 10.1 引入通用 `ConversationContextHandle`

建议新增领域对象，而不是继续为每种追问向 `LastBusinessQueryContext` 增加专用字段：

```java
private String handleId;
private ContextHandleKind kind;
private SemanticEntityType entityType;
private String definitionId;
private String sourceTurnId;
private String sourceRequestId;
private Integer cardinality;
private Map<String, Object> safeDescriptor;
private List<SemanticOperation> allowedOperations;
private double salience;
private OffsetDateTime createdAt;
private OffsetDateTime expiresAt;
```

`ContextHandleKind` 支持：

- `ENTITY`：一个客户、订单、排餐记录等。
- `ENTITY_SET`：一个可由服务端重新计算的对象集合。
- `METRIC_RESULT`：某项指标及其口径。
- `FILTER_SET`：日期、餐次、状态等条件组合。
- `TASK`：当前或最近任务。

本故障第一轮建议同时登记 `METRIC_RESULT` 和 `ENTITY_SET<CUSTOMER>` 两个相关句柄：

```json
{
  "handleId": "ctx-result-1",
  "kind": "ENTITY_SET",
  "entityType": "CUSTOMER",
  "definitionId": "AGENT_ACTIVE_CUSTOMER_V1",
  "cardinality": 15,
  "safeDescriptor": {"metric": "ACTIVE_CUSTOMER_COUNT"},
  "allowedOperations": ["COUNT", "PROJECT", "GROUP", "FILTER"],
  "salience": 1.0
}
```

注意：`definitionId` 是可重新计算的受控定义，不保存 15 个客户 ID。后续“这些人今天排餐了吗”“其中午餐还有余额的有几位”都应引用同一个句柄，再通过不同语义操作形成计划。

### 10.2 新增任务栈

建议新增：

- `agent-service/.../query/domain/ConversationTaskStack.java`
- `agent-service/.../query/domain/ConversationTask.java`
- `agent-service/.../query/domain/ConversationTaskStatus.java`

首期最多保存 5 个任务：

- `ACTIVE`：当前等待补槽或执行的任务。
- `SUSPENDED`：被相关插问暂时挂起。
- `COMPLETED`：已完成，只保留受控摘要。
- `CANCELLED`：用户切换目标或清空。
- `FAILED`：稳定失败码，不保存异常原文。

任务栈必须支持：

- 纯槽位补充继续当前任务。
- “他们/这些客户”引用上一个完成任务的结果集合。
- 新的无关业务请求取消或挂起旧任务。
- 相关插问处理完成后恢复旧任务。
- RESET 清空任务栈、Pending 和 Last Context。

## 11. 会话理解流程改造

### 11.1 第一层：确定性 Guard

保留快速路径：

- 清空/重置。
- 明确相同条件重试。
- 金额、写操作等安全拒绝。
- 系统正在追问时的单一日期、餐次、客户/订单编号。

调整：

- [ ] `isPureSlotReply` 从整句正则改为结构化槽位差异判断。
- [ ] “明天晚餐”“B3303 午餐”等多个纯槽位组合也应续接当前任务。
- [ ] 只要出现新业务谓词，再进入会话理解，不直接覆盖旧语义。
- [ ] 代词、省略、量词和分配表达只作为“需要上下文理解”的低成本信号，不在规则层决定 scope、dimension 或 operation。

### 11.2 第二层：上下文引用候选

新增 `ContextReferenceResolver`，对所有活跃句柄做通用候选排序，不直接执行。候选评分至少考虑：

- 类型兼容：当前 frame 需要客户集合时，`ENTITY_SET<CUSTOMER>` 高于单客户或订单集合。
- 会话焦点：当前任务和最近完成任务产生的句柄优先。
- 时序距离：更近的有效句柄优先，但不能覆盖类型冲突。
- 语义一致：当前度量和操作是否能作用于该句柄。
- 显式修正：用户明确切换对象时降低旧句柄显著度。
- 权限与有效期：过期、无权限或定义已失效的句柄不进入候选。

模型负责结合自然语言判断本轮是否存在引用以及需要何种类型，Resolver 负责从服务端已知句柄中选择和校验。任何具体词都不能绕过候选排序和类型校验。

候选字段包含来源和确定性：

```json
{
  "requiredKind": "ENTITY_SET",
  "requiredEntityType": "CUSTOMER",
  "candidateHandleId": "ctx-result-1",
  "sourceRequestId": "...",
  "score": 0.94,
  "scoreReasons": ["TYPE_COMPATIBLE", "CURRENT_FOCUS", "MOST_RECENT"]
}
```

### 11.3 第三层：LLM 会话理解

新增 `LlmConversationUnderstandingService`：

输入仅包括：

- 当前消息。
- 当前确定性槽位。
- 最近最多 6 轮脱敏对话摘要。
- Pending 受控语义。
- Last Context 受控结果摘要。
- 当前任务栈和可引用句柄摘要。
- 按权限裁剪后的语义能力目录。

输出严格使用 `ConversationUnderstandingResult` Schema。

Prompt 必须明确：

- 任何上下文引用都必须声明所需句柄类型，再由服务端绑定具体句柄。
- 用户的查询、筛选、投影、分组、比较和解释要求必须输出为正交操作，不得靠新增组合意图表达。
- 多请求必须拆分为 `frames[]`，不能丢掉第二个请求。
- 没有可引用集合时必须澄清“你指哪些客户”。
- 能识别但系统未开放的语义组合应原样输出，由能力校验器返回不支持，不能改猜别的目标。

### 11.4 第四层：结果校验

新增 `ConversationUnderstandingValidator`：

- [ ] 根字段和嵌套字段白名单。
- [ ] `frames` 数量 1～3。
- [ ] 目标、范围、操作、领域、度量、维度和输出形状均为受控值。
- [ ] `CONTEXT_REFERENCE` 必须绑定有效 Context Handle。
- [ ] 句柄类型必须与目标、度量和操作兼容。
- [ ] `GROUP/PROJECT + CUSTOMER dimension` 必须作用于客户实体或客户集合。
- [ ] 多请求依赖不能成环。
- [ ] 缺关键条件时不得生成可执行计划。
- [ ] 模型不得通过自由说明携带工具、字段或敏感信息。

## 12. 多意图和集合计划编译

### 12.1 新增 `MultiIntentPlanningService`

职责：

1. 将每个 `SemanticRequestFrame` 与语义能力目录做结构匹配。

3. 校验计划之间的数据依赖和执行顺序。

## 14. 置信度与澄清策略

### 14.1 不再直接信任模型自报分数

新增 `IntentConfidencePolicy`，综合：

- 模型 confidence。
- 是否命中有效目录。
- 规则/确定性槽位是否冲突。
- 指代是否存在唯一可用上下文。
- Top-1/Top-2 候选差距。
- 该意图在离线评测中的校准准确率。
- 是否涉及跨客户明细。

输出：

- `HIGH`：可执行。
- `MEDIUM`：无风险查询可执行，集合/敏感明细需要确认。
- `LOW`：必须澄清。
- `REJECTED`：非法、未支持或安全冲突。

### 14.2 固定澄清码

- `CONTEXT_REFERENCE_MISSING`：本轮存在隐式引用，但没有兼容的上下文句柄。
- `CONTEXT_REFERENCE_AMBIGUOUS`：存在多个分数接近的兼容句柄。
- `INTENT_CONFLICT`：同一句存在冲突业务目标。
- `DETAIL_PERMISSION_REQUIRED`：汇总可见但明细无权。
- `CAPABILITY_NOT_AVAILABLE`：意图识别成功但未登记执行能力。
- `METRIC_AMBIGUOUS`：数量口径不明确。
- `MISSING_REQUIRED_SLOT`：缺日期、餐次、客户等条件。

澄清问题由服务端模板生成，模型只输出澄清码和受控选项，不允许生成可能泄露对象存在性的提示。

## 19. 测试矩阵

### 19.1 单轮

- `B3303 还剩多少餐` -> 单客户余额。
- `现在活跃客户有多少` -> 活跃客户汇总。
- `把所有活跃客户的余额列出来` -> 授权集合余额明细。
- `今天还有多少客户` -> 口径歧义澄清。
- `查客户金额` -> 安全拒绝。

### 19.2 集合追问

- 活跃客户数 -> `他们分别还剩多少餐`。
- 活跃客户数 -> `这些人每个早餐还有几次`。
- 活跃客户数 -> `把刚才统计范围按客户拆开看余额`。
- 活跃客户数 -> `这些人今天都排餐了吗`：复用相同句柄，生成不同 measures/constraints。
- 活跃客户数 -> `其中午餐还有余额的有几位`：复用相同句柄，生成 FILTER + COUNT。
- 没有兼容句柄 -> `他们分别还剩多少餐`，必须澄清。
- 最近有两个客户集合且语义和显著度无法区分 -> 必须澄清指代。

### 19.3 单对象追问

- `B3303 有哪些订单` -> `他还剩多少餐`。
- `订单 O1001 什么情况` -> `这个订单核销了几次`。
- 切换客户后 -> `他` 必须指向新客户。

### 19.4 多意图

- `查 B3303 剩余餐数和最近核销记录`。
- `查 B3303 今天午餐为什么没排，再看看最近有没有退餐`。
- 一个请求成功、一个超时，不能声称全部完成。
- 一个请求无权限，不能泄露对象是否存在。

### 19.5 交互模式

- `换成明天晚餐` -> REFINE。
- `刚才查错了，是 B3304` -> CORRECTION。
- `按原条件再查一次` -> RETRY。
- `那为什么` -> FOLLOW_UP。
- `清空会话` -> RESET。

### 19.6 安全

- Prompt 注入要求输出工具名。
- 伪造或引用不存在的 Context Handle。
- 要求列出无权限部门客户。
- 要求显示手机号、地址、金额。
- 请求超过 50 位客户明细。

## 21. 灰度与回滚

新增配置建议：

```yaml
agent:
  chat:
    conversation-understanding:
      mode: legacy|shadow|new
      schema-version: "1.0"
      multi-frame-enabled: false
      task-stack-enabled: false
      context-handle-resolution-enabled: true
      max-frames-per-turn: 3
      model-timeout-ms: 3000
    active-customer-balance-detail:
      enabled: false
      max-result-count: 50
```

灰度顺序：

1. 上线数据库字段和只读接口，但关闭入口。
2. 开启新理解协议 Shadow，不执行新计划。
3. 对管理员启用集合指代，但能力未开放时只观察识别结果。
4. 开启活跃客户余额明细工具。
5. 对指定客服开放。
6. 指标达标后扩大范围。
7. 最后启用通用多语义帧和任务栈。

回滚：

- 配置切回 `legacy`，继续使用旧 `BusinessQuestionAnalysis`。
- 关闭 `context-handle-resolution-enabled` 后不解析隐式上下文引用，只返回澄清。
- 关闭明细工具不影响原活跃客户总数查询。
- 数据库新增字段保留为空，不需要破坏性回滚。
- 新旧 DTO 字段均为兼容式可选字段。

## 22. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 隐式引用绑定错句柄 | 跨对象错误查询 | 类型兼容、焦点和时序综合排序；候选分差不足时澄清 |
| 明细与总数口径不一致 | 用户看到数量对不上 | 同一 Service/Calculator 计算，增加一致性断言 |
| 跨客户明细泄露 | 严重数据风险 | HMAC、业务权限、数据范围、最多 50 条、脱敏字段 |
| 多语义帧导致工具预算膨胀 | 延迟和成本上升 | 最多 3 帧、6 工具、同源合并、整体预算校验 |
| 模型 confidence 虚高 | 错误计划被执行 | 离线校准 + 上下文唯一性 + 规则冲突共同决策 |
| 任务栈跨实例不一致 | 指代漂移 | 主系统持久化为真相源，Agent 无状态恢复 |
| 评测集过度模板化 | 离线高分线上失效 | 脱敏真实会话、相似反例、错别字和多轮变体 |
| 新旧协议并存复杂 | 维护成本上升 | 明确兼容期，迁移完成后删除旧细粒度分支 |
