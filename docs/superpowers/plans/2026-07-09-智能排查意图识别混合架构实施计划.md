# 智能排查意图识别混合架构实施计划

> **For agentic workers:** 优先按阶段推进，每完成一个阶段后先运行对应测试并记录实际偏差。本文档使用 `- [ ]` 复选框追踪进度。

**Goal:** 将当前以规则为主的聊天输入理解，重构为“规则抽槽位 + 混合意图分类 + 会话策略编排”架构，解决多轮对话中代词、省略、同义表达导致的误判问题，避免继续堆叠正则。

**Scope:** 本次改造仅聚焦 `agent-service` 聊天理解层及其测试，尽量不改 `eladmin-system` 与 `eladmin-web` 的接口契约；如响应字段需补充，采用兼容式新增，不做破坏性修改。

**Non-Goals:**
- 不在首版引入“全量 LLM 槽位抽取”
- 不让 LLM 直接执行诊断、查库或生成业务结论
- 不改现有客户查询、核销查询、订单查询、排餐诊断的数据接口

**Architecture:** 将当前 [RuleBasedMealPlanChatExtractor.java](/Users/qqx/job/code/eladmin-mp/agent-service/src/main/java/me/zhengjie/agent/chat/RuleBasedMealPlanChatExtractor.java) 的职责拆分为 4 层：
- `RuleBasedSlotExtractor`：稳定提取 `customerCode/customerId/recordDate/mealType/orderStatus`
- `RuleBasedIntentClassifier`：处理 reset/retry/out_of_scope 和少量高确定性显式意图
- `LlmIntentClassifier`：处理多轮、省略、代词、追问、变体表达
- `HybridMealPlanChatExtractor`：统一编排，输出现有 `ChatExtractionResult`

**Tech Stack:** Java 17, Spring Boot 3.5, Spring AI, JUnit 5, Maven Surefire.

## 二、目标设计

### 2.1 意图边界

- [ ] **Step 1: 固化首版支持的意图集合**
  - `DIAGNOSE`
  - `FOLLOW_UP`
  - `RETRY`
  - `RESET`
  - `OUT_OF_SCOPE`
  - `CUSTOMER_MEAL_BALANCE_QUERY`
  - `CUSTOMER_VERIFICATION_QUERY`
  - `CUSTOMER_ORDER_QUERY`

- [ ] **Step 2: 固化每个意图的最小槽位要求**
  - `CUSTOMER_*`：客户即可
  - `DIAGNOSE`：客户 + 日期 + 餐次
  - `FOLLOW_UP`：依赖最近诊断结果，通常不要求重新补齐全部槽位
  - `RESET/RETRY/OUT_OF_SCOPE`：不要求业务槽位

### 2.2 路由策略

- [ ] **Step 3: 定义混合识别总原则**
  - 结构化槽位优先由规则提取
  - 规则能高置信度判意图时直接返回
  - 规则不确定、多轮上下文、省略指代时调用 LLM 分类
  - LLM 输出必须经枚举校验和置信度校验
  - LLM 失败或低置信度时可回退规则或触发澄清

## 五、混合路由策略

### 5.1 规则直判场景

- [ ] **Step 1: 明确无需 LLM 的输入**
  - `清空会话`
  - `重新排查`
  - `B2201 核销了多少餐`
  - `B2201 有哪些进行中订单`
  - `查 C10001 今天午餐`

### 5.2 升级到 LLM 的场景

- [ ] **Step 2: 明确必须调用 LLM 的输入**
  - 使用代词且依赖上下文：`他一共多少餐`
  - 槽位已继承，但问法不在规则白名单
  - 同句出现多个业务信号，规则难以判定主意图
  - follow-up 与局部改查边界模糊

### 5.3 回退策略

- [ ] **Step 3: 定义失败回退机制**
  - LLM 超时：回退规则判定
  - LLM 非法 JSON：回退规则判定
  - LLM 输出未知意图：回退规则判定
  - LLM 低置信度：进入澄清提问或回退规则

### 5.4 开关与配置

- [x] **Step 4: 新增模式开关**
