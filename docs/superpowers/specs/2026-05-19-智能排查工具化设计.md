# 智能排查数据工具化设计

## 背景

当前智能排查链路中，`agent-service` 会先通过 `eladmin-system` 的内部接口拉取完整诊断上下文，再由 `DiagnosisPromptBuilder` 将客户档案、订单、排餐明细、候选菜统计等业务数据整体放入 LLM prompt。这样能让模型一次性获得充分证据，但也会把客户相关数据默认上传给 LLM。

本次优化目标是改为按需取数：LLM 首轮只接收用户问题、规则和可用工具说明；当模型判断需要证据时，再调用细粒度工具查询对应业务数据。这个方向也为后续前端改造成对话式排查保留接口形态。

## 目标

- 默认诊断请求不再预取或上传完整客户业务上下文。
- 将客户档案、客户订单、排餐详情、候选菜统计拆成独立工具。
- LLM 只能看到它主动调用过的工具返回数据。
- 保持现有前端诊断响应结构不变，降低上线影响。
- 保留现有聚合上下文接口，避免影响已有调用和回退能力。

## 非目标

- 本次不改造成完整多轮对话前端。
- 本次不改变 `DiagnosisResponse` 的对外结构。
- 本次不删除 `/api/internal/agent/meal-plan/context` 聚合接口。
- 本次不引入复杂权限模型；内部接口继续面向 `agent-service` 使用。

## 现状

`eladmin-system` 当前只暴露一个内部聚合接口：

- `POST /api/internal/agent/meal-plan/context`

该接口调用 `AgentDiagnosisContextService.buildContext()`，一次性聚合：

- 客户档案
- 客户订单
- 指定日期餐次的排餐详情
- 指定日期的候选菜统计

底层查询能力已经存在于 `AgentDiagnosisContextServiceImpl` 的私有方法中，但尚未作为独立 service 方法或独立 controller 接口暴露。

## 方案

采用多个细粒度工具。

`agent-service` 首轮诊断不再调用 `DiagnosisContextBuilder.build()` 构建完整上下文，而是构建轻量诊断输入，仅包含：

- `customerId`
- `customerCode`
- `recordDate`
- `mealType`
- 规则版本和规则列表
- 可用工具说明

LLM 需要证据时通过 Spring AI tool calling 调用工具。工具由 `agent-service` 注册，每个工具内部通过 HTTP 调用 `eladmin-system` 的对应内部接口。

## 内部接口设计

在 `eladmin-system` 新增四个内部接口，现有聚合接口保持兼容。

### 查询客户档案

`POST /api/internal/agent/customer-profile`

请求字段：

- `customerId`
- `customerCode`

返回：

- `CustomerProfileDetailDto`
- 未找到时返回 200，body 为 `null`

### 查询客户订单

`POST /api/internal/agent/customer-orders`

请求字段：

- `customerId`
- `customerCode`
- `page`
- `size`

默认分页：

- `page = 1`
- `size = 100`

返回：

- `List<CustomerOrderDetailDto>`

如果只传 `customerCode`，服务端先解析客户档案，再用解析出的 `customerId` 查询订单。

### 查询排餐详情

`POST /api/internal/agent/meal-plan`

请求字段：

- `recordDate`
- `mealType`

返回：

- `MealPlanDetailVO`

没有对应排餐时返回 200，body 为 `null`。

### 查询候选菜统计

`POST /api/internal/agent/candidate-dish-stats`

请求字段：

- `recordDate`

返回：

- `List<MealPackageStatDto>`

## Service 设计

扩展 `AgentDiagnosisContextService`，将当前私有查询能力提升为可复用方法：

- `resolveCustomerProfile(Long customerId, String customerCode)`
- `resolveOrders(Long customerId, String customerCode, Integer page, Integer size)`
- `resolveMealPlan(String recordDate, String mealType)`
- `resolveCandidateDishStats(String recordDate)`

`buildContext()` 继续复用这些方法，保持现有聚合接口行为。

分页和限制：

- 订单查询默认最多返回 100 条。
- `size` 超过 100 时按 100 处理。
- `page` 小于 1 时按 1 处理。

## Agent 工具设计

`agent-service` 新增工具组件并注册到 `ChatClient`：

- `getCustomerProfile`
- `listCustomerOrders`
- `getMealPlan`
- `getCandidateDishStats`

工具入参使用明确 DTO，避免让模型拼接 URL 或自由构造 HTTP 请求。

工具返回值使用现有 DTO 映射成 `Map` 或轻量 DTO，保持与当前 prompt JSON 兼容。工具调用日志记录：

- `requestId`
- 工具名
- 关键入参
- 返回对象是否存在或返回条数
- 耗时

日志不记录完整返回体。

## Prompt 设计

`DiagnosisPromptBuilder` 改为构建轻量 prompt：

- 说明模型是排餐未生成原因诊断助手。
- 明确只能基于请求参数、规则和工具返回证据判断。
- 如果证据不足，必须调用工具或返回证据不足，不允许猜测。
- 不再包含完整业务上下文 JSON。
- 输出仍必须映射到 `DiagnosisResponse`。

## 诊断流程

1. 前端仍调用现有智能排查接口。
2. `eladmin-system` 代理请求到 `agent-service`。
3. `agent-service` 加载规则，构建轻量诊断输入。
4. LLM 判断是否需要证据，并按需调用工具。
5. 工具调用 `eladmin-system` 的细粒度内部接口取数。
6. LLM 基于已调用工具返回的数据输出 `DiagnosisResponse`。
7. `DiagnosisResultValidator` 继续校验和兜底。

## 错误处理

- 单个工具调用失败时，工具返回明确的错误结果，不抛出未处理异常给诊断主流程。
- LLM 如果没有足够证据，应输出需要人工核对的诊断结果。
- ChatClient 调用失败时，继续使用现有 fallback 逻辑。
- `eladmin-system` 内部接口保持当前容错风格：查询失败记录 warn，并返回空结果。

## 测试计划

`eladmin-system`：

- Service 测试覆盖四个独立查询方法。
- Controller 测试覆盖四个新增内部接口。
- 保留并更新现有聚合上下文测试，确认兼容行为不变。

`agent-service`：

- Prompt 测试确认不再包含完整业务上下文 JSON。
- HTTP 工具客户端测试覆盖四类内部接口调用。
- AI client 测试确认工具注册到 ChatClient。
- fallback 测试确认工具失败时仍能返回人工核对提示。

## 上线兼容

- 前端接口不变。
- 对外响应结构不变。
- 老的内部聚合接口保留。
- 可以通过配置开关控制是否启用工具化诊断；必要时回退到现有聚合上下文模式。
