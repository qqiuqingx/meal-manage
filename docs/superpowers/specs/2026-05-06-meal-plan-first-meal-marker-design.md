# 排餐计划首次排餐标记设计

## 背景

排餐计划管理页左侧编号区当前只展示客户编号、无汤红圈和加菜标签。业务希望更快筛查“成单后首次排餐”的客户，尤其是早餐、午餐、晚餐各自第一次成功排餐的场景。现有接口未提供该状态，前端也没有可复用的判定口径。

## 目标

在排餐计划管理页和打印页的左侧编号区，为“订单维度 + 餐次维度”的首次成功排餐客户增加显眼标记，并将这些客户排序到前面，方便业务筛查。

## 非目标

- 不在客户排餐管理弹窗中展示该标记
- 不修改右侧菜品汇总、换菜明细和编号明细的统计口径
- 不新增数据库字段，不将首次标记结果持久化到 `meal_plan_customer`
- 不改变接口默认客户排序，只在左侧编号区做前端展示排序

## 业务口径

### 判定维度

- 首次标记按 `订单 + 餐次` 判定
- 同一客户后续新签订单时，新订单重新计算首次标记
- 早餐、午餐、晚餐相互独立，同一订单最多出现三次首次标记

### 成功口径

- 仅 `meal_plan_customer.status = 1` 的成功排餐参与首次判定
- 失败排餐不占用首次资格
- 如果某订单某餐次第一次排餐失败，后续第一次成功排餐时仍应显示首次标记

### 历史范围

- 只统计有效历史记录：
  - `meal_plan_customer.deleted = 0`
  - `meal_plan.deleted = 0`
- 当前记录为首次的条件：在相同 `order_id + meal_type` 下，不存在更早的成功排餐记录

### 更早记录判定

- 先按 `meal_plan.record_date` 升序比较
- 同一天按 `meal_plan_customer.id` 升序兜底
- 这样可保证同一天重复生成、补生成或指定客户生成时，只有最早的成功记录拿到首次标记

## 接口设计

### 返回字段

在 `MealPlanDetailVO.CustomerPlanDetail` 中新增字段：

- `firstMealOfOrder: Boolean`

语义：

- `true`：当前客户排餐记录是该订单在当前餐次上的首次成功排餐
- `false`：否则

### 计算位置

在排餐详情聚合接口中统一计算并返回该字段，不在前端推导。

涉及链路：

- `GET /api/meal-plan/{id}/detail`
- `MealPlanServiceImpl.queryMealPlanDetail`
- `MealPlanServiceImpl.assembleCustomerDetail`

### 查询策略

后端在加载当前排餐计划的客户列表后，批量计算哪些 `meal_plan_customer.id` 属于首次成功排餐。

建议在 `MealPlanCustomerMapper` 增加批量查询方法，输入当前排餐计划下的客户记录 ID 列表，返回命中的首次记录 ID 集合。

判定 SQL 语义：

1. 先圈定当前页面涉及的 `meal_plan_customer` 记录
2. 对每条记录取出：
   - `mpc.id`
   - `mpc.order_id`
   - `mp.meal_type`
   - `mp.record_date`
   - `mpc.status`
3. 仅对当前成功记录继续判定
4. 判断是否存在同 `order_id + meal_type` 的更早成功记录
5. 不存在则该 `mpc.id` 记为首次

## 前端展示设计

### 页面范围

- 排餐计划管理页：
  - `eladmin-web/src/views/meal/scheduleRecord/index.vue`
- 打印页：
  - `eladmin-web/src/views/meal/productionSheet/index.vue`

### 排序规则

左侧编号区客户列表按以下顺序展示：

1. `firstMealOfOrder = true` 的客户
2. 其余客户

约束：

- 两组内部保持接口原始顺序不变
- 不影响右侧菜品统计和明细逻辑
- 不修改接口返回顺序，排序仅作用于左侧编号区 `allCustomers`

### 标记样式

采用已确认的 `C` 方案：左侧色条 + 右侧徽章，并使用绿色系，避免与现有异常提示冲突。

样式语义：

- 首次标记：绿色系，表示业务识别状态
- 无汤标记：保留现有红色编号红圈，表示异常/缺项提示
- 加菜标签：保留现有黄色标签，表示补菜信息

建议色值：

- 左侧色条：`#16a34a`
- 徽章底色：`#dcfce7`
- 徽章文字：`#166534`
- 边框色：`#86efac`

呈现规则：

- 在 `code-cell` 容器层显示绿色左侧色条和边框强化
- 在编号右侧展示小徽章，文案为 `首`
- 如果客户无 `customerCode`，则对当前姓名同样展示 `首`
- `无汤` 仍挂在编号文本层，不改现有红圈逻辑
- `加菜` 标签仍位于第二行，不和 `首` 混排

### 打印样式

- 打印页同步显示绿色左侧色条和 `首` 徽章
- 颜色和边框要足够深，确保黑白打印也能识别层级
- 不依赖悬浮态或半透明效果传达关键信息

## 边界场景

### 删除与重生成

- 被软删除的历史排餐记录不参与首次判定
- 指定客户重生成时，如果历史上已存在更早成功排餐，则当前记录不应再次显示首次

### 同日多次生成

- 同一订单同一餐次在同一天多次成功生成时，只标记最早成功的一条

### 当前记录失败

- 当前记录若失败，`firstMealOfOrder` 必须返回 `false`

### 混合状态页面

- 一个排餐计划中可同时存在：
  - 首次成功客户
  - 非首次成功客户
  - 失败客户
- 左侧置顶仅针对 `firstMealOfOrder = true` 的记录，失败客户不参与置顶

## 测试要求

### 后端

至少覆盖以下场景：

1. 同一订单首次早餐成功，返回 `true`
2. 同一订单已有更早早餐成功，返回 `false`
3. 首次失败、第二次成功时，第二次返回 `true`
4. 历史记录被软删除后，当前成功记录可重新成为首次
5. 同一客户新订单首次午餐成功，返回 `true`
6. 同一订单早餐首次不影响午餐首次

### 前端

至少验证以下行为：

1. `allCustomers` 将首次客户排在前面，普通客户随后
2. 组内原始顺序保持不变
3. 首次客户显示绿色左侧色条和 `首` 徽章
4. 首次客户叠加无汤红圈时，两者不互相覆盖
5. 打印页与管理页样式一致

## 实施建议

采用“后端统一计算，前端只消费”的方案：

- 后端负责统一口径，避免页面重复推导
- 前端只负责左侧编号区排序和样式展示
- 不落库，避免对现有幂等生成和软删除逻辑引入额外维护成本
