# 智能排查对话模式设计

## 背景

当前 `/meal/agent/diagnosis` 页面是一次性表单工具：用户填写客户 ID 或客户编号、日期、餐次，前端调用 `POST /api/agent/meal-plan/diagnose`，后端返回摘要、原因列表和证据表格。

目标是改为 B2 方案：页面变成纯对话模式，用户可以用自然语言描述排查诉求。系统在后端完成意图识别、槽位抽取、缺信息追问、会话状态维护，并在信息齐全时复用现有诊断链路生成诊断结果。

## 目标

- 将前端改为聊天式体验，不再暴露结构化表单作为主入口。
- 新增后端聊天接口，前端只提交 `sessionId` 和用户消息。
- 后端识别客户、日期、餐次等诊断槽位，缺失时以助手消息追问。
- 槽位齐全后调用现有 `MealPlanDiagnosisService.diagnose()` 链路，不重复实现诊断逻辑。
- 诊断结果以聊天消息中的结构化卡片展示，继续保留原因、建议、证据。
- 第一版聚焦“排餐诊断型聊天”，不支持开放闲聊和跨业务动作。

## 非目标

- 不在第一版支持修改订单、修改客户档案、重新生成排餐等写操作。
- 不把会话状态第一版落库；先使用有 TTL 的内存会话。
- 不替换现有 `/api/agent/meal-plan/diagnose` 接口，该接口继续作为底层诊断能力保留。
- 不支持无限制自由问答；无关问题应明确收敛到排餐诊断范围。

## 总体架构

链路分为三层：

1. `eladmin-web`
   - `/meal/agent/diagnosis` 页面改成聊天 UI。
   - 调用新的 `chat` API。
   - 渲染用户消息、助手消息、追问按钮和诊断结果卡片。

2. `eladmin-system`
   - 保持后台网关角色。
   - 新增 `POST /api/agent/meal-plan/chat`。
   - 负责权限校验、请求 ID、日志、调用 `agent-service`、服务不可用兜底。

3. `agent-service`
   - 新增聊天编排服务。
   - 维护会话状态。
   - 调用 LLM 抽取意图和槽位。
   - 信息不足时生成追问。
   - 信息齐全时组装 `DiagnosisRequest` 并复用现有诊断服务。

## 接口设计

### 前端到 eladmin-system

`POST /api/agent/meal-plan/chat`

请求：

```json
{
  "sessionId": "uuid-or-null",
  "message": "帮我看下客户 C10001 明天午餐为什么没排出来"
}
```

响应：

```json
{
  "sessionId": "uuid",
  "status": "ANSWERED",
  "assistantMessage": "已完成诊断，发现 2 个可能原因，请结合证据人工确认。",
  "slots": {
    "customerId": null,
    "customerCode": "C10001",
    "recordDate": "2026-05-22",
    "mealType": "LUNCH"
  },
  "diagnosisResult": {
    "summary": "...",
    "reasons": []
  },
  "quickReplies": ["继续追问", "重新排查", "清空会话"]
}
```

### eladmin-system 到 agent-service

建议新增同路径接口：

`POST {agent.service.base-url}/api/agent/meal-plan/chat`

`eladmin-system` 的 `AgentServiceClient` 新增：

```java
AgentChatResponse chatMealPlan(AgentChatRequest request);
```

保留现有：

```java
AgentDiagnosisResponse diagnoseMealPlan(AgentDiagnosisRequest request);
```

## DTO 设计

### ChatRequest

- `sessionId`：可空。为空时后端创建新会话。
- `message`：必填，用户输入。

### ChatResponse

- `sessionId`：当前会话 ID。
- `status`：当前响应状态。
- `assistantMessage`：助手文本回复。
- `slots`：当前已识别诊断条件。
- `diagnosisResult`：诊断完成时返回；追问阶段为空。
- `quickReplies`：可选快捷回复。

### ChatStatus

- `NEED_MORE_INFO`：缺少客户、日期或餐次，需要继续追问。
- `DIAGNOSING`：可选状态，用于后续异步化；第一版同步接口可不返回该状态。
- `ANSWERED`：已完成诊断或已回答本次追问。
- `RESET`：会话已重置。
- `ERROR`：服务异常或无法处理。

### DiagnosisSlots

- `customerId`
- `customerCode`
- `recordDate`
- `mealType`

## 会话状态

`agent-service` 第一版使用内存会话：

- key：`sessionId`
- TTL：30 分钟
- 内容：
  - 当前槽位 `DiagnosisSlots`
  - 最近一次用户消息
  - 最近一次诊断结果
  - 消息摘要或最近 N 条消息

会话状态只用于诊断上下文，不作为审计存储。后续如果部署多实例或需要刷新后恢复，再切换到 Redis 或数据库。

## 聊天编排流程

1. 接收 `ChatRequest`。
2. 如果 `sessionId` 为空，创建新会话。
3. 如果用户表达“清空”“重新开始”“换一个客户”，重置对应会话槽位。
4. 调用 LLM 抽取结构化结果：

```json
{
  "intent": "DIAGNOSE",
  "slots": {
    "customerId": null,
    "customerCode": "C10001",
    "recordDate": "2026-05-22",
    "mealType": "LUNCH"
  },
  "missingSlots": [],
  "reply": ""
}
```

5. 合并新槽位到会话。
6. 如果缺少客户标识，追问：“请提供客户 ID 或客户编号。”
7. 如果缺少日期，追问：“请补充要排查的日期，例如今天、明天或 2026-05-22。”
8. 如果缺少餐次，追问：“请补充餐次：早餐、午餐还是晚餐？”
9. 如果槽位齐全，构造 `DiagnosisRequest` 调用现有 `MealPlanDiagnosisService.diagnose()`。
10. 把 `DiagnosisResponse` 包装成 `ChatResponse` 返回。

## LLM 职责边界

LLM 在聊天链路中负责：

- 判断用户意图。
- 从自然语言中抽取客户、日期、餐次。
- 基于缺失槽位生成简短追问。
- 对最近一次诊断结果做解释型追问回答。

LLM 不负责：

- 修改数据库。
- 创建设定或生成业务数据。
- 绕过现有诊断工具直接臆测结论。
- 输出未经过现有诊断链路支持的最终诊断。

## 意图类型

- `DIAGNOSE`：发起或继续补全一次排餐诊断。
- `FOLLOW_UP`：基于最近一次诊断结果追问，例如“为什么订单无效？”
- `RESET`：清空会话或重新开始。
- `OUT_OF_SCOPE`：与排餐诊断无关。

第一版 `FOLLOW_UP` 可以先基于最近一次 `diagnosisResult` 让 LLM 做解释，不重新查库。涉及新客户、新日期、新餐次时转为 `DIAGNOSE`。

## 前端设计

`eladmin-web/src/views/agent/diagnosis/index.vue` 改为聊天页面：

- 顶部标题：智能排查助手。
- 右上角：清空会话。
- 中间消息列表：
  - 用户消息右侧气泡。
  - 助手消息左侧气泡。
  - 诊断结果作为助手消息内的结果卡片。
- 底部输入区：
  - 多行输入框。
  - 发送按钮。
  - loading 状态。
- 快捷回复：
  - 今天
  - 明天
  - 早餐
  - 午餐
  - 晚餐
  - 重新排查

诊断结果卡片复用现有展示信息：

- `summary`
- 客户、日期、餐次、模型、规则版本
- `fallback` 标签
- `reasons` 折叠面板
- evidence 表格

## 错误处理

- `agent-service` 不可用：`eladmin-system` 返回 `ERROR` 状态和人工核对提示。
- LLM 抽取失败：返回追问，让用户按“客户 + 日期 + 餐次”重新描述。
- 诊断接口失败：复用现有 fallback 结果，并在聊天中提示服务暂不可用。
- 用户输入空消息：前端禁用发送或提示请输入内容。
- 用户输入无关内容：返回“我目前只能处理排餐诊断，请提供客户、日期和餐次。”

## 权限与日志

- 新 chat 接口沿用 `agentDiagnosis:list` 权限。
- `eladmin-system` 每次请求生成或透传 `X-Request-Id`。
- `agent-service` 在 MDC 中记录 `requestId`。
- 日志至少包含：
  - `sessionId`
  - `intent`
  - 已识别槽位
  - 缺失槽位
  - 是否触发诊断
  - 诊断耗时
  - fallback 状态

日志不应记录完整敏感业务明细；工具调用日志沿用现有策略。

- 初始欢迎消息展示。
- 发送消息后渲染用户气泡和助手气泡。
- `NEED_MORE_INFO` 时展示追问和快捷回复。
- `ANSWERED` 时展示诊断结果卡片。

- `AgentDiagnosisFacadeServiceImpl` 转发 chat 请求。

### agent-service 单测

- 新会话创建。
- 槽位抽取结果合并。
- 缺客户、缺日期、缺餐次分别追问。
- 槽位齐全时调用 `MealPlanDiagnosisService.diagnose()`。

- `OUT_OF_SCOPE` 返回范围提示。

## 迁移与兼容

- 原页面路径 `/meal/agent/diagnosis` 不变。
- 原菜单权限不变。
- 原 `/api/agent/meal-plan/diagnose` 保留，供 chat 编排和其他调用方继续使用。
- 前端 API 文件 `agentDiagnosis.js` 新增 `chatMealPlan`，保留 `diagnoseMealPlan`。

## 风险

- 自然语言日期解析需要明确当前日期和时区，LLM 提示词必须注入服务端当前日期。
- 内存会话在多实例部署下不共享，第一版只适合单实例或弱会话一致性场景。
- LLM 抽取不稳定时会影响体验，需要严格 JSON 输出解析和兜底追问。
- 同步诊断可能响应较慢，前端需要清晰 loading；后续可演进为异步或流式响应。

## 后续演进

- 会话状态迁移到 Redis。
- 支持 SSE 流式输出。
- 支持诊断过程事件展示，例如“正在查询客户档案”“正在查询订单”。
- 支持更多排餐相关追问，并按工具返回数据重新查询。
- 增加会话历史列表和最近诊断入口。
