# 排餐计划生成接口

> 文档版本：1.0
> 创建日期：2026-03-31
> 状态：已实现

---

## 1. 接口概览

- **接口地址**：`POST /api/meal-plan/generate`
- **权限标识**：`mealPlan:generate`
- **接口说明**：按日期 + 餐次生成客户排餐计划，并写入 `meal_plan`、`meal_plan_customer`、`meal_plan_customer_item` 三张表。

---

## 2. 请求参数

### 2.1 请求体

```json
{
  "recordDate": "2026-04-01",
  "mealType": "LUNCH",
  "customerId": 123
}
```

### 2.2 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `recordDate` | string | 是 | 排餐日期，格式 `yyyy-MM-dd` |
| `mealType` | string | 是 | 餐次，只支持 `LUNCH` / `DINNER` |
| `customerId` | long | 否 | 指定客户ID，不传则生成全部客户的排餐计划 |

---

## 3. scheduleKey 规则

`scheduleKey` 用于匹配 `dish.schedule`。

- 1-7 日 → 第 1 周
- 8-14 日 → 第 2 周
- 15-21 日 → 第 3 周
- 22-28 日 → 第 4 周
- 29-31 日 → 映射为第 1 周
- 周一 = 1，周日 = 7

格式：`W-D`

示例：

- `2026-04-01` → `1-3`
- `2026-04-08` → `2-3`
- `2026-04-25` → `4-6`
- `2026-04-29` → `1-3`

---

## 4. 幂等逻辑

同一天、同餐次重复生成时，采用**软删除旧记录 + 重新生成新记录**。

事务内顺序：

1. 查询 `meal_plan` 中 `(record_date, meal_type, deleted=0)` 的有效主记录
2. 软删除该主记录下的 `meal_plan_customer_item`
3. 软删除该主记录下的 `meal_plan_customer`
4. 软删除 `meal_plan`
5. 插入新一轮排餐结果

说明：

- 三张表都带 `deleted` 字段
- 不使用唯一约束，保留软删除历史

---

## 5. 业务规则

### 5.1 有效订单过滤

同时满足以下条件才参与排餐：

- `status = 1`
- `remaining_count > 0`
- `start_date <= recordDate <= end_date`
- `meal_type in (当前 mealType, ALL)`
- `schedule_mode` 满足当前日期：
  - `DAILY`
  - `WEEKDAY`
  - `WEEKEND`
  - `SCHEDULE` 且 `delivery_dates` 包含当天

### 5.2 候选菜过滤

候选菜同时满足：

- `enabled = 1`
- `meal_types` 包含当前餐次
- `schedule` 包含当前 `scheduleKey`
- `meal_packages` 匹配订单父套餐，兼容：
  - 数字 ID
  - 字符串数字
  - 套餐编码

### 5.3 过敏过滤

- 使用客户 `allergy_tags` 与 `dish_ingredient.name` **完全相等** 匹配
- 不使用 `contains`
- 不做模糊匹配

### 5.4 选菜规则

- `meatCount = 1`：优先 `MAIN`，没有则回退 `SIDE`
- `meatCount = 2`：必须同时选出 `MAIN + SIDE`
- `vegCount >= 1`：从 `VEGETABLE` 中选
- `includeSoup = true`：从 `SOUP` 中选
- `includeRice = true`：从 `RICE` 中选
- 同一客户同一餐次内不允许重复同一道菜

## 6. 返回示例

```json
{
  "mealPlanId": 101,
  "recordDate": "2026-04-01",
  "mealType": "LUNCH",
  "totalCount": 3,
  "successCount": 2,
  "failCount": 1,
  "failDetails": [
    {
      "failReason": "荤菜不足，必须同时选出主菜和副菜"
    }
  ]
}
```

---

## 7. 失败原因说明

当前实现可能返回的失败原因包括：

- `客户档案不存在`
- `子套餐不存在`
- `父套餐不存在`
- `当前父套餐没有可用候选菜`
- `荤菜不足`
- `荤菜不足，必须同时选出主菜和副菜`
- `荤菜数量配置不支持`
- `素菜不足`
- `SOUP类型菜品不足`
- `RICE类型菜品不足`
