# 排餐计划生成接口

> 文档版本：2.1
> 创建日期：2026-03-31
> 更新日期：2026-05-04
> 状态：已实现

---

## 1. 接口概览

- **接口地址**：`POST /api/meal-plan/generate`
- **权限标识**：`mealPlan:generate`
- **接口说明**：按日期 + 餐次生成客户排餐计划，并写入 `meal_plan`、`meal_plan_customer`、`meal_plan_customer_item` 三张表。

---

## 2. 请求参数

### 2.1 请求体

```json
{
  "recordDate": "2026-04-01",
  "mealType": "LUNCH",
  "menuWeekNum": 2,
  "menuDayOfWeek": 3,
  "customerId": 123
}
```

### 2.2 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `recordDate` | string | 是 | 实际排餐/配送日期，格式 `yyyy-MM-dd` |
| `mealType` | string | 是 | 餐次，支持 `BREAKFAST` / `LUNCH` / `DINNER` |
| `menuWeekNum` | int | 否 | 指定菜单周次，仅用于查询 `meal_schedule_plan`，范围 `1-4`，需与 `menuDayOfWeek` 同时传入；不传则按 `recordDate` 推导 |
| `menuDayOfWeek` | int | 否 | 指定菜单星期，仅用于查询 `meal_schedule_plan`，范围 `1-7`，需与 `menuWeekNum` 同时传入；不传则按 `recordDate` 推导 |
| `customerId` | long | 否 | 指定客户ID，不传则生成全部客户的排餐计划 |

---

## 3. 菜单定位规则

`recordDate` 始终表示实际排餐/配送日期，用于：

- 有效订单过滤
- 排餐主记录写入
- 客户排除日期判断
- 已排餐数量统计

`menuWeekNum + menuDayOfWeek` 仅用于查询 `meal_schedule_plan`：

- 两个字段要么都不传，要么都传
- 两个字段都不传时，按 `recordDate` 推导菜单周次和菜单星期
- 两个字段都传时，`recordDate` 不变，仅候选菜查询改为使用指定菜单周次和菜单星期

推导规则如下：

- 1-7 日 → 第 1 周
- 8-14 日 → 第 2 周
- 15-21 日 → 第 3 周
- 22-28 日 → 第 4 周
- 29-31 日 → 映射为第 1 周
- 周一 = 1，周日 = 7

示例：

- `recordDate=2026-04-01`，未传菜单定位字段 → 查询 `meal_schedule_plan(week_num=1, day_of_week=3)`
- `recordDate=2026-04-01`，传 `menuWeekNum=2`、`menuDayOfWeek=3` → 实际排餐日期仍是 `2026-04-01`，但查询 `meal_schedule_plan(week_num=2, day_of_week=3)`

---

## 4. 幂等逻辑

同一天、同餐次重复生成时，采用**软删除旧记录 + 重新生成新记录**。

事务内顺序：

1. 查询 `meal_plan` 中 `(record_date, meal_type, deleted=0)` 的有效主记录
2. 软删除该主记录下的 `meal_plan_customer_item`
3. 软删除该主记录下的 `meal_plan_customer`
4. 软删除 `meal_plan`
5. 插入新一轮排餐结果

**指定客户模式**：如果传入 `customerId`，只软删除该客户的排餐明细，保留主记录和其他客户的记录。

说明：

- 三张表都带 `deleted` 字段
- 不使用唯一约束，保留软删除历史

---

## 5. 业务规则

### 5.1 有效订单过滤

同时满足以下条件才参与排餐：

- `status = 1`
- `remaining_count > 0`
- `start_date <= recordDate <= end_date`
- `meal_type in (当前 mealType, ALL)`
- `schedule_mode` 满足当前日期：
  - `DAILY`
  - `WEEKDAY`
  - `WEEKEND`
  - `SCHEDULE` 且 `delivery_dates` 命中当天
    - 旧格式：`["2026-04-01", "2026-04-02"]`，只按日期判断
    - 新格式：`[{"date": "2026-04-01", "mealTypes": ["LUNCH", "DINNER"]}]`，同时要求 `mealTypes` 包含当前餐次
- 当前餐次的已排餐数量 `<` 订单对应餐数上限：
  - `BREAKFAST` 对比 `breakfastCount`
  - `LUNCH` / `DINNER` 对比 `lunchDinnerCount`
  - 统计口径为该订单在当前餐次下的有效排餐记录数（`meal_plan_customer.deleted=0` 且关联 `meal_plan.deleted=0`）

### 5.2 客户排除日期过滤

检查客户档案中的 `exclude_dates` 字段，如果该日期+餐次被排除，则跳过该订单。

### 5.3 候选菜过滤

候选菜同时满足：

- `enabled = 1`
- `meal_types` 包含当前餐次
- 普通菜品通过 `meal_schedule_plan` 按 `week_num + day_of_week + meal_time` 查询
- `meal_packages` 匹配订单父套餐，兼容：
  - 数字 ID
  - 字符串数字
  - 套餐编码
- 米饭从 `dish` 主档单独读取，`dish_type = RICE_TYPE`，不依赖 `meal_schedule_plan`

说明：

- `menuWeekNum + menuDayOfWeek` 只影响普通菜品的 `meal_schedule_plan` 查询
- `BREAKFAST` / `LUNCH` / `DINNER` 都支持生成；其中 `BREAKFAST` 生成客户排餐记录，但不生成菜品明细

### 5.4 过敏过滤

- 三级匹配按以下顺序执行：
  - 配料名精确匹配：客户 `allergy_tags` 与 `dish_ingredient.name` **完全相等**
  - 二级分类展开匹配：客户标签命中某个二级分类名时，展开该分类下全部配料名再匹配菜品配料
  - 一级分类展开匹配：客户标签命中某个一级分类名时，展开该分类下全部配料名再匹配菜品配料
- 不使用 `contains`
- 不做模糊匹配

### 5.5 选菜规则

**每日固定提供规则**：每日每类菜品固定提供 1 份，超出部分通过补菜数量记录。

| 菜品类型 | 每日固定 | 数量来源 |
|---------|---------|---------|
| 主菜 | 1 份 | 从 MAIN 类型选 |
| 副菜 | 1 份 | 从 SIDE 类型选 |
| 素菜 | 1 份 | 从 VEGETABLE 类型选 |
| 汤 | 0-1 份 | 从 SOUP 类型选（`soup_count > 0` 时才选） |
| 米饭 | 0-1 份 | 从 `dish` 主档的 `RICE_TYPE` 类型选（`rice_count > 0` 时才选，不依赖 `meal_schedule_plan`，进入排餐后按 `RICE` 处理） |

**补菜数量计算**：
```
补菜数量 = max(0, 订单配置数 - 每日固定1份)
```

示例：

| 订单配置 | 每日固定 | 补菜数量 |
|---------|---------|---------|
| 主菜×1 | 1 | 0 |
| 主菜×2 | 1 | 1 |
| 主菜×3 | 1 | 2 |

补菜数量记录在 `meal_plan_customer` 表的 `supplementary_*_count` 字段。

### 5.6 回退逻辑（暂未启用）

当日菜品因过敏全部过滤时，暂不启用：
- 次选类型替换（如 MAIN 全过敏用 SIDE）
- 次日菜品回退

## 6. 返回示例

```json
{
  "mealPlanId": 101,
  "recordDate": "2026-04-01",
  "mealType": "LUNCH",
  "totalCount": 3,
  "successCount": 2,
  "failCount": 1,
  "failDetails": [
    {
      "failReason": "主菜不足"
    }
  ]
}
```

---

## 7. 失败原因说明

当前实现可能返回的失败原因包括：

- `客户档案不存在`
- `父套餐不存在`
- `当前父套餐没有可用候选菜`
- `主菜不足`
- `副菜不足`
- `素菜不足`
- `SOUP类型菜品不足`
- `米饭类型菜品不足`
