# 智能排查助手接口文档

## 1. 后台对前端接口

### 1.1 排餐未生成诊断

- Method: `POST`
- Path: `/api/agent/meal-plan/diagnose`
- 调用方: `eladmin-web`
- 提供方: `eladmin-system`
- 权限: `agentDiagnosis:list`

请求体:

```json
{
  "customerId": 1001,
  "customerCode": "C20260518001",
  "recordDate": "2026-05-17",
  "mealType": "LUNCH"
}
```

字段说明:

- `customerId`: 客户 ID，和 `customerCode` 至少传一个
- `customerCode`: 客户编号，和 `customerId` 至少传一个
- `recordDate`: 诊断日期，格式 `yyyy-MM-dd`
- `mealType`: 餐次，可选 `BREAKFAST`、`LUNCH`、`DINNER`

响应体:

```json
{
  "requestId": "diag-20260518-001",
  "customerId": 1001,
  "customerName": "张三",
  "recordDate": "2026-05-17",
  "mealType": "LUNCH",
  "summary": "客户 1001 在 2026-05-17 午餐未生成排餐，最可能原因是命中客户排除日期。",
  "confidence": "HIGH",
  "ruleVersionDigest": "a4f8c1e9d320",
  "modelName": "spring-ai-chat-client",
  "fallback": false,
  "fallbackReason": null,
  "nextActions": ["核对客户档案停送配置", "确认是否需要恢复配送"],
  "actionDrafts": [],
  "diagnosisTrace": [],
  "toolCallSummary": [],
  "reasons": [
    {
      "code": "CUSTOMER_EXCLUDE_DATE_HIT",
      "title": "命中客户排除日期",
      "level": "HIGH",
      "confidence": "HIGH",
      "ruleIds": ["CUSTOMER_EXCLUDE_DATE_HIT"],
      "description": "客户档案配置了目标日期和餐次不配送。",
      "suggestion": "先核对客户是否已登记停送。",
      "nextActions": ["核对客户档案停送配置", "确认是否需要恢复配送"],
      "evidence": [
        {
          "label": "customerProfile.excludeDates",
          "value": "2026-05-17:LUNCH"
        }
      ]
    }
  ]
}
```

响应字段说明:

- `summary`: AI 诊断摘要，供客服快速阅读
- `confidence`: 模型建议可信度，仅表示模型对当前建议的把握，不代表业务最终结论
- `ruleVersionDigest`: 本次使用的规则版本摘要，用于排查和审计
- `modelName`: 模型调用方标识；兜底结果可能为空
- `fallback`: 是否为兜底结果
- `fallbackReason`: 兜底原因；`fallback=true` 时应优先展示给客服
- `fallbackSource`: 兜底来源；当前主系统调用 `agent-service` 失败时为 `ELADMIN_CLIENT`
- `failureType`: 失败类型；当前包括 `AGENT_SERVICE_TIMEOUT`、`AGENT_SERVICE_UNAVAILABLE`、`AGENT_SERVICE_BAD_RESPONSE`、`AGENT_SERVICE_4XX`、`AGENT_SERVICE_5XX`
- `nextActions`: 建议客服继续人工核对的动作列表
- `actionDrafts`: 为兼容历史消息保留的字段；当前只读 Agent 始终返回空列表，前端不得显示或调用确认入口
- `diagnosisTrace`: 完整诊断链路摘要，面向客服和技术排查，不代表任何数据库操作
- `toolCallSummary`: 工具调用摘要，默认不包含原始敏感入参/出参
- `reasons`: 可能原因列表
- `reasons[].code`: 稳定原因码
- `reasons[].level`: `HIGH`、`MEDIUM`、`LOW`
- `reasons[].confidence`: 原因级建议可信度
- `reasons[].ruleIds`: 该原因关联的规则 ID 列表
- `reasons[].nextActions`: 该原因对应的建议动作
- `reasons[].evidence`: 字段级证据，`label` 必须命中关联规则的 `evidenceFields`，并能追溯到规则或上下文数据

### 1.2 排餐诊断聊天接口

- Method: `POST`
- Path: `/api/agent/meal-plan/chat`
- 调用方: `eladmin-web`
- 提供方: `eladmin-system`
- 权限: `agentDiagnosis:list`

请求体:

```json
{
  "sessionId": "session-001",
  "clientMessageId": "3b26a8ad-9bde-4f79-ae75-7a0df6c142dd",
  "message": "看下 C10001 明天午餐为什么没排出来"
}
```

响应体:

```json
{
  "requestId": "chat-20260707-001",
  "sessionId": "session-001",
  "status": "NEED_MORE_INFO",
  "assistantMessage": "请补充餐次：早餐、午餐还是晚餐？",
  "slots": {
    "customerCode": "C10001",
    "recordDate": "2026-07-08",
    "slotConfidence": {
      "customer": "HIGH",
      "recordDate": "HIGH"
    },
    "slotSource": {
      "customer": "EXPLICIT_INPUT",
      "recordDate": "EXPLICIT_INPUT"
    }
  },
  "slotConfidence": {
    "customer": "HIGH",
    "recordDate": "HIGH"
  },
  "missingSlots": ["MEAL_TYPE"],
  "diagnosisResult": null,
  "quickReplies": ["早餐", "午餐", "晚餐"],
  "conversationStage": "COLLECTING_SLOTS",
  "pendingBusinessQueryContext": {
    "missingFields": ["mealType"],
    "originalQuestionSummary": "查询指定客户排餐",
    "expiresAt": "2026-07-08T10:51:15+08:00"
  },
  "lastBusinessQueryContext": null,
  "semanticTraceSummary": {
    "semanticSource": "LLM",
    "semanticCatalogVersion": "2026.07",
    "temporalExpression": "CURRENT_DAY",
    "resolvedRecordDate": "2026-07-08",
    "pendingContextReused": false
  }
}
```

聊天响应字段说明:

- `requestId`: 本轮聊天请求标识，用于日志排查
- `sessionId`: 会话标识；首次可不传，服务端会生成
- `clientMessageId`: 前端消息幂等 ID；重复提交同一条消息时必须保持不变，后端命中后直接回放已有助手响应
- `slotConfidence`: 当前会话对各槽位的置信度摘要，取值 `HIGH`、`MEDIUM`、`LOW`
- `slots.slotSource`: 槽位来源，当前可能为 `EXPLICIT_INPUT`、`CONTEXT_INHERIT`、`CORRECTION_OVERRIDE`、`AMBIGUOUS_INPUT`
- `missingSlots`: 当前仍缺失的关键诊断槽位
- `conversationStage`: 当前会话阶段，可取 `COLLECTING_SLOTS`、`READY_TO_DIAGNOSE`、`DIAGNOSING`、`DIAGNOSED`、`FOLLOWING_UP`、`RESET`、`ERROR`
- `diagnosisResult`: 当本轮触发诊断或复用上一轮结果时返回
- `responseType`: 客服业务查询响应类型，例如 `BUSINESS_QUERY_CUSTOMER`、`BUSINESS_QUERY_ORDER`、`BUSINESS_QUERY_MEAL_PLAN`
- 客户历史排餐问法（例如在查询 B2200 后追问“他排过餐吗”）返回 `BUSINESS_QUERY_MEAL_PLAN`；无日期表示全部历史，回答直接说明是否曾经排餐、总记录数和最近一条记录，不返回日期澄清。
- 公共菜单查询的 `responseType` 为 `BUSINESS_QUERY_SCHEDULED_MENU`；该查询仅需日期，不要求客户信息，`今天`、`明天`会被解析为对应的 `slots.recordDate`
- `insightResult`: 业务只读查询的脱敏结构化结果，仅包含当前响应类型允许展示的字段
- `BUSINESS_QUERY_CUSTOMER_CANDIDATES`: 客户姓名命中多个候选时返回，`status=NEED_MORE_INFO`，`insightResult.items` 为最多 10 个脱敏候选客户；前端以选择卡片展示，客服必须选择客户编号或客户 ID 后才能继续查询
- `facts`: 回答引用的可追溯事实列表，`factId` 会在助手话术中以 `F1`、`F2` 等形式引用
- `warnings`: 受控查询告警列表，例如计划非法、工具调用失败或回答安全校验未通过
- `cached`: 是否命中同一轮业务查询编排器缓存；为 `true` 时表示相同 QueryPlan/工具/参数未重复调用主系统
- `partial`: 是否为部分结果；工具预算、计划校验或下游只读接口失败时会返回 `true`
- 本期响应不会生成可执行动作草稿，也不会触发动作确认接口；`nextActions` 仅表示人工核对建议。
- `queriedAt`: 业务查询响应生成时间，东八区 ISO-8601 字符串
- `queryPlan`: 本轮服务端确定性 QueryPlan，用于前端展示、审计和问题追溯
- `pendingBusinessQueryContext`: 待补充条件的内部受控上下文。该字段由主系统会话服务持久化并下发给 Agent，前端请求中的同名字段不会成为信任来源，不得自行构造或修改。
- `lastBusinessQueryContext`: 最近一次已执行业务查询的脱敏摘要，用于跨实例继承时间和纠错重规划；不包含工具原始结果。
- `semanticTraceSummary`: 脱敏语义追踪，包含来源、fallback 原因、目录版本、时间枚举、解析日期和 Pending 复用标记，不包含用户原文、Prompt 或模型原始响应。
- `semanticTraceSummary.semanticSource`: `LLM`、`RULE_FALLBACK` 或 `PENDING_CONTEXT`。`NEED_MORE_INFO` 是澄清状态，不自动视为 fallback。
- “现在/当前/目前/截至现在”在每日指标中由模型输出 `CURRENT_DAY`，具体 `recordDate` 由 Agent 按 `Asia/Shanghai` 解析；模型不得直接计算相对日期。

### 1.3 会话管理接口

#### 1.3.1 查询会话列表

- Method: `GET`
- Path: `/api/agent/chat-sessions`
- 权限: `agentDiagnosis:list`

查询参数:

- `keyword`: 按客户编号、标题、最近摘要模糊查询

- `recordDateStart`

- 权限: `agentDiagnosis:list`

请求体示例:

```json
{
  "customerId": 1001,
  "customerCode": "C10001",
```

#### 1.3.3 查询会话详情

- Method: `GET`
- Path: `/api/agent/chat-sessions/{sessionId}`
- 权限: `agentDiagnosis:list`

响应体包含:

- 会话摘要字段
- `orderId`、`orderCode`、`mealPlanRecordId`: 当前订单和排餐焦点；客户切换后服务端会清空旧焦点，避免跨客户查询
- `currentSlots`: 当前槽位快照
- `latestDiagnosisResult`: 最近一次诊断结果
- `messages`: 完整消息列表
- `recentAudits`: 最近动作确认审计
- `recentFeedbacks`: 最近客服反馈
- `pendingBusinessQueryContext`: 当前待补条件的受控上下文，仅用于会话恢复和内部排障
- `lastBusinessQueryContext`: 最近已执行业务查询的脱敏摘要

已有数据库需执行 `eladmin/sql/alter_agent_chat_session_semantic_context.sql`。两个字段只允许主系统与 Agent 内部链路读写，前端不能利用会话接口覆盖它们。

### 1.7 运营统计接口补充说明

`GET /api/agent/operation/stats` 当前除原有诊断次数、fallback 率、采纳率、动作确认率外，还返回：

- `failureTypeDistribution`: 按失败类型聚合的分布，如 `AGENT_SERVICE_TIMEOUT`
- `fallbackSourceDistribution`: 按兜底来源聚合的分布，如 `ELADMIN_CLIENT`
- `businessQueryCount`、`businessQueryPartialRate`、`businessQueryCachedRate`、`businessQueryFailureRate`: Agent 业务只读查询健康度指标
- `businessQueryDomainDistribution`、`businessQueryToolDistribution`、`businessQueryFailureTypeDistribution`: 业务查询领域、工具和失败类型分布

### 1.7.1 业务只读查询审计接口

- Method: `GET`
- Path: `/api/agent/business-query-audits`
- 权限: `agentDiagnosis:list`

查询参数:

- `operator`: 客服账号
- `sessionId`
- `requestId`
- `queryDomain`: 查询领域，例如 `CUSTOMER`、`ORDER`、`MEAL_PLAN`
- `queryAction`: 查询动作，例如 `OVERVIEW`、`LIST`、`DETAIL`
- `customerId` / `customerCode`
- `orderId` / `orderCode`
- `cached`: 是否命中同轮缓存
- `partial`: 是否部分成功或截断
- `failureType`: 稳定失败类型
- `createTimeStart` / `createTimeEnd`: 创建时间范围，格式 `yyyy-MM-dd HH:mm:ss`
- `page` / `size`

响应为 `PageResult<AgentBusinessQueryAudit>`。审计记录只保存客服、会话、请求、QueryPlan 摘要、工具名称摘要、结果条数、缓存/部分成功标记、失败类型和耗时，不保存工具原始请求或响应。

### 1.7.2 业务只读查询审计统计接口

- Method: `GET`
- Path: `/api/agent/business-query-audits/stats`
- 权限: `agentDiagnosis:list`

查询参数同 `/api/agent/business-query-audits`。

响应字段:

- `queryCount`: 业务查询总次数
- `partialCount` / `partialRate`: 部分成功或截断次数与比例
- `cachedCount` / `cachedRate`: 同轮缓存命中次数与比例
- `failureCount` / `failureRate`: 失败次数与比例
- `directAnswerCount` / `directAnswerRate`: 无澄清、无失败且非部分结果的完整回答次数与比例
- `clarificationRequiredCount` / `clarificationRequiredRate`: 关键澄清次数与比例
- `clarificationSuccessRate`: 同一会话发起澄清后最终获得完整有效回答的比例
- `semanticFallbackCount` / `semanticFallbackRate`: LLM 不可用、非法或低置信度后使用规则兜底的次数与比例，正常澄清不计入
- `pendingContextReuseCount` / `pendingContextReuseRate`: 成功复用待补查询上下文的次数与比例
- `answerValidationRejectedCount` / `answerValidationRejectedRate`: 回答事实安全校验拒绝次数与比例
- `permissionDeniedCount`: 工具白名单或主系统权限拒绝次数
- `averageCostMs` / `p95CostMs`: 平均与 P95 查询耗时
- `domainDistribution`: 查询领域分布
- `toolDistribution`: 工具调用分布
- `metricDistribution`: 登记指标代码分布
- `failureTypeDistribution`: 失败类型分布
- `unsupportedReasonDistribution`: 未支持问题和澄清原因分布，可按次数取 Top N
- `semanticSourceDistribution`: `LLM`、`RULE_FALLBACK`、`PENDING_CONTEXT` 等受控语义来源分布
- `semanticFallbackReasonDistribution`: `MODEL_TIMEOUT`、`MODEL_INVALID`、`MODEL_LOW_CONFIDENCE`、`MODEL_UNAVAILABLE`、`MODEL_RULE_GUARDRAIL_CONFLICT` 等最长 32 位的稳定兜底原因分布；旧码 `MODEL_CONFLICTS_WITH_RULE_GUARDRAIL` 由主系统兼容转换

### 1.4 历史动作草稿确认接口（不属于本期 Agent）

以下接口仍由遗留管理功能维护，但本期智能客服 Agent 不会返回草稿、前端不展示入口，也不会调用该接口。其请求和响应示例仅供维护旧功能时参考，不构成当前只读查询协议。

- Method: `POST`
- Path: `/api/agent/action-drafts/confirm`
- 调用方: `eladmin-web`
- 提供方: `eladmin-system`
- 权限: `agentDiagnosis:confirm`

请求体:

```json
{
  "requestId": "diag-20260518-001",
  "sessionId": "session-001",
  "idempotencyKey": "diag-20260518-001:CREATE_MANUAL_RECHECK_TASK:RECHECK_TASK:2026-05-17|LUNCH",
  "secondConfirmed": false,
  "comment": "已人工核对，创建复核记录",
  "actionDraft": {
    "actionCode": "CREATE_MANUAL_RECHECK_TASK",
    "title": "创建人工复核任务",
    "riskLevel": "LOW",
    "targetType": "RECHECK_TASK",
    "targetId": "2026-05-17|LUNCH",
    "beforeSnapshot": {
      "source": "fallback"
    },
    "afterPreview": {
      "executeMode": "MANUAL_CONFIRM_REQUIRED",
      "taskType": "MEAL_PLAN_DIAGNOSIS_RECHECK"
    },
    "requiredPermission": "agentDiagnosis:confirm",
    "confirmApi": "/api/agent/action-drafts/confirm"
  }
}
```

响应体:

```json
{
  "auditId": 1,
  "actionCode": "CREATE_MANUAL_RECHECK_TASK",
  "status": "CONFIRMED",
  "success": true,
  "idempotentHit": false,
  "message": "动作确认已记录",
  "failureReason": null,
  "executionResult": null
}
```

字段说明:

- `idempotencyKey`: 幂等键，同一动作确认请求必须稳定复用；后端命中已有记录时返回原结果并置 `idempotentHit=true`
- `secondConfirmed`: 高风险动作必须为 `true`，否则返回 `status=NEED_SECOND_CONFIRM`
- `status`: `CONFIRMED`、`EXECUTED`、`NEED_SECOND_CONFIRM`、`PERMISSION_DENIED`、`VALIDATION_FAILED`、`STALE_DRAFT`、`EXECUTION_FAILED`、`UNSUPPORTED_ACTION`
- `success`: 是否完成当前动作确认
- `failureReason`: 失败原因；例如缺少动作所需权限、高风险未二次确认、草稿已过期，或动作主系统正式执行接口尚未接入
- `executionResult`: 正式执行动作的业务结果；审计型动作或未执行动作为空
- 后端会校验 `requiredPermission`；除 `admin` 外，当前用户必须具备动作草稿声明的业务权限
- 当前 `CREATE_MANUAL_RECHECK_TASK` 完成审计确认，不写入其他业务表
- 当前 `CREATE_CUSTOMER_PROFILE_DRAFT`、`CREATE_CUSTOMER_ORDER_DRAFT`、`SUPPLEMENT_DISH_CANDIDATES` 完成审计确认，用于登记客服后续建档、建单或配置补齐工作；不会自动写入客户、订单或菜品配置
- 当前 `RESUME_CUSTOMER_DELIVERY` 在人工确认后调用主系统客户档案服务，移除指定客户、日期、餐次的排除日期配置；不会自动补排餐
- 当前 `ADJUST_ORDER_EFFECTIVE_DATE` 在高风险二次确认后调用主系统订单服务，只调整订单开始日期或结束日期，并复用订单冲突校验
- 当前 `RECALCULATE_ORDER_BALANCE` 在高风险二次确认后调用主系统订单服务，按未删除核销日志重算订单已核销餐数、已核销金额、剩余餐数和餐费余额
- 当前 `REGENERATE_MEAL_PLAN` 在人工确认后调用主系统 `MealPlanService.generateMealPlan` 正式重新排餐，并返回排餐生成汇总
- 当前会在正式执行前做草稿过期校验：若客户排除配置、订单有效期、订单余额或排餐生成状态已变化，则返回 `STALE_DRAFT`
- `STALE_DRAFT` 时响应 `message` 为“业务数据已变化，请重新排查后再确认动作。”，`failureReason` 返回具体变化原因
- 未识别动作会记录审计并返回 `UNSUPPORTED_ACTION`，不会写入业务数据

### 1.10 规则缺口状态更新接口

- Method: `PUT`
- Path: `/api/agent/rule-gaps/{id}/status`
- 权限: `agentDiagnosis:rule`

请求体:

```json
{
  "status": "IN_PROGRESS",
  "owner": "rule-admin",
  "comment": "已确认需要补充订单餐次类型规则"
}
```

请求约束:

- `status=IN_PROGRESS` 时 `owner` 必填，用于明确规则维护负责人
- `status=RESOLVED` 时 `comment` 必填，用于记录规则 YAML、提示词策略、评测用例或发布验证证据
- 非法状态、缺少处理人或缺少解决证据时返回 `status=VALIDATION_FAILED`，不会更新缺口记录

响应体:

```json
{
  "id": 1,
  "status": "IN_PROGRESS",
  "message": "规则缺口状态已更新"
}
```

会话约定:

- 用户发送“重新排查”时，会保留当前已识别的客户、日期、餐次，并重新发起一次诊断
- 用户发送“清空会话”时，会清空槽位和最近诊断结果，重新进入槽位收集阶段
- 追问场景会复用上一轮诊断结果，优先返回证据摘要和下一步建议
- 当某个槽位已提取但置信度为 `LOW` 时，系统会优先发起单一确认问题，而不是直接诊断
- `diagnosisTrace` 和 `toolCallSummary` 仅用于排查本次诊断链路，不表示系统已执行任何业务写操作

### 20.11 槽位缺失规则

排餐诊断和客户信息查询的槽位缺失要求不同。对外意图只有顶层 `BUSINESS_QUERY`，其领域由受控分析和 QueryPlan 选择；表中细粒度名称仅描述兼容路由，不是前端可传字段：

## 4. 兜底策略

以下情况返回 `fallback=true`:

- `agent-service` 不可用
- AI 模型超时
- AI 输出不是合法结构
- AI 输出包含“已修复”“已修改数据库”等不允许的写操作描述
- rule registry 加载失败

兜底结果只提示人工核对方向，不给出确定业务结论。
