# 排餐计划列表接口优化实施计划

> 文档版本: 1.1
> 更新日期: 2026-04-06
> 状态: 待确认后实施

---

## 需求背景

### 当前接口现状

1. 列表接口 `GET /api/meal-plan?page=0&size=10`
   返回 `PageResult<MealPlan>`，仅包含主表数据。

2. 详情接口 `GET /api/meal-plan/{id}/detail`
   返回 `MealPlanDetailVO`，包含主表、客户列表、菜品明细、配料信息。

### 当前问题

- 前端列表页如果要展示客户和菜品信息，需要对列表中的每条记录继续调用详情接口。
- 当一页有 10 条计划时，请求数会变成 `1 + 10 = 11`。
- 后端已有详情聚合能力，但列表场景没有复用，导致接口层出现明显的 N+1 请求问题。

### 目标

- 保留现有列表接口的默认行为和返回结构。
- 在明确需要时，让列表接口返回带详情的聚合结果。
- 复用现有聚合查询逻辑，避免新增重复 SQL 和重复组装代码。
- 把数据库查询控制在批量查询，不引入新的内存扫描热点。

---

## 设计原则

### 1. 保持接口契约稳定

- 默认 `GET /api/meal-plan` 仍返回现有列表结构。
- 仅当传入 `includeDetail=true` 时，返回扩展后的列表详情结构。
- Controller 和 Service 不使用 `PageResult<?>` 这类弱类型签名，避免 Swagger、前端类型和调用约定变模糊。

### 2. 优先复用现有能力

- 复用现有 `MealPlanCustomerMapper` 查询客户。
- 复用现有 `MealPlanCustomerItemMapper.selectByCustomerPlanIds(...)` 查询菜品明细。
- 复用现有 `buildDishIngredientsMap(...)` 和 `convertToVO(...)` 组装配料与菜品 VO。
- 把 `queryMealPlanDetail(...)` 和列表详情场景共用的装配逻辑抽成私有方法。

### 3. 性能优化要做到“查询批量化 + 内存分组化”

- 数据库层避免循环查询。
- 内存层避免对每个 plan 反复 `stream().filter(...)` 扫描全集。
- 所有客户先按 `mealPlanId` 分组。
- 所有菜品明细先按 `customerPlanId` 分组。

---

## 推荐方案

### 方案概述

在现有列表接口上增加可选参数 `includeDetail`：

- `includeDetail=false` 或不传：返回 `PageResult<MealPlan>`
- `includeDetail=true`：返回 `PageResult<MealPlanListDetailVO>`

这里的“向后兼容”只体现在默认行为不变，不建议把 service/controller 的返回值改成泛型通配符。

### 接口定义

#### 默认场景

```http
GET /api/meal-plan?page=0&size=10
```

返回：

```java
PageResult<MealPlan>
```

#### 详情场景

```http
GET /api/meal-plan?page=0&size=10&includeDetail=true
```

返回：

```java
PageResult<MealPlanListDetailVO>
```

## 实施方案

### Phase 1：后端改造

#### 1.1 新增列表详情 VO

新增文件：

- `eladmin-system/src/main/java/me/zhengjie/modules/meal/domain/dto/MealPlanListDetailVO.java`

设计建议：

- 不直接继承 `MealPlan`
- 显式声明接口真正需要的字段
- 避免把实体对象直接作为列表聚合返回模型

参考结构：

```java
@Data
public class MealPlanListDetailVO {

    private Long id;
    private String recordDate;
    private String mealType;
    private Integer totalCount;
    private Integer successCount;
    private Integer failCount;
    private String status;
    private String generateTime;

    private Integer totalCustomers;
    private List<MealPlanDetailVO.CustomerPlanDetail> customers;
}
```

说明：

- 客户明细可直接复用现有 `MealPlanDetailVO.CustomerPlanDetail`
- 如果后续发现列表和详情字段逐渐分叉，再单独拆分内部明细 VO

#### 1.2 只补需要的新查询，不重复造轮子

现状：

- 已有 `MealPlanCustomerMapper.selectByMealPlanId(...)`
- 已有 `MealPlanCustomerItemMapper.selectByCustomerPlanIds(...)`

需要补充的仅是：

- `MealPlanCustomerMapper.selectByMealPlanIds(@Param("mealPlanIds") List<Long> mealPlanIds)`

不建议做的事：

- 不要在 `MealPlanMapper` 里新增“按客户计划批量查明细”的方法
- 不要把 `meal_plan_customer_item` 的查询复制到 `MealPlanMapper.xml`

建议新增 SQL：

文件：

- `eladmin-system/src/main/java/me/zhengjie/modules/meal/mapper/MealPlanCustomerMapper.java`
- `eladmin-system/src/main/resources/mapper/MealPlanCustomerMapper.xml`

```xml
<select id="selectByMealPlanIds" resultMap="BaseResultMap">
    select id, meal_plan_id, customer_id, customer_name, phone, order_id, parent_package_id, child_package_id,
           status, is_verified, verification_time, verification_operator, fail_reason,
           meat_required_count, veg_required_count, include_soup, include_rice,
           create_by, update_by, deleted, create_time, update_time
    from meal_plan_customer
    where deleted = 0
      and meal_plan_id in
    <foreach collection="mealPlanIds" item="id" open="(" separator="," close=")">
        #{id}
    </foreach>
    order by meal_plan_id asc, id asc
</select>
```

说明：

- 明细查询继续复用 `MealPlanCustomerItemMapper.selectByCustomerPlanIds(...)`
- 要确认其 XML 已带 `deleted = 0` 过滤；如果没有，需要一并补齐

#### 1.3 Service 设计

不建议直接把原方法改成：

```java
PageResult<?> queryAll(MealPlanQueryCriteria criteria, Boolean includeDetail);
```

建议改为以下两种方式之一。

方式 A，最小改动：

```java
PageResult<?> queryAll(MealPlanQueryCriteria criteria, boolean includeDetail);
```

说明：

- controller 层根据 `includeDetail` 分支调用
- service 实现内部返回两种明确结构
- 仅用于内部实现，controller 层仍应保持明确分支，不要直接暴露弱类型签名

方式 B，更稳妥，推荐：

```java
PageResult<MealPlan> queryAll(MealPlanQueryCriteria criteria);

PageResult<MealPlanListDetailVO> queryAllWithDetail(MealPlanQueryCriteria criteria);
```

推荐采用方式 B，原因：

- 更符合当前代码风格
- 方法语义明确
- Swagger、前端、IDE 类型提示都更稳定
- 不会让默认列表和详情列表共用一个模糊返回值

#### 1.4 Service 实现逻辑

核心思路：

1. 先分页查主表
2. 收集当前页 `mealPlanIds`
3. 批量查所有客户
4. 收集 `customerPlanIds`
5. 批量查所有菜品明细
6. 收集 `dishIds`
7. 批量查所有配料
8. 先分组，再装配

推荐伪代码：

```java
public PageResult<MealPlanListDetailVO> queryAllWithDetail(MealPlanQueryCriteria criteria) {
    Page<MealPlan> page = new Page<>(criteria.getPage(), criteria.getSize());
    Page<MealPlan> result = mealPlanMapper.selectPageByCriteria(criteria, page);
    List<MealPlan> plans = result.getRecords();
    if (plans.isEmpty()) {
        return new PageResult<>(Collections.emptyList(), result.getTotal());
    }

    List<Long> mealPlanIds = plans.stream()
            .map(MealPlan::getId)
            .collect(Collectors.toList());

    List<MealPlanCustomer> allCustomers = mealPlanCustomerMapper.selectByMealPlanIds(mealPlanIds);
    Map<Long, List<MealPlanCustomer>> customersByPlanId = allCustomers.stream()
            .collect(Collectors.groupingBy(MealPlanCustomer::getMealPlanId));

    List<Long> customerPlanIds = allCustomers.stream()
            .map(MealPlanCustomer::getId)
            .collect(Collectors.toList());

    Map<Long, List<MealPlanCustomerItem>> itemsByCustomerPlanId = Collections.emptyMap();
    if (!customerPlanIds.isEmpty()) {
        List<MealPlanCustomerItem> allItems = mealPlanCustomerItemMapper.selectByCustomerPlanIds(customerPlanIds);
        itemsByCustomerPlanId = allItems.stream()
                .collect(Collectors.groupingBy(MealPlanCustomerItem::getCustomerPlanId));
    }

    List<Integer> dishIds = itemsByCustomerPlanId.values().stream()
            .flatMap(List::stream)
            .map(MealPlanCustomerItem::getDishId)
            .filter(Objects::nonNull)
            .distinct()
            .collect(Collectors.toList());

    Map<Integer, List<DishIngredientItemVO>> ingredientsMap = buildDishIngredientsMap(dishIds);

    List<MealPlanListDetailVO> content = plans.stream()
            .map(plan -> assembleMealPlanListDetail(plan,
                    customersByPlanId.getOrDefault(plan.getId(), Collections.emptyList()),
                    itemsByCustomerPlanId,
                    ingredientsMap))
            .collect(Collectors.toList());

    return new PageResult<>(content, result.getTotal());
}
```

#### 1.5 抽取公共装配方法

当前 `queryMealPlanDetail(...)` 已经具备大部分组装逻辑，见现有 `MealPlanServiceImpl`。

建议抽出两个私有方法：

```java
private MealPlanDetailVO.CustomerPlanDetail assembleCustomerDetail(
        MealPlanCustomer customer,
        Map<Long, List<MealPlanCustomerItem>> itemsByCustomerPlanId,
        Map<Integer, List<DishIngredientItemVO>> ingredientsMap)

private MealPlanListDetailVO assembleMealPlanListDetail(
        MealPlan plan,
        List<MealPlanCustomer> customers,
        Map<Long, List<MealPlanCustomerItem>> itemsByCustomerPlanId,
        Map<Integer, List<DishIngredientItemVO>> ingredientsMap)
```

这样可以保证：

- 列表详情和单计划详情使用同一套客户明细装配逻辑
- 字段不会一边新增一边漏改
- 后续维护成本更低

#### 1.6 Controller 设计

当前 controller 方法是：

```java
public ResponseEntity<PageResult<MealPlan>> queryMealPlan(MealPlanQueryCriteria criteria)
```

建议改成显式分支：

```java
@GetMapping
public ResponseEntity<?> queryMealPlan(
        MealPlanQueryCriteria criteria,
        @RequestParam(required = false, defaultValue = "false") Boolean includeDetail) {
    if (Boolean.TRUE.equals(includeDetail)) {
        return new ResponseEntity<>(mealPlanService.queryAllWithDetail(criteria), HttpStatus.OK);
    }
    return new ResponseEntity<>(mealPlanService.queryAll(criteria), HttpStatus.OK);
}
```

说明：

- controller 使用 `ResponseEntity<?>` 可以接受，因为这里只是 HTTP 出口分支
- service 层不要变成 `PageResult<?>`
- 如果 Swagger 展示效果不理想，可考虑拆成两个 endpoint

#### 1.7 是否拆成新接口

如果前端长期稳定依赖“列表直接带详情”，更推荐拆出新接口：

```http
GET /api/meal-plan/detail-page
```

优点：

- 契约清晰
- Swagger 明确
- 不需要在同一接口上处理两种返回结构

缺点：

- 前端要改单独接口地址

结论：

- 如果追求最小改动，用 `includeDetail`
- 如果追求长期稳定维护，用新 endpoint

当前阶段建议先用 `includeDetail`，但实现时保留后续平滑拆分为新接口的空间。

## 前端改造建议

### API 层

现有列表接口调用方法可以增加可选参数：

```javascript
export function queryMealPlans(params, includeDetail = false) {
  return request({
    url: 'api/meal-plan',
    method: 'get',
    params: {
      ...params,
      includeDetail
    }
  })
}
```

### 页面层

建议只在“确实需要展开客户和菜品详情”的列表页传 `includeDetail: true`。

不建议：

- 所有列表场景都默认请求完整详情
- 一上来就把一页所有计划的全部客户和菜品都展开渲染

建议：

- 默认按当前页加载详情
- UI 上只在展开行时渲染客户表格和菜品标签
- 如果单页数据过大，再评估“先拿客户，不带菜品”的分级加载方案

## 测试方案

### 功能测试

| 场景 | 预期 |
|------|------|
| `includeDetail=false` | 返回原始 `PageResult<MealPlan>` |
| `includeDetail=true` | 返回 `PageResult<MealPlanListDetailVO>` |
| 当前页无数据 | 返回空数组，不报错 |
| 某计划无客户 | `customers=[]`，`totalCustomers=0` |
| 某客户无菜品 | `items=[]` |
| 菜品无配料 | `ingredients=[]` |

### 性能测试

观察指标：

- 单次请求响应时间
- SQL 查询次数
- JVM 堆占用变化
- 响应体大小

目标建议：

- 查询次数固定在 4 次以内
  - 主表分页
  - 批量查客户
  - 批量查菜品明细
  - 批量查配料
- 列表页单次请求响应时间在常规分页场景下保持可接受

说明：

- 文档里不建议写死 `< 500ms` 这类绝对值目标，实际应以测试环境、数据规模、网络环境为准
- 更关键的是“查询次数固定”和“性能随页数线性增长，不出现 N+1”

### 回归测试

- 原有 `GET /api/meal-plan` 默认调用不受影响
- 原有 `GET /api/meal-plan/{id}/detail` 行为不变
- 前端未传 `includeDetail` 的页面全部正常

## 风险评估

| 风险 | 等级 | 影响 | 缓解措施 |
|------|------|------|----------|
| 单次响应体过大 | Medium | 列表慢、前端渲染慢 | 限制分页大小，仅在必要页面传 `includeDetail=true` |
| 组装逻辑重复 | Medium | 后续字段漂移、维护困难 | 抽取公共装配方法，复用现有详情逻辑 |
| SQL 条件不一致 | Medium | 列表详情与单详情数据不一致 | 统一复用 mapper，统一 `deleted=0` 条件 |
| 返回结构双态 | Low | Swagger/前端类型不清晰 | service 分方法实现，必要时拆新 endpoint |

## 实施前确认项

- 前端当前列表页分页大小是多少，是否允许调小
- 列表页是否真的需要“客户 + 菜品 + 配料”完整结构，还是只需要客户层级
- 是否接受一个接口返回两种结构；如果不接受，直接拆新接口
- 当前 `MealPlanCustomerItemMapper.xml` 是否已经过滤软删数据

---

## 验收标准

1. 默认列表接口返回结构不变。
2. `includeDetail=true` 时，列表接口不再依赖逐条调用详情接口。
3. 列表详情场景下，数据库查询次数固定，不出现 N+1。
4. 列表详情与单计划详情在客户、菜品、配料字段上保持一致。
5. 前端目标页面完成适配，加载成功且无明显性能回退。
