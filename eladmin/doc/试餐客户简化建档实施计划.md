# 试餐客户简化建档实施计划

## 1. 背景与目标

当前新增客户会同时创建客户档案和首单，普通订单要求填写客户姓名、手机号、地址、首单父套餐、餐数、开始日期、菜品配置等信息。试餐客户实际录入场景更轻量，业务希望当首单父套餐为“试餐类型”时，尽量减少录入字段，但仍保留手机号、地址和父套餐这三个核心信息。

本次目标：

- 当新增客户选择的父套餐属于试餐类型时，允许最小入参只保留手机号、至少一个有效地址和首单父套餐。
- 后端负责补齐必要默认值，保证数据可落库、可生成客户编号、可形成试餐订单，并可被后续“试餐成单”关联。
- 普通套餐新增客户的现有校验规则保持不变。
- 前端根据父套餐动态切换简化录入体验，但后端仍作为最终规则兜底。

## 2. 试餐类型判定规则

沿用系统现有试餐订单判断口径：

- 通过 `orderInfo.parentPackageId` 查询 `parent_package`。
- `parent_package.package_name` 包含 `试餐` 时，视为试餐类型父套餐。

原因：

- 当前订单“试餐成单”关联逻辑已经使用“父套餐名称包含试餐”作为可关联试餐订单的判断条件。
- 本次先保持口径一致，避免新增一套试餐标识导致历史数据和现有查询逻辑不一致。

后续优化方向：

- 如套餐命名不稳定，可在 `parent_package` 增加显式字段，例如 `package_type` 或 `trial_flag`，再统一迁移相关判断逻辑。

## 3. 业务规则

### 3.1 普通套餐新增

保持现状：

- 客户姓名必填。
- 手机号必填且格式正确。
- 至少一个有效地址。
- 首单信息必填。
- 首单父套餐必填且启用。
- 早餐数与午晚餐数至少填写一个，且总餐数大于 0。
- 开始日期必填。
- 餐次类型与开始餐次必须匹配。
- 菜品配置数量不能为负数，米饭数量不能超过 10。

### 3.2 试餐套餐新增

最小请求体示例：

```json
{
  "phone": "13800000000",
  "addresses": [
    { "addressType": "DEFAULT", "addressDetail": "成都市高新区天府大道1号" }
  ],
  "orderInfo": {
    "parentPackageId": 1
  }
}
```

后端自动补齐：

| 字段 | 默认值 | 说明 |
|------|--------|------|
| `customerName` | `试餐客户` + 手机号后 4 位 | 例如 `试餐客户0000`，避免客户姓名为空 |
| `breakfastCount` | `0` | 默认不含早餐 |
| `lunchDinnerCount` | `1` | 默认创建 1 次午晚餐试餐订单 |
| `totalCount` | `1` | 根据餐数计算 |
| `breakfastPrice` | `0` | 金额默认 0 |
| `lunchDinnerPrice` | `0` | 金额默认 0 |
| `totalAmount` | `0` | 金额默认 0 |
| `depositAmount` | `0` | 金额默认 0 |
| `finalAmount` | `0` | 金额默认 0 |
| `scheduleMode` | `SCHEDULE` | 默认指定日期送 |
| `startDate` | 当前日期 | 使用服务器当前日期，格式 `yyyy-MM-dd` |
| `startMealType` | `LUNCH` | 午晚餐订单默认从午餐开始 |
| `mealType` | `LUNCH_DINNER` | 试餐默认使用午晚餐餐数池 |
| `mainDishCount` | `0` | 菜品配置默认 0 |
| `sideDishCount` | `0` | 菜品配置默认 0 |
| `vegCount` | `0` | 菜品配置默认 0 |
| `riceCount` | `1` | 沿用订单默认米饭数量 |
| `riceType` | `白米饭` | 沿用订单默认米饭类型 |
| `soupCount` | `0` | 菜品配置默认 0 |
| `trialConverted` | `false` | 试餐订单本身不是试餐成单 |
| `trialOrderId` | `null` | 试餐订单本身不关联其他试餐订单 |

仍需校验：

- 手机号必填且格式正确。
- 至少一个有效地址。
- 父套餐必须存在且启用。
- 客户编号仍按父套餐编号池生成或校验手动编号。
- 如果前端传入了地址、过敏、排除菜品、排除日期等可选字段，仍按现有规则校验合法性。

## 4. 后端实施步骤

### 4.1 修改客户保存服务

文件：

- `eladmin/eladmin-system/src/main/java/me/zhengjie/modules/customer/profile/service/impl/CustomerProfileServiceImpl.java`

建议调整点：

1. 在 `normalizeAndValidate(CustomerProfileSaveDto dto, boolean createMode)` 中，新增试餐模式判断。
2. 新增私有方法：
   - `private boolean isTrialParentPackage(Long parentPackageId)`
   - `private ParentPackage requireEnabledParentPackage(Long parentPackageId)`
   - `private void applyTrialCustomerDefaults(CustomerProfileSaveDto dto)`
   - `private void applyTrialOrderDefaults(CustomerProfileSaveDto.OrderInfoDto orderInfo)`
3. 新增客户时，先根据 `dto.orderInfo.parentPackageId` 判断是否试餐。
4. 试餐模式下：
   - 跳过 `customerName` 非空校验，但保存前自动补默认姓名。
   - 地址仍保持必填校验，且至少一个地址明细非空。
   - 在进入首单餐数、开始日期校验前，先补齐试餐首单默认值。
5. 普通模式走现有校验，不改变错误提示。

实现建议：

```java
private boolean isTrialCreateMode(CustomerProfileSaveDto dto, boolean createMode) {
    return createMode
            && dto != null
            && dto.getOrderInfo() != null
            && isTrialParentPackage(dto.getOrderInfo().getParentPackageId());
}
```

注意事项：

- `normalizeAndValidateOrderInfo()` 当前会自己查询父套餐并校验启用。为避免重复查询，可以先保持重复查询，确保改动范围小。
- `saveFirstOrder()` 依赖 `orderInfo.startDate` 可解析，因此试餐默认值必须在调用前补齐。
- `resolveCustomerCode()` 依赖父套餐编号池，试餐套餐也必须配置编号池或沿用现有 sequential fallback。

### 4.2 方法注释要求

根据项目约定，新增或修改方法需要补充清晰方法注释。新增私有方法也建议添加简短说明：

- 说明试餐父套餐判定口径。
- 说明默认值补齐的业务含义。
- 说明哪些字段只在试餐简化建档时生效。

### 4.3 后端单元测试

优先新增或补充客户档案服务测试。如果当前项目测试基础不足，可至少补充 service 层单测或集成测试用例。

建议用例：

1. `createTrialCustomerWithPhoneAndParentPackageOnly_success`
   - 入参只包含手机号和试餐父套餐。
   - 断言客户档案创建成功。
   - 断言客户姓名自动生成。
   - 断言订单创建成功，餐数为 `breakfastCount=0`、`lunchDinnerCount=1`、`totalCount=1`。
   - 断言订单 `mealType=LUNCH_DINNER`、`scheduleMode=SCHEDULE`。
   - 测试结束清理新增客户、订单、地址等数据。

2. `createNormalCustomerWithPhoneAndParentPackageOnly_failed`
   - 入参只包含手机号和普通父套餐。
   - 断言仍返回“客户姓名不能为空”或“地址信息不能为空”等现有错误。

3. `createTrialCustomerWithInvalidPhone_failed`
   - 手机号格式错误。
   - 断言失败且错误为手机号格式相关。

4. `createTrialCustomerWithDisabledParentPackage_failed`
   - 父套餐停用。
   - 断言失败。

5. `createTrialCustomerWithManualCode_successOrConflict`
   - 如果传手动编号，仍按编号池规则校验。

测试数据清理：

- 遵守项目规则，只删除当前测试新增的数据。
- 建议在 `@After` / `@AfterEach` 中按记录 ID 精确清理。

## 5. 前端实施步骤

### 5.1 修改客户新增页面

文件：

- `eladmin-web/src/views/customer/profile/index.vue`
- `eladmin-web/src/components/Order/OrderForm.vue`

目标：

- 新增客户时，用户选择试餐父套餐后，页面切换为简化表单。
- 用户只需要填写手机号和父套餐即可保存。
- 普通套餐保持当前完整表单。

### 5.2 父套餐选择位置

当前父套餐选择在 `OrderForm` 内。为了让“客户基本信息区域”根据父套餐切换必填规则，有两种实现方式：

方案 A：保持父套餐在 `OrderForm` 内，父组件监听 `package-change`

- `OrderForm` 已经在 `onParentPackageChange(parentId)` 中 `$emit('package-change', parentId, null)`。
- `CustomerProfile` 给 `OrderForm` 增加监听：

```vue
<OrderForm
  ref="orderFormRef"
  v-model="form.orderInfo"
  mode="firstOrder"
  :readonly="false"
  @package-change="onFirstOrderPackageChange"
/>
```

- 父组件根据 `parentPackageId` 到 `OrderForm` 的 `parentPackages` 不方便直接取套餐名，因此建议由 `OrderForm` emit 时带出 parent 对象，或在父组件单独加载父套餐列表。

方案 B：父套餐选择上移到客户新增页

- 在客户基本信息区域展示父套餐。
- `OrderForm` 中仍保留父套餐字段但可在 `firstOrder` 模式下根据 prop 隐藏。
- 改动较大，不建议本次采用。

推荐方案 A，改动最小。

### 5.3 `OrderForm` 增强

文件：

- `eladmin-web/src/components/Order/OrderForm.vue`

建议新增：

- computed：

```js
selectedParentPackage() {
  return this.parentPackages.find(item => item.id === this.form.parentPackageId) || null
},
isTrialPackage() {
  return !!(this.selectedParentPackage && this.selectedParentPackage.packageName && this.selectedParentPackage.packageName.includes('试餐'))
}
```

- `onParentPackageChange(parentId)` emit 父套餐对象：

```js
const parent = this.parentPackages.find(item => item.id === parentId) || null
this.$emit('package-change', parentId, null, parent)
```

- 增加 prop：

```js
simplifiedTrialMode: {
  type: Boolean,
  default: false
}
```

- 在模板中，当 `mode === 'firstOrder' && simplifiedTrialMode` 时：
  - 保留父套餐选择。
  - 隐藏或折叠销售渠道、排餐模式、餐次类型、试餐成单、送餐日期、菜品配置、换菜规则、金额、开始日期、餐数等字段。

注意：

- 即使前端隐藏字段，也不要依赖前端默认值保证正确，后端必须兜底。
- 如果需要展示提示，建议使用轻量 `el-alert` 或表单旁提示：“试餐套餐将按 1 次午晚餐创建试餐订单”。避免大面积说明文本影响录入效率。

### 5.4 `CustomerProfile` 增强

文件：

- `eladmin-web/src/views/customer/profile/index.vue`

建议新增 data：

```js
firstOrderTrialPackage: false
```

新增方法：

```js
onFirstOrderPackageChange(parentId, childPackageId, parent) {
  this.firstOrderTrialPackage = !!(parent && parent.packageName && parent.packageName.includes('试餐'))
}
```

表单规则调整：

- `customerName` 必填规则改为自定义 validator：
  - 试餐模式允许为空。
  - 普通模式仍必填。
- `phone` 必填和格式规则不变。
- 地址不在 Element rules 中，但提交前后端会校验；试餐模式下地址区域仍需保留。

提交逻辑调整：

- `submitForm()` 中，试餐模式下不调用完整 `OrderForm.validate()`，或 `OrderForm` 自身根据 `simplifiedTrialMode` 跳过隐藏字段校验。
- `buildSubmitPayload()` 中，试餐模式只需要保证：

```js
payload.orderInfo = {
  parentPackageId: orderInfo.parentPackageId
}
```

也可以继续传当前默认值，但建议最小化传参，让后端默认规则成为唯一事实来源。

UI 展示建议：

- 试餐模式下显示字段：
  - 手机号
  - 地址信息
  - 父套餐
  - 客户编号（可选，不填自动生成）
  - 备注（可选）
- 隐藏字段：
  - 客户姓名
  - 孕周
  - 过敏食物
  - 排除菜品
  - 排除日期
  - 医嘱要求
  - 特殊要求
  - 生产日期
  - 首单详细字段

如果业务希望试餐也能补充更多偏好信息，可以改为折叠“更多信息”，默认收起。

## 8. 验收标准

### 8.1 后端接口验收

1. 试餐父套餐下，只传手机号和父套餐 ID，接口返回成功。
2. 创建成功后客户列表能看到该客户。
3. 客户详情能正常打开。
4. 订单管理能查到该客户自动生成的试餐订单。
5. 试餐订单能被“试餐成单”关联订单接口搜索到。
6. 普通父套餐下，同样最小入参仍失败。
7. 手机号为空或格式错误时仍失败。

### 8.2 前端页面验收

1. 新增客户弹窗打开后默认仍是完整模式。
2. 选择试餐父套餐后，页面切换为简化录入。
3. 简化录入填写手机号、至少一个有效地址和父套餐可保存成功。
4. 切回普通父套餐后，完整字段和校验恢复。
5. 新增成功后列表刷新，客户编号、手机号可见。
6. 不出现隐藏字段校验导致无法提交的问题。

### 8.3 回归验收

1. 普通客户新增流程不受影响。
2. 编辑客户档案流程不受影响。
3. 订单新增/编辑流程不受影响。
4. 试餐成单关联订单流程不受影响。
5. 客户编号生成逻辑不受影响。

## 9. 风险与处理

| 风险 | 影响 | 处理 |
|------|------|------|
| 试餐套餐命名不规范 | 无法识别试餐模式 | 短期要求套餐名包含“试餐”；长期增加显式字段 |
| 试餐套餐未配置编号池 | 客户编号生成失败 | 运营需为试餐套餐配置编号池，或沿用现有 sequential fallback |
| 前端隐藏字段但仍触发校验 | 简化模式无法提交 | `submitForm()` 和 `OrderForm.validate()` 同步增加试餐模式分支 |
| 默认创建 1 次午晚餐不符合部分试餐场景 | 餐数口径偏差 | 如存在早餐试餐或多餐试餐，再增加可选餐数配置 |
| 试餐客户缺少地址 | 无法建档或后续配送异常 | 当前规则保留地址必填，避免生成缺少配送基础信息的客户和订单 |

## 10. 推荐开发顺序

1. 后端增加试餐识别和默认值补齐。
2. 后端补充测试或至少完成接口自测。
3. 前端增加试餐父套餐识别。
4. 前端调整简化模式展示和提交校验。
5. 更新 API 文档和业务文档。
6. 启动前后端做页面验收。
7. 回归普通客户新增、订单新增、试餐成单关联。
