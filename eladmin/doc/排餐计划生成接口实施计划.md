## 需求确认汇总

| # | 问题 | 结论 |
|---|------|------|
| Q1 | 幂等策略 | 先软删除/标记 `(record_date, meal_type)` 的旧记录，再生成新记录 |
| Q2 | 候选菜规则 | `meatCount=1` 时 `MAIN` 优先、`SIDE` 兜底；`meatCount=2` 时固定选择 `MAIN + SIDE`；`meatCount` 当前业务约束最多为 `2`；`vegCount` 当前业务约束为 `0/1` |
| Q3 | 失败条件 | 过敏过滤后不满足子套餐要求 → 直接记录为失败，不降级 |
| Q4 | 过敏匹配 | `allergy_tags` 与 `dish_ingredient.name` **完全相等** |
| Q5 | 医嘱要求 | 仅展示，不参与选菜逻辑 |
| Q6 | N+1 优化 | 先批量加载当天候选菜的配料，内存过滤 |
| Q7 | 旧表处理 | 不使用 `dish_schedule_record`，新建独立表 |
| Q8 | 同类菜品重复 | 同一客户同一餐次内，不允许重复分配同一道菜 |
| Q9 | 套餐过滤 | 候选菜池必须按 `meal_packages` 过滤，仅使用该订单所属父套餐可用菜品 |

## Phase 1：数据库表（3 张新表）

### 1.1 meal_plan — 排餐计划主表

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | BIGINT | 主键，自增 |
| `record_date` | DATE | 排餐日期 |
| `meal_type` | VARCHAR(20) | 餐次：`LUNCH` / `DINNER` |
| `total_count` | INT | 当天该餐次总订单数 |
| `success_count` | INT | 成功排餐数 |
| `fail_count` | INT | 失败数 |
| `status` | VARCHAR(20) | 生成状态：`GENERATING` / `SUCCESS` / `FAILED` |
| `generate_time` | DATETIME | 生成时间 |
| `create_by` | VARCHAR(64) | 生成人 |
| `update_time` | DATETIME | 更新时间 |

**唯一约束**：`UNIQUE(record_date, meal_type)`，保证幂等

### 1.2 meal_plan_customer — 客户排餐结果表

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | BIGINT | 主键，自增 |
| `meal_plan_id` | BIGINT | 关联 `meal_plan.id` |
| `customer_id` | BIGINT | 客户ID |
| `customer_name` | VARCHAR(64) | 客户姓名（冗余存储） |
| `phone` | VARCHAR(20) | 手机号（冗余存储） |
| `order_id` | BIGINT | 对应订单ID |
| `parent_package_id` | BIGINT | 父套餐ID（冗余存储） |
| `child_package_id` | BIGINT | 子套餐ID（冗余存储） |
| `status` | TINYINT | 状态：0=失败，1=成功 |
| `fail_reason` | VARCHAR(255) | 失败原因（如"主菜不足，需2道，实际1道"） |
| `meat_required_count` | INT | 子套餐要求的荤菜数量 |
| `veg_required_count` | INT | 子套餐要求的素菜数量 |
| `include_soup` | TINYINT | 是否需要汤 |
| `include_rice` | TINYINT | 是否需要米饭 |
| `create_time` | DATETIME | 创建时间 |

**索引**：`INDEX(meal_plan_id)`, `INDEX(customer_id)`, `INDEX(order_id)`

### 1.3 meal_plan_customer_item — 客户排餐菜品表

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | BIGINT | 主键，自增 |
| `customer_plan_id` | BIGINT | 关联 `meal_plan_customer.id` |
| `dish_type` | VARCHAR(20) | 菜品类型：`MAIN` / `SIDE` / `VEGETABLE` / `SOUP` / `RICE` |
| `dish_id` | INT | 菜品ID |
| `dish_name` | VARCHAR(64) | 菜品名称（冗余存储） |
| `seq` | INT | 同类型菜品顺序，从 1 开始 |
| `create_time` | DATETIME | 创建时间 |

> 由于 `meatCount` 存在取 `2` 的场景，客户维度结果与菜品明细拆表，避免使用定宽字段导致无法表达 `MAIN + SIDE` 的组合。

**索引**：`INDEX(customer_plan_id)`

### 1.4 DDL

```sql
CREATE TABLE meal_plan (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    record_date DATE NOT NULL,
    meal_type VARCHAR(20) NOT NULL,
    total_count INT DEFAULT 0,
    success_count INT DEFAULT 0,
    fail_count INT DEFAULT 0,
    status VARCHAR(20) NOT NULL DEFAULT 'GENERATING',
    generate_time DATETIME NOT NULL,
    create_by VARCHAR(64) DEFAULT '',
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_date_meal (record_date, meal_type)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE meal_plan_customer (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    meal_plan_id BIGINT NOT NULL,
    customer_id BIGINT NOT NULL,
    customer_name VARCHAR(64) NOT NULL DEFAULT '',
    phone VARCHAR(20) NOT NULL DEFAULT '',
    order_id BIGINT NOT NULL,
    parent_package_id BIGINT NOT NULL DEFAULT 0,
    child_package_id BIGINT NOT NULL DEFAULT 0,
    status TINYINT NOT NULL DEFAULT 0 COMMENT '0=失败,1=成功',
    fail_reason VARCHAR(255) DEFAULT '',
    meat_required_count INT NOT NULL DEFAULT 0,
    veg_required_count INT NOT NULL DEFAULT 0,
    include_soup TINYINT NOT NULL DEFAULT 0,
    include_rice TINYINT NOT NULL DEFAULT 0,
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_meal_plan_id (meal_plan_id),
    INDEX idx_customer_id (customer_id),
    INDEX idx_order_id (order_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE meal_plan_customer_item (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    customer_plan_id BIGINT NOT NULL,
    dish_type VARCHAR(20) NOT NULL,
    dish_id INT NOT NULL,
    dish_name VARCHAR(64) NOT NULL DEFAULT '',
    seq INT NOT NULL DEFAULT 1,
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_customer_plan_id (customer_plan_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

## Phase 3：后端 — 核心 Service（2 个新文件）

### 3.1 ScheduleKeyUtil.java — 工具类

**路径**：`modules/meal/util/ScheduleKeyUtil.java`

**职责**：根据日期计算 `scheduleKey`

```
输入：adate（格式 yyyy-MM-dd）
输出：scheduleKey 格式 "W-D"（W=第几周1-4，D=星期几1-7）

计算规则（按每月固定7天分段）：
  第1周: 该月第 1-7 天
  第2周: 第 8-14 天
  第3周: 第 15-21 天
  第4周: 第 22-28 天（第5周不足7天也归入第4周）

示例：
  2026-04-01 → 第1周 → 周二(2) → "1-2"
  2026-04-08 → 第2周 → 周三(3) → "2-3"
  2026-04-25 → 第4周 → 周六(6) → "4-6"
```

### 3.2 MealPlanService.java — 接口定义

```java
public interface MealPlanService {
    /**
     * 生成排餐计划
     * @param recordDate 日期（yyyy-MM-dd）
     * @param mealType 餐次（LUNCH / DINNER）
     * @return 生成结果
     */
    MealPlanGenerateResult generateMealPlan(String recordDate, String mealType);
}
```

### 3.3 MealPlanServiceImpl.java — 核心实现

**依赖注入**：
- `DishMapper`、`DishIngredientMapper`
- `CustomerOrderMapper`
- `CustomerProfileMapper`
- `SubPackageMapper`
- `MealPlanMapper`、`MealPlanCustomerMapper`、`MealPlanCustomerItemMapper`
- `ParentPackageMapper`

**`generateMealPlan(recordDate, mealType)` 完整流程**：

```
Step 1: 参数校验
  - recordDate 非空，格式正确（yyyy-MM-dd）
  - mealType ∈ {LUNCH, DINNER}
  - 幂等：查询 UNIQUE(record_date, meal_type)
    - 有旧记录 → DELETE meal_plan_customer_item + DELETE meal_plan_customer + DELETE meal_plan
  - 整个 generateMealPlan 使用事务包裹

Step 2: 生成当天候选菜池
  - DishMapper 查询 enabled=1 的菜品
  - 过滤：meal_types JSON 包含 mealType
  - 过滤：schedule JSON 包含 calcScheduleKey(recordDate)
  - 过滤：meal_packages JSON 包含订单 parentPackageId
  - 按 (parentPackageId, dish_type) 分组缓存
  - 每组按 sort 升序排序，作为候选列表，不截断为单个菜品

Step 3: 批量加载候选菜配料（N+1 优化）
  - 一次性 IN 查询所有候选菜 ID 对应的 dish_ingredient_relation
  - 组装 Map<dishId, List<ingredientName>>

Step 4: 查询有效订单
  - CustomerOrderMapper.selectList(
      status=1,
      remaining_count > 0,
      start_date <= recordDate <= end_date
    )
  - 过滤：meal_type ∈ {mealType, 'ALL'}
  - 过滤：schedule_mode
      - DAILY → 通过
      - WEEKDAY → isWeekday(recordDate) ? 通过 : 跳过
      - WEEKEND → isWeekend(recordDate) ? 通过 : 跳过
      - SCHEDULE → recordDate ∈ deliveryDates JSON ? 通过 : 跳过
  - 批量加载 customer_profile，组装 Map<customerId, CustomerProfile>
  - 批量加载 sub_package，组装 Map<childPackageId, SubPackage>
  - totalCount = 过滤后的有效订单数

Step 5: 创建 meal_plan 主记录（status = GENERATING, totalCount = 有效订单数）

Step 6: 逐单分配菜品

  6a: 加载客户信息
      - 从 customerProfileMap 获取 customerId 对应客户
      - 取 allergy_tags（List<String>）

  6b: 加载子套餐规则
      - 从 subPackageMap 获取 childPackageId 对应子套餐
      - 取 meatCount, vegCount, includeSoup, includeRice

  6c: 从 parentPackageId 对应候选池选菜
      荤菜：
        - meatCount = 1 → 先取 MAIN 候选列表，遍历直到不过敏
        - MAIN 无可用菜 → 再取 SIDE 候选列表兜底 1 道
        - meatCount = 2 → 必须分别选出 1 道 MAIN 和 1 道 SIDE，二者均需不过敏且不重复
        - MAIN 或 SIDE 任一不足 → 失败原因"荤菜不足，需MAIN+SIDE"
        - meatCount > 2 → 视为非法套餐配置，直接记录失败或抛参数异常
      素菜：
        - vegCount >= 1 → 取 VEGETABLE 候选列表，遍历直到不过敏且不重复
        - 无候选 → 失败原因"素菜不足"
        - 全过敏 → 失败原因"素菜过敏"
      汤（如需）：
        - includeSoup=true → 取 SOUP 候选，遍历直到不过敏且不重复
        - 无候选 → 失败原因"汤不足"
        - 全过敏 → 失败原因"汤过敏"
      米饭（如需）：
        - includeRice=true → 取 RICE 候选，遍历直到不过敏且不重复
        - 无候选 → 失败原因"米饭不足"
        - 全过敏 → 失败原因"米饭过敏"

  6d: 过敏过滤逻辑
      - 从批量加载的配料 Map 中取该菜的配料列表
      - 配料.name 与 allergy_tags 任一完全相等 → 该菜不可用，尝试下一个
      - 同一客户同一餐次内不允许重复分配同一道菜
      - 该类别最终可用数量不足 → 记录对应失败原因

  6e: 满足要求（所有类别均有可用菜）
      → 写 meal_plan_customer（status=1）
      → 遍历已选菜品，写 meal_plan_customer_item（dishType, dishId, dishName, seq）

  6f: 不满足要求（任一类别无候选或全过敏）
      → 写 meal_plan_customer（status=0, fail_reason=具体原因）

Step 7: 更新 meal_plan.success_count / fail_count / status=SUCCESS

Step 8: 返回生成结果
```

## 风险评估

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| `scheduleKey` 计算规则理解有误 | **高** | 先和业务方确认"第几周"的定义（按月/按自然周/按固定7天分段） |
| `meatCount = 2` 导致定宽字段无法表达 `MAIN + SIDE` | **高** | 改为 `meal_plan_customer` + `meal_plan_customer_item` 两层结构 |
| 候选菜未按套餐过滤导致串套餐 | **高** | 候选池必须按 `parentPackageId` + `mealType` + `scheduleKey` 建立缓存 |
| 批量加载配料时 IN 列表过长 | 中 | 候选菜通常 < 100 道，风险低；若超限可分批 |
| 定时任务重复触发 | 中 | 接口内部支持幂等（先删后插），外部调度器需保证不并发 |
| 失败原因区分不够精确 | 低 | 区分"XXX不足（无候选）"和"XXX过敏（全过滤）"两种语义 |

## 复杂度评估

**整体：Medium**

| 维度 | 评分 | 说明 |
|------|------|------|
| 后端逻辑复杂度 | Medium-High | 候选池生成、订单多条件过滤、过敏交叉匹配、`meatCount=2` 时 `MAIN + SIDE` 组合分配 |
| 前端变更 | 无 | 纯后端接口 |
| 测试复杂度 | Medium | 需要覆盖 `meatCount=1`、`meatCount=2`、套餐过滤、过敏失败等场景 |
| 文档工作量 | Low | 新建一个 Markdown |
