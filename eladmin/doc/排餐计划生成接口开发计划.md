## 一、最终技术方案

### 1.1 开发目标

新增一个独立的“排餐计划生成接口”，按**日期 + 餐次**生成客户排餐结果，并落库到新表，不复用现有 `dish_schedule_record` / `customer_menu_record` 逻辑。

本次开发范围：

1. 新建 3 张排餐计划表
2. 新增 meal 模块实体、Mapper、Service、Controller、DTO、工具类
3. 实现生成接口 `POST /api/meal-plan/generate`
4. 新增接口文档
5. 补充最少可验收测试

---

### 1.2 scheduleKey 最终规则

`scheduleKey` 用于**匹配菜品表 `dish.schedule` 字段**，决定某天哪些菜能进入候选池。

规则如下：

- 1-7 日 → 第 1 周
- 8-14 日 → 第 2 周
- 15-21 日 → 第 3 周
- 22-28 日 → 第 4 周
- 29-31 日 → **映射为第 1 周**

星期规则：

- 周一 = 1
- 周二 = 2
- ...
- 周日 = 7

最终格式：

```text
W-D
```

示例：

- `2026-04-01` → `1-3`
- `2026-04-08` → `2-3`
- `2026-04-25` → `4-6`
- `2026-04-29` → `1-3`

建议通过 `ScheduleKeyUtil` 统一计算，避免和现有 `DishServiceImpl` 内部逻辑重复实现。

---

### 1.3 幂等最终实现

幂等采用：**软删除旧记录 + 重新生成新记录**。

事务内流程：

1. 根据 `record_date + meal_type + deleted=0` 查询当前生效主记录
2. 若存在旧记录：
   - 软删除 `meal_plan_customer_item`
   - 软删除 `meal_plan_customer`
   - 软删除 `meal_plan`
3. 插入新的主记录及明细记录

说明：

- 因为采用软删除，所以三张表都必须带 `deleted` 字段
- 不使用 `UNIQUE(record_date, meal_type)`
- 也不使用 `UNIQUE(record_date, meal_type, deleted)`，否则会限制历史软删除记录数量
- 改为普通组合索引提高查询效率

### 1.4 核心业务流程

#### Step 1：参数校验

校验：

- `recordDate` 非空，格式为 `yyyy-MM-dd`
- `mealType` 只能是 `LUNCH` / `DINNER`

参数错误时抛：

```java
throw new BadRequestException("...")
```

#### Step 2：幂等软删除

根据 `(record_date, meal_type, deleted=0)` 查询有效主记录：

- 有旧记录 → 软删除三张表对应旧数据
- 无旧记录 → 直接进入生成流程

#### Step 3：计算 scheduleKey

根据 `recordDate` 计算 `scheduleKey`，用于筛选当天可用菜品。

#### Step 4：查询有效订单

有效订单过滤条件：

- `status = 1`
- `remaining_count > 0`
- `start_date <= recordDate <= end_date`
- `meal_type in (mealType, ALL)`
- `schedule_mode` 符合当前日期：
  - `DAILY`
  - `WEEKDAY`
  - `WEEKEND`
  - `SCHEDULE` 且 `delivery_dates` 包含当天

#### Step 5：批量加载关联数据

一次性批量加载：

- 客户信息
- 子套餐
- 父套餐
- 候选菜
- 候选菜配料

避免 N+1 查询。

#### Step 6：构建候选菜池

候选菜池按以下维度缓存：

```java
Map<Long, Map<String, List<Dish>>> candidateDishMap
```

含义：

- `Long` = `parentPackageId`
- `String` = `dishType`
- `List<Dish>` = 当前父套餐下该类型可选菜品列表

候选菜过滤条件：

- `enabled = 1`
- `meal_types` 包含当前 `mealType`
- `schedule` 包含当前 `scheduleKey`
- `meal_packages` 包含订单 `parentPackageId`

#### Step 7：逐订单排餐

对每个订单：

1. 取客户过敏标签
2. 取子套餐规则
3. 从候选池中按规则选菜
4. 同一客户同一餐次内不允许重复同一道菜

选菜规则：

##### 荤菜

- `meatCount = 1`
  - 优先取 `MAIN`
  - `MAIN` 无可用菜时用 `SIDE` 兜底
- `meatCount = 2`
  - 必须选出 `MAIN + SIDE`
  - 任一缺失则失败

##### 素菜

- `vegCount >= 1` 时，从 `VEGETABLE` 中选

##### 汤

- `includeSoup = true` 时，从 `SOUP` 中选

##### 米饭

- `includeRice = true` 时，从 `RICE` 中选

#### Step 8：过敏过滤逻辑

过敏匹配规则：

- `allergy_tags` 与 `dish_ingredient.name` **完全相等** 时，视为该菜不可用
- 不使用 `contains`
- 不做模糊匹配

#### Step 9：保存结果

成功时写入：

- `meal_plan_customer`
- `meal_plan_customer_item`

失败时写入：

- `meal_plan_customer`

#### Step 10：回写主表统计

更新：

- `success_count`
- `fail_count`
- `status`

状态建议：

- `GENERATING`
- `SUCCESS`
- `FAILED`

## 二、最新 DDL

建议新增文件：`sql/meal_plan.sql`

```sql
create table meal_plan (
    id bigint auto_increment primary key,
    record_date date not null,
    meal_type varchar(20) not null,
    total_count int not null default 0,
    success_count int not null default 0,
    fail_count int not null default 0,
    status varchar(20) not null default 'GENERATING',
    generate_time datetime not null,
    create_by varchar(64) default '',
    update_by varchar(64) default '',
    deleted bit(1) not null default b'0',
    create_time datetime default current_timestamp,
    update_time datetime default current_timestamp on update current_timestamp,
    index idx_record_date_meal_deleted (record_date, meal_type, deleted),
    index idx_generate_time (generate_time)
) engine=innodb default charset=utf8mb4;

create table meal_plan_customer (
    id bigint auto_increment primary key,
    meal_plan_id bigint not null,
    customer_id bigint not null,
    customer_name varchar(64) not null default '',
    phone varchar(20) not null default '',
    order_id bigint not null,
    parent_package_id bigint not null default 0,
    child_package_id bigint not null default 0,
    status tinyint not null default 0 comment '0=失败,1=成功',
    fail_reason varchar(255) default '',
    meat_required_count int not null default 0,
    veg_required_count int not null default 0,
    include_soup tinyint not null default 0,
    include_rice tinyint not null default 0,
    deleted bit(1) not null default b'0',
    create_time datetime default current_timestamp,
    update_time datetime default current_timestamp on update current_timestamp,
    index idx_meal_plan_id (meal_plan_id),
    index idx_customer_id (customer_id),
    index idx_order_id (order_id),
    index idx_meal_plan_order (meal_plan_id, order_id),
    index idx_meal_plan_status (meal_plan_id, status)
) engine=innodb default charset=utf8mb4;

create table meal_plan_customer_item (
    id bigint auto_increment primary key,
    customer_plan_id bigint not null,
    dish_type varchar(20) not null,
    dish_id int not null,
    dish_name varchar(64) not null default '',
    seq int not null default 1,
    deleted bit(1) not null default b'0',
    create_time datetime default current_timestamp,
    update_time datetime default current_timestamp on update current_timestamp,
    index idx_customer_plan_id (customer_plan_id),
    index idx_dish_id (dish_id)
) engine=innodb default charset=utf8mb4;
```

## 五、风险与注意事项

### 5.1 scheduleKey 规则与现有逻辑不一致

现有 `DishServiceImpl` 中周次计算仍是 1-5 周逻辑；本次新接口按“第 5 周映射第 1 周”处理。

当前建议：

- 新接口独立使用 `ScheduleKeyUtil`
- 暂不改动旧排餐逻辑
- 后续若要统一全系统，再整体收敛

### 5.2 meal_packages 字段类型兼容

参考现有 `DishServiceImpl` 实现，`dish.meal_packages` 可能混有：

- 数字 ID
- 字符串数字
- 字符串套餐编码

新接口匹配父套餐时要兼容这三类数据。

### 5.3 过敏匹配规则不能复用旧逻辑

现有逻辑中存在 `contains` 匹配；本次需求必须使用：

- `allergy_tags` 与 `ingredientName` **完全相等**

因此需要在新实现中单独处理。
