# 排餐计划表结构切分实施文档

## 一、背景与目标

### 现状问题

当前 `dish` 表承担了双重职责：
1. **菜品物料信息**：存放"红烧肉是什么"（名称、图片、做法、配料）
2. **排期调度信息**：通过 JSON 字段存储"红烧肉在哪天出现"

存在的问题：
- `schedule` 字段格式为 `["1-3", "2-5"]`，查询通过 `JSON_CONTAINS` 字符串匹配，**无法建索引**，每次查询全表扫描
- `meal_types` 同理，无法建索引
- 排期数据分散在各条菜品记录中，无法独立管理（增删改都要操作整条 dish 记录）
- 前端无法直观地看到"周几哪个坑位放了哪道菜"

### 目标

将排期信息从 `dish` 表剥离到独立的 `meal_schedule_plan` 表，实现：
- **坑位显式化**：每个坑位（week_num + day_of_week + meal_time + dish_category）对应一条记录
- **查询可索引**：独立字段可建普通索引，替代 JSON 函数查询
- **职责分离**：dish 表只负责"菜品是什么"，排期由独立表管理

## 二、数据库改造

### 步骤 A：`dish` 表精简

**移除字段**（迁移到新表）：

| 字段 | 说明 | 迁移目标 |
|------|------|----------|
| `schedule` | 排期 JSON 数组（如 `["1-3", "2-5"]`） | → `meal_schedule_plan.schedule` |
| `meal_types` | 适用餐次 JSON 数组（如 `["LUNCH", "DINNER"]`） | → `meal_schedule_plan.meal_time` |
| `dish_type` | 菜品类型（MAIN/SIDE/SOUP 等） | → `meal_schedule_plan.dish_category` |

**保留字段**：

| 字段 | 说明 |
|------|------|
| `id` | 主键 |
| `name` | 菜品名称 |
| `cooking_method` | 做法/流程 |
| `ingredients` | 配料描述 |
| `image_url` | 图片路径 |
| `meal_packages` | 所属套餐（JSON 数组，**保留在 dish 表**） |
| `sort` | 排序号 |
| `enabled` | 是否启用 |
| `create_time` | 创建时间 |
| `update_time` | 更新时间 |

**迁移脚本**：

```sql
-- 从 dish.schedule 和 dish.meal_types 提取数据写入新表 (双层展开)
INSERT INTO meal_schedule_plan (week_num, day_of_week, meal_time, dish_category, dish_id, enabled, create_time, update_time)
SELECT
  CAST(SUBSTRING_INDEX(s.split_value, '-', 1) AS UNSIGNED) AS week_num,
  CAST(SUBSTRING_INDEX(s.split_value, '-', -1) AS UNSIGNED) AS day_of_week,
  m.meal_type AS meal_time,
  d.dish_type AS dish_category,
  d.id AS dish_id,
  COALESCE(d.enabled, 1) AS enabled,
  NOW() AS create_time,
  NOW() AS update_time
FROM dish d
CROSS JOIN JSON_TABLE(
  d.schedule,
  '$[*]' COLUMNS (split_value VARCHAR(10) PATH '$')
) AS s
CROSS JOIN JSON_TABLE(
  d.meal_types,
  '$[*]' COLUMNS (meal_type VARCHAR(20) PATH '$')
) AS m
WHERE d.schedule IS NOT NULL AND JSON_LENGTH(d.schedule) > 0;
```

> **注意**：`meal_packages` 字段建议保留在 dish 表，这是菜品的固有属性（"这道菜属于哪个套餐"），暂不改为关联表方式。

### 步骤 B：创建 `meal_schedule_plan` 表

```sql
CREATE TABLE meal_schedule_plan (
    id            BIGINT AUTO_INCREMENT PRIMARY KEY COMMENT '主键ID',
    week_num      TINYINT NOT NULL COMMENT '周序号(1-4)',
    day_of_week   TINYINT NOT NULL COMMENT '星期几(1-7)',
    meal_time     VARCHAR(20) NOT NULL COMMENT 'LUNCH午餐/DINNER晚餐',
    dish_category VARCHAR(20) NOT NULL COMMENT '菜品坑位分类：MAIN主菜/SIDE副菜/SOUP汤品/VEGETABLE蔬菜/RICE米饭',
    dish_id       INT NOT NULL COMMENT '关联dish表主键',
    enabled       TINYINT DEFAULT 1 COMMENT '是否启用：0禁用/1启用',
    create_time   TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time   TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    UNIQUE KEY uk_slot (week_num, day_of_week, meal_time, dish_category),
    KEY idx_dish_id (dish_id),
    KEY idx_enabled (enabled)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='餐品菜单计划映射表';
```

**设计说明**：
- **唯一索引** `uk_slot(week_num, day_of_week, meal_time, dish_category)`：保证每个坑位只放一道菜
- `dish_category` 替代原 `dish.dish_type`，表示该菜品在此坑位中的角色
- `meal_time` 从原 `dish.meal_types` JSON 数组转为单值枚举

### 步骤 C：清理 `dish` 表废弃字段

```sql
-- 确认迁移完成后再执行
ALTER TABLE dish
  DROP COLUMN schedule,
  DROP COLUMN meal_types,
  DROP COLUMN dish_type;
```

### 核心实体类

```java
@Data
@TableName("meal_schedule_plan")
public class MealSchedulePlan implements Serializable {

    @TableId(value = "id", type = IdType.AUTO)

    @ApiModelProperty(value = "LUNCH午餐/DINNER晚餐/ALL全部")
    private String mealTime;

    @ApiModelProperty(value = "MAIN主菜/SIDE副菜/SOUP汤品/VEGETABLE蔬菜/RICE米饭")

    @ApiModelProperty(value = "关联dish表主键")

    private Boolean enabled;

    @TableField("create_time")
    private Timestamp createTime;
```

### 核心 Mapper SQL

```xml
<select id="findBySchedule" resultMap="DishMapper.BaseResultMap">
    SELECT d.*
    FROM meal_schedule_plan msp
    JOIN dish d ON d.id = msp.dish_id
    WHERE msp.enabled = 1
      AND msp.week_num = #{week}
      AND msp.day_of_week = #{day}
      AND msp.meal_time = #{mealType}
    ORDER BY d.sort ASC
</select>

<select id="findByWeek" resultMap="BaseResultMap">
    SELECT msp.*, d.id AS dish_id, d.name AS dish_name, d.image_url
    FROM meal_schedule_plan msp
    JOIN dish d ON d.id = msp.dish_id
    WHERE msp.enabled = 1
      AND msp.week_num = #{weekNum}
    ORDER BY msp.day_of_week ASC,
             FIELD(msp.meal_time, 'BREAKFAST', 'LUNCH', 'DINNER') ASC,
             FIELD(msp.dish_category, 'SOUP', 'MAIN', 'SIDE', 'VEGETABLE', 'RICE') ASC
</select>

<insert id="insert">
    INSERT INTO meal_schedule_plan (week_num, day_of_week, meal_time, dish_category, dish_id, enabled)
    VALUES (#{weekNum}, #{dayOfWeek}, #{mealTime}, #{dishCategory}, #{dishId}, #{enabled})
</insert>

<delete id="deleteById">
    DELETE FROM meal_schedule_plan WHERE id = #{id}
</delete>
```

### 改造 `MealPlanServiceImpl`

**仅改动一处方法**，其余调用链完全不变：

```java
// MealPlanServiceImpl.java

// 新增依赖注入
private final MealSchedulePlanMapper mealSchedulePlanMapper;

@Override
protected List<Dish> loadScheduledDishes(LocalDate targetDate, String mealType) {
    int week = ScheduleKeyUtil.calcWeek(targetDate);
    int day = ScheduleKeyUtil.calcDay(targetDate);

    List<Dish> dishes = mealSchedulePlanMapper.findBySchedule(week, day, mealType);

    if (dishes.isEmpty()) {
        log.warn("排餐计划表无数据，week={}, day={}, mealType={}", week, day, mealType);
    }

    return dishes;
}
```

**`generateMealPlan` 接口的入参、出参、调用方式完全不变**，仅在内部将 JSON 查询改为独立表查询。

### 2. 落位菜品到坑位

将菜品库中的某道菜分配到指定坑位。

| 项目 | 说明 |
|------|------|
| **接口地址** | `POST /api/meal-schedule` |
| **权限** | `mealSchedule:add` |
| **Content-Type** | `application/json` |

**请求体 (JSON)**:

| 参数名 | 类型 | 必填 | 说明 | 示例 |
|--------|------|------|------|------|
| weekNum | Integer | 是 | 周序号(1-4) | 1 |
| dayOfWeek | Integer | 是 | 星期几(1-7) | 3 |
| mealTime | String | 是 | 餐次(LUNCH/DINNER/ALL) | LUNCH |
| dishCategory | String | 是 | 坑位分类(MAIN/SIDE等) | MAIN |
| dishId | Integer | 是 | 菜品ID | 88 |

**请求示例**:
```json
{
  "weekNum": 1,
  "dayOfWeek": 3,
  "mealTime": "LUNCH",
  "dishCategory": "MAIN",
  "dishId": 88
}
```

**响应**: `201 Created`
```json
{
  "id": 15,
  "weekNum": 1,
  "dayOfWeek": 3,
  "mealTime": "LUNCH",
  "dishCategory": "MAIN",
  "dishId": 88,
  "enabled": true,
  "createTime": "2026-04-12T10:00:00"
}
```

**错误响应**（坑位已被占用）: `400 Bad Request`
```json
{
  "status": 400,
  "message": "该坑位已存在排餐记录，请先移除再添加"
}
```

## 五、`generateMealPlan` 接口影响分析

### 调用链

```
POST /api/meal-plan/generate
  └── MealPlanServiceImpl.generateMealPlan()
        └── doGenerateMealPlan()
              ├── loadValidOrders()          ← 查订单，无关
              ├── loadCustomers()            ← 查客户档案，无关
              ├── loadSubPackages()          ← 查套餐，无关
              ├── loadScheduledDishes()      ← 👈 唯一改动点
              │       └── dishMapper.findBySchedule(week, day, mealType)
              │               └── 旧: JSON_CONTAINS(schedule, ...)
              │               └── 新: mealSchedulePlanMapper.findBySchedule(week, day, mealType)
              ├── loadDishIngredients()      ← 按 dishIds 查食材，无关
              └── buildCandidateDishPool()   ← 按父套餐归类，无关
```
