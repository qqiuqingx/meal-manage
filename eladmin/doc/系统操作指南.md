# 系统操作指南

## 概述

本系统是一套专业的月子餐/营养餐管理系统，涵盖客户管理、套餐配置、订单处理、排餐计划、核销管理等全流程业务。

**核心业务流程**：
```
套餐配置 → 客户建档 → 创建订单 → 生成排餐 → 执行核销
```

## 一、套餐管理

### 模块概述

套餐管理是系统的基础配置模块，管理父套餐（产品线）和子套餐（规格）的两级结构，并为每个父套餐配置专属的编号池。

**访问路径**：套餐管理 → 套餐管理

### 核心概念

#### 父子套餐结构

```
父套餐（产品线）
  └── 子套餐（规格）
        ├── 荤菜数量
        ├── 素菜数量
        ├── 是否含汤
        └── 是否含米饭
```

**示例**：
- 父套餐：月子餐、营养餐
- 子套餐：7天月子餐、28天月子餐、30天营养餐

#### 编号池配置

每个父套餐可独立配置编号池，用于生成客户唯一编号：

| 字段 | 说明 | 示例 |
|------|------|------|
| 前缀 | 单字母大写前缀 | A、B |
| 编号池前缀 | 可多个字符 | A1、B1 |
| 起始号 | 起始编号 | 1001 |
| 结束号 | 结束编号 | 1199 |

客户编号格式：`编号池前缀 + 序号`，如 A11001、A11002

### 操作步骤

#### 1. 新增父套餐

**入口**：点击"新增父套餐"按钮

**必填字段**：
- 套餐代码
- 套餐名称
- 前缀（单字母）
- 编号池配置（前缀、起始号、结束号）

**操作步骤**：
1. 填写套餐基本信息
2. 配置编号池参数
3. 添加子套餐规格（可一次性添加多个）
4. 点击"确认"保存

#### 2. 编辑父套餐

**入口**：点击列表中套餐的"编辑"按钮

**可修改内容**：
- 套餐名称
- 状态（启用/停用）
- 子套餐列表
- 编号池配置

**注意**：修改编号池配置会影响后续生成的客户编号，但不影响已有客户。

#### 3. 启用/停用套餐

**操作**：点击套餐列表中的状态开关

- 启用状态：可被客户选择
- 停用状态：不可选择新客户，但不影响已有客户

#### 4. 管理子套餐

**入口**：
- 父套餐编辑界面中的子套餐列表
- 或直接在子套餐管理页面

**操作**：
- 新增子套餐
- 编辑子套餐规格（荤菜数、素菜数、是否含汤、是否含米饭）
- 启用/停用子套餐
- 删除子套餐

### 字段说明

#### 父套餐字段

| 字段 | 说明 | 必填 |
|------|------|------|
| 套餐代码 | 业务编码，唯一 | 是 |
| 套餐名称 | 产品线名称 | 是 |
| 前缀 | 单字母，用于客户编号 | 是 |
| 编号池前缀 | 编号池标识 | 是 |
| 起始号 | 编号池起始数字 | 是 |
| 结束号 | 编号池结束数字 | 是 |
| 状态 | 启用/停用 | 是 |
| 备注 | 备注信息 | 否 |

#### 子套餐字段

| 字段 | 说明 | 必填 |
|------|------|------|
| 子套餐代码 | 规格编码，唯一 | 是 |
| 子套餐名称 | 规格名称 | 是 |
| 荤菜数量 | 每顿荤菜数量 | 是 |
| 素菜数量 | 每顿素菜数量 | 是 |
| 是否含汤 | 是否包含汤品 | 是 |
| 是否含米饭 | 是否包含主食 | 是 |
| 状态 | 启用/停用 | 是 |
| 备注 | 备注信息 | 否 |

### 注意事项

1. **编号池管理**
   - 编号池用尽后需要手动扩号或报错
   - 不同套餐的编号池应避免冲突
   - 修改编号池不影响已有客户编号

2. **父子套餐关系**
   - 一个父套餐可以关联多个子套餐
   - 删除父套餐前需先删除或解绑所有子套餐
   - 子套餐不可独立于父套餐存在

3. **状态管理**
   - 停用父套餐会同时停用所有子套餐
   - 停用套餐不影响已有客户的订单和排餐

## 二、菜品管理

### 模块概述

菜品管理负责维护菜品主档、配料字典、以及菜品与配料的关联关系。菜品是面向客户的最终交付物。

**访问路径**：菜品管理 → 菜品管理、菜品配料

### 核心概念

#### 菜品类型

| 类型 | code | 说明 | 典型数量 |
|------|------|------|---------|
| 主菜 | MAIN | 核心荤菜 | 1个/餐 |
| 副菜 | SIDE | 辅助菜品 | 1个/餐 |
| 汤 | SOUP | 汤品 | 1个/餐（可选） |
| 素菜 | VEGETABLE | 蔬菜 | 1-2个/餐 |
| 米饭 | RICE | 主食 | 1个/餐 |

#### 餐次类型

| 类型 | code | 说明 |
|------|------|------|
| 早餐 | BREAKFAST | 独立计算餐数 |
| 午餐 | LUNCH | 计入午晚数池 |
| 晚餐 | DINNER | 计入午晚数池 |

#### 排期格式

格式：`W-D`，W=周序号（1-4），D=星期（1-7）

```
["1-1", "2-1", "3-1", "4-1"]  → 每周一出现
["3-5"]                       → 第3周周五出现
["1-1", "1-2", "1-3", "1-4", "1-5"]  → 第1周工作日出现
```

#### 餐品线代码

| code | 含义 | 说明 |
|------|------|------|
| yuezi | 月子餐 | 产后调理 |
| yunqi | 孕期餐 | 孕期营养 |
| xiaoyuezi | 小月子 | 流产后调理 |
| yingyang | 营养餐 | 通用营养 |
| fenmian | 分娩餐 | 分娩前后 |

一道菜品可以属于多个餐品线。

### 操作步骤

#### 1. 新增菜品

**入口**：点击"新增菜品"按钮

**必填字段**：
- 菜品名称
- 菜品类型
- 适用餐次
- 所属餐品线
- 排期

**操作步骤**：

##### 步骤一：填写基本信息
- 菜品名称：唯一标识
- 菜品类型：选择主菜/副菜/汤/素菜/米饭
- 做法：详细制作流程

##### 步骤二：设置排期和适用范围
- 适用餐次：选择早餐/午餐/晚餐（可多选）
- 所属餐品线：选择该菜品适用的产品线（可多选）
- 排期设置：配置菜品出现的时间规律

##### 步骤三：关联配料
- 搜索配料名称
- 设置用量和单位
- 添加备注（如"少盐"、"切丁"）

##### 步骤四：上传图片（可选）
- 上传菜品图片
- 系统自动保存图片路径

##### 步骤五：保存
点击"确认"完成创建。

#### 2. 编辑菜品

**入口**：点击列表中菜品的"编辑"按钮

**可修改内容**：
- 菜品基本信息
- 排期设置
- 适用范围
- 配料列表
- 启用状态

**操作步骤**：
1. 系统自动加载菜品详情
2. 修改需要更新的字段
3. 配料列表可以增删改
4. 点击"确认"保存

#### 3. 管理配料

**入口**：菜品管理 → 菜品配料

**操作**：
- 新增配料
- 编辑配料信息
- 删除配料

**配料字段**：
- 配料名称（唯一）
- 分类：肉类/蔬菜/海鲜/豆制品/调料/其他
- 单位：克(g)/毫升(ml)/个
- 热量：每单位卡路里

#### 4. 启用/停用菜品

**操作**：点击菜品列表中的启用/停用开关

- 启用状态：可被排餐系统选择
- 停用状态：不再出现在新生成的排餐计划中

### 字段说明

#### 菜品字段

| 字段 | 说明 | 必填 |
|------|------|------|
| 菜品名称 | 菜品名称，唯一 | 是 |
| 菜品类型 | 主菜/副菜/汤/素菜/米饭 | 是 |
| 适用餐次 | 早餐/午餐/晚餐（可多选） | 是 |
| 所属餐品线 | 月子餐/营养餐等（可多选） | 是 |
| 排期 | 出现时间规律 | 是 |
| 做法 | 详细制作流程 | 否 |
| 配料 | 配料列表（关联表） | 是 |
| 图片 | 菜品图片 | 否 |
| 排序值 | 排序，越小越靠前 | 否 |
| 启用状态 | 是否启用 | 是 |

#### 配料字段

| 字段 | 说明 | 必填 |
|------|------|------|
| 配料名称 | 配料名称，唯一 | 是 |
| 分类 | 肉类/蔬菜/海鲜等 | 是 |
| 单位 | 克/毫升/个 | 是 |
| 热量 | 每单位卡路里 | 否 |

### 注意事项

1. **排期设置**
   - 排期决定菜品在哪些日期出现
   - 周一为1，周日为7
   - 月底29-31日会映射回第1周

2. **配料管理**
   - 配料可在菜品管理中独立维护
   - 一道菜可以包含多个配料
   - 配料用量和单位需要准确填写

3. **餐品线与套餐的区别**
   - 餐品线是菜品属性（枚举）
   - 套餐是客户购买的商品规格（数据库表）
   - 两者没有直接关联，但名称相似容易混淆

4. **菜品启用/停用**
   - 停用菜品不影响已生成的排餐记录
   - 只影响新生成的排餐计划

## 三、客户管理

### 模块概述

客户管理模块用于管理客户档案、地址、订单、饮食限制等信息。一个客户档案对应一个活客户，支持签约套餐、记录过敏食物、设置排除菜品等核心功能。

**访问路径**：客户管理 → 客户档案

### 核心概念

#### 客户编号
- 格式：`{编号池前缀}{序号}`，如 A001、B002
- 由系统根据父套餐前缀自动生成
- 创建后不可修改

#### 剩余餐数
- 剩余餐数 = 订单餐数 - 已核销餐数
- 早餐和午晚数独立计算
- 实时计算，不存储在数据库

#### 地址管理
每个客户支持三个地址槽位：
- 默认地址（DEFAULT）
- 工作日地址（WORKDAY）
- 周末地址（WEEKEND）

### 操作步骤

#### 1. 查询客户

**支持筛选条件**：
- 客户编号（精确匹配）
- 客户姓名（模糊匹配）
- 手机号（精确匹配）

**操作**：
1. 在对应的输入框中输入筛选条件
2. 按回车键或点击查询图标
3. 列表自动刷新显示匹配结果

#### 2. 新增客户

**入口**：点击列表页右上角"新增"按钮

**必填字段**：
- 客户姓名
- 手机号
- 默认地址（至少填一个地址）
- 首单信息（父套餐、子套餐、签约日期）

**操作步骤**：

##### 步骤一：填写客户基本信息
- 客户编号：留空自动生成
- 客户姓名：填写客户真实姓名
- 手机号：填写有效手机号
- 孕周：根据实际情况填写（可选）

##### 步骤二：设置饮食限制
- 过敏食物：输入配料名称搜索选择，可多选
- 排除菜品：输入菜品名称搜索选择，可多选
- 排除日期：点击展开日历，选择需要排除的日期和餐次

##### 步骤三：填写地址信息
- 默认地址：必填
- 工作日地址：可选
- 周末地址：可选

##### 步骤四：填写首单信息
- 父套餐：选择客户签约的父套餐
- 子套餐：选择具体的子套餐
- 早餐数：填写早餐份数（可选）
- 午晚数：填写午餐+晚餐份数（可选）
- 签约日期：选择订单开始和结束日期
- 送餐模式：选择 SCHEDULE（排餐）或 MANUAL（手动）

##### 步骤五：提交
点击"确认"按钮完成创建。

**创建后自动完成**：
- 生成客户编号
- 创建首笔订单（状态为"进行中"）
- 创建签约记录

#### 3. 编辑客户

**入口**：点击列表中客户的"编辑"按钮

**可修改内容**：
- 客户基本信息（姓名、手机号、孕周）
- 过敏食物
- 排除菜品
- 排除日期
- 医嘱要求
- 特殊要求
- 三个地址信息

**不可修改内容**：
- 客户编号
- 首单信息（套餐、餐数、日期等）
- 订单状态

#### 4. 查看客户详情

**入口**：点击列表中客户的"详情"按钮

**显示内容**：
- 客户基本信息
- 最新订单信息（包括剩余餐数）
- 所有历史订单列表
- 地址信息
- 饮食限制（过敏食物、排除菜品、排除日期）

#### 5. 删除客户

**删除条件**：
- 客户不存在进行中的订单（status=1）
- 已取消或已完成的订单不影响删除

**删除后级联清除**：
- 客户地址记录
- 签约记录
- 订单记录（保留，不删除）

### 字段说明

#### 列表显示字段

| 字段 | 说明 |
|------|------|
| 客户编号 | 系统自动生成，唯一标识 |
| 姓名 | 客户姓名 |
| 手机号 | 联系电话 |
| 地址 | 默认送餐地址 |
| 早餐数 | 订单中早餐总数 |
| 午晚数 | 订单中午餐+晚餐总数 |
| 剩余早餐 | 可用的早餐份数 |
| 剩余午晚 | 可用的午晚份数 |
| 送餐模式 | SCHEDULE（排餐）或 MANUAL（手动） |
| 过敏食物 | 客户过敏的食材列表 |
| 特殊要求 | 客户备注的特殊要求 |
| 排除菜品 | 不出现在客户餐单中的菜品 |
| 排除日期 | 特定日期不送餐的配置 |
| 创建时间 | 客户档案创建时间 |

### 注意事项

1. **客户编号**
   - 由系统根据父套餐前缀自动生成
   - 格式：`{前缀}{三位数字}`
   - 编辑时不可修改

2. **剩余餐数**
   - 剩余餐数实时计算
   - 计算公式：剩余数 = 订单数 - 已核销数
   - 早餐和午晚独立计算

3. **地址管理**
   - 三个地址槽位：默认、工作日、周末
   - 至少有一个地址非空
   - 编辑时地址整体覆盖

4. **饮食限制**
   - 过敏食物会传递到配菜环节，用于过滤/替换菜品
   - 排除菜品会直接从客户餐单中移除
   - 排除日期可以精确到餐次

5. **删除限制**
   - 仅删除无进行中订单的客户
   - 已取消或已完成的订单不影响删除

6. **编辑与订单的关联**
   - 编辑客户档案不会影响已有订单
   - 修改套餐、餐数需要通过订单管理模块

## 四、订单管理

### 模块概述

订单管理是系统的交易核心，记录客户购买的餐数套餐、金额、核销进度和餐费余额。一个客户可以有多笔订单（增购时新建订单）。

**访问路径**：客户管理 → 订单管理

### 核心概念

#### 订单生命周期

```
已取消(0) ──┐
            │
进行中(1) ──┼──→ 已完成(2)
            │      ↑
            │      └── 核销完所有餐数或到期后手动标记
            │
            └── 客户增购时：新建订单
```

**状态说明**：
- `已取消`：手动取消，不可逆
- `进行中`：正常服务状态，可核销、可排餐
- `已完成`：服务结束

#### 餐数体系

订单中餐数分为两套独立账户：

| 餐次 | 字段 | 说明 |
|------|------|------|
| 早餐 | breakfast_count | 独立计算 |
| 午餐+晚餐 | lunch_dinner_count | 共享一个餐数池 |

#### 金额体系

| 字段 | 说明 |
|------|------|
| 总金额（total_amount） | 原价 |
| 成交价（final_amount） | 实际付款 |
| 定金（deposit_amount） | 已付定金 |
| 已核销金额（verified_amount） | 已核销金额 |
| 餐费余额（meal_balance） | final_amount - verified_amount |

#### 排餐模式

| 模式 | code | 说明 |
|------|------|------|
| 每天送 | DAILY | 每天送餐 |
| 工作日送 | WEEKDAY | 工作日送餐 |
| 周末送 | WEEKEND | 周末送餐 |
| 指定日期送 | SCHEDULE | delivery_dates 指定具体日期列表 |

#### 餐次类型

| 类型 | code | 说明 |
|------|------|------|

#### 订单字段

| 字段 | 说明 | 必填 |
|------|------|------|
| 订单编号 | 系统自动生成，唯一标识 | 自动 |
| 客户 | 关联客户 | 是 |
| 父套餐 | 关联父套餐 | 是 |
| 子套餐 | 关联子套餐 | 是 |
| 早餐数 | 早餐总份数 | 与午晚数至少一个 |
| 午晚数 | 午餐+晚餐总份数 | 与早餐数至少一个 |
| 早餐单价 | 早餐单价 | 是 |
| 午晚单价 | 午餐晚餐单价 | 是 |
| 总金额 | 原价 | 是 |
| 成交价 | 实际付款 | 是 |
| 定金 | 已付定金 | 否 |
| 签约开始日期 | 订单开始日期 | 是 |
| 签约结束日期 | 订单结束日期 | 是 |
| 餐次类型 | ALL/LUNCH/DINNER | 是 |
| 送餐模式 | DAILY/WEEKDAY/WEEKEND/SCHEDULE | 是 |
| 送餐日期列表 | JSON格式日期数组 | 指定日期送时必填 |
| 客户来源 | 销售渠道 | 否 |
| 备注 | 备注信息 | 否 |
| 订单状态 | 0=已取消，1=进行中，2=已完成 | 自动 |

#### 自动计算字段

| 字段 | 计算方式 |
|------|---------|
| 已核销餐数（verified_count） | 核销记录累加 |
| 已核销金额（verified_amount） | 核销数量 × 单价累加 |
| 餐费余额（meal_balance） | final_amount - verified_amount |
| 剩余餐数（remaining_count） | total_count - verified_count |

### 注意事项

1. **餐数计算**
   - 早餐和午晚数分开计算
   - 午餐和晚餐共享午晚数池
   - 核销时按实际餐次扣减

2. **金额管理**
   - 成交价不能大于总金额
   - 餐费余额自动计算，无需手动维护
   - 修改成交价会影响餐费余额

3. **订单状态**
   - 增购不自动关闭老订单
   - 已取消的订单不可恢复
   - 已完成的订单不可重新激活

4. **核销与订单**
   - 核销操作会更新订单的已核销餐数和金额
   - 餐数耗尽时订单不会自动完成
   - 需要手动标记完成或取消

5. **删除订单**
   - 删除订单不影响核销记录
   - 删除操作谨慎使用

## 五、排餐管理

### 模块概述

排餐管理根据每日生效订单，生成客户的菜品配送计划。是连接订单（客户买了什么）和配菜（实际送什么）的执行层。

**访问路径**：排餐管理 → 排餐计划

### 核心概念

#### 三层表结构

```
排餐计划主表（meal_plan）
  └── 排餐客户表（meal_plan_customer）
        └── 客户菜品明细表（meal_plan_customer_item）
```

#### 餐次类型

| 类型 | code | 说明 |
|------|------|------|
| 午餐 | LUNCH | |
| 晚餐 | DINNER | |

**注意**：早餐不参与排餐计划，早餐核销直接由核销管理模块处理。

#### 排餐状态

| 状态 | code | 说明 |
|------|------|------|
| 生成中 | GENERATING | 事务进行中 |
| 成功 | SUCCESS | 所有客户排餐成功 |
| 失败 | FAILED | 部分或全部客户排餐失败 |

#### 客户排餐状态

| 状态 | 值 | 说明 |
|------|------|------|
| 失败 | 0 | 该客户本次排餐失败 |
| 成功 | 1 | 排餐成功 |

### 操作步骤

#### 1. 生成排餐计划

**入口**：点击"生成排餐"按钮

**必填参数**：
- 排餐日期
- 餐次（午餐/晚餐）
- 可选：指定客户ID

**操作步骤**：

##### 步骤一：选择排餐参数
- 选择排餐日期
- 选择餐次（午餐/晚餐）
- 可选：指定客户（若不指定则为所有客户生成）

##### 步骤二：执行生成
- 点击"确认"开始生成
- 系统自动执行排餐逻辑

##### 步骤三：查看结果
- 查看生成成功/失败统计
- 查看失败原因
- 查看排餐明细

**排餐逻辑**：

##### 第一步：有效订单过滤
对每个`status=1`（进行中）的订单，检查：
- 剩余餐数 > 0
- 签约日期包含当天
- 餐次包含当前餐次
- 送餐模式满足当前日期

##### 第二步：候选菜匹配
从菜品库中选出候选菜（同时满足）：
- 启用状态 = 1
- 适用餐次包含当前餐次
- 排期包含当前日期的 scheduleKey
- 餐品线匹配客户订单

##### 第三步：过敏过滤
用客户过敏食物与菜品配料做匹配，过敏菜品从候选中移除。

##### 第四步：按子套餐规格选菜
根据子套餐配置的荤菜数、素菜数、是否含汤、是否含米饭填充菜品槽位。

##### 第五步：回退逻辑
当某类型菜品全部过敏或不可用时，触发同类替代。

#### 2. 查看排餐计划

**入口**：点击排餐列表中的"查看详情"按钮

**显示内容**：
- 排餐计划基本信息
- 排餐客户列表
- 每个客户的菜品明细
- 排餐失败原因（如有）

#### 3. 删除排餐记录

**入口**：
- 按日期+餐次删除
- 按ID批量删除
- 按客户排餐ID删除

**删除规则**：
- 删除排餐记录是软删除（deleted=true）
- 不影响核销数据（meal_verification_log 独立保留）
- 可重新生成同一天同一餐次的排餐

#### 4. 查看客户地址

**入口**：排餐详情 → 查看客户地址

**地址选择规则**：

### 注意事项

1. **排餐时机**
   - 排餐在送餐前生成
   - 可以手动触发或定时任务自动生成
   - 重复生成会软删除旧记录

2. **过敏处理**
   - 过敏菜品会被自动过滤
   - 过滤后触发同类替代
   - 替换信息会记录在明细中

3. **排餐与核销**
   - 排餐是计划，核销是实际
   - 删除排餐不影响核销记录
   - 核销后会标记排餐记录为已核销

4. **地址选择**
   - 根据排餐日期自动选择地址
   - 优先级：周末 > 工作日 > 默认
   - 三个地址都为空时无法排餐

5. **排餐失败**
   - 排餐失败不影响其他客户
   - 失败原因会记录
   - 可以修正后重新生成

## 六、核销管理

### 模块概述

核销管理是送餐交付的核心环节，记录客户每次取餐的事实，作为餐费结算和订单状态推进的触发器。一次核销对应一次送餐成功。

**访问路径**：核销管理 → 核销管理

### 核心概念

#### 核销粒度

核销的操作对象是**客户排餐记录**，而非菜品明细。核销一次，扣一餐。

| 餐次 | 扣减字段 | 单价来源 |
|------|---------|---------|
| 午餐 LUNCH | remaining_count（合计） | lunch_dinner_price |
| 晚餐 DINNER | remaining_count（合计） | lunch_dinner_price |
| 早餐 BREAKFAST | 不参与排餐，直接核销 | breakfast_price |

#### 核销日志

每一笔核销都会生成一条日志记录，包含核销前后的状态快照，用于对账和追溯。

#### 自动完成订单

核销后若剩余餐数减至 0，系统在核销事务内自动将订单状态更新为"已完成"。

### 操作步骤

#### 1. 执行核销

**入口**：核销管理 → 选择待核销记录 → 点击"核销"按钮

**操作步骤**：

##### 步骤一：选择待核销记录
- 筛选条件：日期、餐次、客户
- 勾选需要核销的客户记录
- 支持批量选择

##### 步骤二：执行核销
- 点击"核销"按钮
- 系统执行核销校验链路

##### 步骤三：查看结果
- 查看成功/失败统计
- 查看失败原因
- 查看核销日志

**核销校验链路**（顺序执行，失败立即返回）：
1. 客户排餐记录是否存在？
2. 是否已核销？
3. 关联订单是否存在？
4. 订单剩余餐数是否 > 0？
5. 订单单价是否有效？

**通过全部校验后，原子性执行**：
1. 计算核销单价
2. 更新订单（已核销餐数+1，已核销金额+单价，餐费余额-单价，剩余餐数-1）
3. 若剩余餐数==0，订单状态→已完成
4. 更新排餐客户记录（标记已核销）
5. 写入核销日志

#### 2. 批量核销

**操作**：
1. 勾选多个待核销记录
2. 点击"批量核销"按钮
3. 系统逐个执行核销链路

**结果**：
- 返回成功/失败统计
- 显示失败原因明细

#### 3. 查看核销日志

**入口**：核销管理 → 核销日志

**筛选条件**：
- 客户姓名
- 订单ID
- 餐次
- 日期范围
- 操作时间范围
- 操作人

**显示内容**：
- 核销前后的状态快照
- 操作人和操作时间
- 备注信息

#### 4. 查看订单核销记录

**入口**：订单详情 → 核销记录标签页

**显示内容**：
- 该订单的所有核销记录
- 每次核销的时间和数量
- 累计核销餐数和金额

### 字段说明

#### 核销日志字段

| 字段 | 说明 |
|------|------|
| 客户排餐记录ID | meal_plan_customer.id |
| 排餐计划ID | meal_plan.id |
| 客户ID | 客户ID |
| 订单ID | 关联订单ID |
| 排餐日期 | 排餐日期 |
| 餐次 | LUNCH / DINNER |
| 核销餐数 | 默认 1 |
| 核销前剩余餐数 | remaining_before |
| 核销后剩余餐数 | remaining_after |
| 核销前已核销总数 | verified_total_before |
| 核销后已核销总数 | verified_total_after |
| 操作人 | 操作人 |
| 操作时间 | 操作时间 |
| 备注 | 备注 |

#### 排餐客户核销状态字段

| 字段 | 说明 |
|------|------|
| 是否已核销 | 0=未核销，1=已核销 |
| 核销时间 | 核销时间 |
| 核销操作人 | 核销操作人 |

### 注意事项

1. **禁止重复核销**
   - 已核销的记录不能再核销
   - 系统会严格校验

2. **核销即扣餐数和余额**
   - 每次核销剩余餐数-1
   - 每次核销餐费余额-单价

3. **餐数耗尽自动完单**
   - 剩余餐数==0时自动将订单状态改为已完成
   - 此逻辑在事务内执行，保证一致性

4. **早餐核销**
   - 早餐不参与排餐
   - 早餐核销直接操作订单
   - 不关联排餐记录

5. **核销日志不可删除**
   - 核销日志只增不减
   - 用于对账和追溯
   - 即使删除排餐记录，核销日志也保留

6. **金额计算**
   - 核销单价来自订单配置
   - 午餐和晚餐使用同一单价
   - 早餐使用早餐单价

### 金额计算示例

假设某订单配置：
- 午餐晚餐数：20
- 午餐晚餐单价：45.00 元
- 成交价：5000.00 元

核销 5 次午餐后：
- 已核销餐数 = 5
- 已核销金额 = 5 × 45 = 225.00 元
- 餐费余额 = 5000 - 225 = 4775.00 元
- 剩余餐数 = 20 - 5 = 15

第 20 次核销后：剩余餐数 == 0，订单自动变为已完成。

## 附录：常见问题

### 客户管理

**Q: 为什么有些客户无法删除？**
A: 客户存在进行中的订单（status=1），需要先取消或完成订单才能删除。

**Q: 客户编号可以修改吗？**
A: 不可以。客户编号由系统根据父套餐前缀自动生成，创建后不可修改。

**Q: 编辑客户会影响已有的排餐计划吗？**
A: 修改基本信息、地址不会影响。修改过敏食物、排除菜品、排除日期会影响后续的排餐生成，但不影响已生成的排餐记录。

**Q: 如何给客户增购套餐？**
A: 需要在订单管理模块创建新订单，不能通过编辑客户档案实现。

### 订单管理

**Q: 增购时老订单会自动关闭吗？**
A: 不会。增购时新建订单，老订单状态不变，需要手动将老订单标记为完成。

**Q: 餐费余额如何计算？**
A: 餐费余额 = 成交价 - 已核销金额，系统自动计算。

**Q: 已核销的订单可以取消吗？**
A: 技术上可以，但不建议。需要特殊处理退款和核销记录。

**Q: 如何查看订单的核销记录？**
A: 在订单详情中查看核销记录标签页，或通过核销管理模块按订单ID查询。

### 排餐管理

**Q: 排餐失败怎么办？**
A: 查看失败原因，修正配置（如菜品不足、地址缺失等）后重新生成。

**Q: 删除排餐记录会影响核销吗？**
A: 不会。删除排餐记录是软删除，核销日志独立保留，不受影响。

**Q: 排餐时为什么没有选中某些客户？**
A: 检查客户订单状态（必须为进行中）、剩余餐数（必须>0）、签约日期范围、送餐模式配置。

**Q: 排餐可以重复生成吗？**
A: 可以。重复生成会软删除旧记录并重新生成，不会影响核销数据。

### 核销管理

**Q: 为什么不能核销？**
A: 检查：① 排餐记录是否存在 ② 是否已核销 ③ 订单剩余餐数是否>0 ④ 订单单价是否配置

**Q: 核销后订单会自动完成吗？**
A: 会。当剩余餐数减至 0 时，系统自动将订单状态改为已完成。

**Q: 可以取消核销吗？**
A: 不可以。核销是不可逆操作，用于保证数据的一致性和可追溯性。

**Q: 早餐如何核销？**
A: 早餐不参与排餐，可以直接在核销管理模块按客户和日期核销早餐。

### 套餐与菜品管理

**Q: 套餐和餐品线有什么区别？**
A: 套餐是客户购买的商品规格（数据库表），餐品线是菜品的属性（枚举），两者是两套独立的体系。

**Q: 停用菜品会影响已生成的排餐吗？**
A: 不会。停用菜品只影响新生成的排餐计划。

**Q: 配料可以重复使用吗？**
A: 可以。配料是字典，可以被多道菜品重复使用。

**Q: 排期格式如何理解？**
A: 格式为 W-D，W=周序号（1-4），D=星期（1-7）。例如"1-1"表示第1周周一。

## 相关模块关系图

```
套餐管理（parent_package / sub_package）
  │
  ├── 客户订单（customer_order）
  │     ├── 关联 parent_package_id → 确定产品线
  │     └── 关联 sub_package_id → 确定规格
  │
  ├── 客户档案（customer_profile）
  │     └── 通过订单关联 → 生成客户编号
  │
  └── 配菜管理（dish）
        └── 客户餐品线代码 → 筛选菜品

客户管理（customer_profile）
  │
  ├── 订单管理（customer_order）
  │     └── 一个客户可有多笔订单
  │
  ├── 配菜管理（dish）
  │     └── allergy_tags → 过滤/替换菜品
  │
  └── 核销管理（meal_verification_log）
        └── 影响剩余餐数计算

订单管理（customer_order）
  │
  ├── 排餐管理（meal_plan）
  │     └── status=1 时参与排餐
  │
  └── 核销管理（meal_verification_log）
        └── 核销记录累加更新订单数据

排餐管理（meal_plan）
  │
  ├── 订单管理（customer_order）
  │     └── 查 status=1 + remaining_count>0
  │
  ├── 套餐管理（sub_package）
  │     └── 查规格 → 决定菜品槽位
  │
  ├── 配菜管理（dish）
  │     └── 查菜品 + 过滤过敏
  │
  ├── 客户管理（customer_profile）
  │     └── 查 allergy_tags
  │
  └── 核销管理（meal_verification_log）
        └── 核销操作标记 is_verified

核销管理（meal_verification_log）
  │
  ├── 订单管理（customer_order）
  │     ├── 扣减 verified_count / verified_amount
  │     ├── 扣减 meal_balance / remaining_count
  │     └── 剩余为0时自动 status → 2
  │
  └── 排餐管理（meal_plan_customer）
        └── 标记 is_verified / verification_time
```
