# 智能客服助手运维配置说明

## 必填配置

主系统与 `agent-service` 必须配置一致的内部调用凭证：

- `agent.internal-token`：服务间内部 Token。不得写入前端、浏览器缓存或普通日志。
- `agent.access-context-secret`：主系统签发客服访问上下文的 HMAC 密钥，生产环境至少 32 字符，通过 `AGENT_ACCESS_CONTEXT_SECRET` 注入。
- `agent.access-context-ttl-seconds`：访问上下文有效期，默认 120 秒；不得设置为长期 Token。
- `agent.context-base-url`：`agent-service` 回调主系统的固定地址；不可由模型或用户输入决定。

## 调用约束

内部业务查询请求必须同时携带：

- `X-Agent-Internal-Token`
- `X-Agent-Access-Context`
- `X-Agent-Session-Id`
- `X-Request-Id`

主系统校验 Token、上下文签名、过期时间、请求 ID、会话归属和客服业务权限后才执行查询。

## 超时与容量建议

- 单工具查询目标 P95 不超过 3 秒；多工具组合目标 P95 不超过 8 秒。
- 单轮 QueryPlan 最多 6 个工具调用；列表、日期范围、菜品与配料数由服务端上限控制。
- 下游超时或单工具失败时必须返回稳定错误/部分结果，不得把异常堆栈透传到前端或模型上下文。

## 多实例与灰度

- `agent-service` 不保存业务会话真相。主系统在每轮请求中下发已持久化的客户、订单、排餐、日期、餐次和查询范围槽位，因此负载均衡可以将同一 `sessionId` 的相邻请求分发到不同实例。
- 灰度至少使用两个 Agent 实例验证：实例一识别客户或订单后，实例二仅依赖主系统下发的 `contextSlots` 继续查询；验证结果不得出现客户、订单、日期或餐次丢失、串会话或越权。
- 灰度只向拥有 `agentDiagnosis:list` 及所需业务权限的内部测试客服开放。先观察审计中的工具失败率、部分回答率、关键澄清率、未支持问题 Top N 与 P95，再扩大范围。
- 每次发布须保留单工具和多工具真实请求的 P95 采样。开发环境基线为 2026-07-13 的 20 次受控请求：单工具运营统计 P95 为 956ms，多工具餐数变化查询 P95 为 1630ms；该数据仅用于回归比较，生产是否达标以目标环境采样为准。
- 发布操作和签字项见《智能客服助手灰度验收清单》。

## 日志与审计

- 日志仅记录 requestId、sessionId、工具名摘要、结果数量、耗时和稳定失败码。
- 不记录手机号、完整地址、特殊要求全文、工具原始响应、模型提示词或任何金额字段。
- `agent_business_query_audit` 用于按客服、客户、请求和工具追踪只读查询；保留周期由生产数据治理策略确认。
