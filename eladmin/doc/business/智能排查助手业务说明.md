# 智能排查助手业务说明

## 1. 产品定位

智能客服助手用于帮助内部客服查询客户、订单、排餐、核销、退餐和套餐等业务信息，并保留“客户为什么没有生成排餐计划”的诊断能力。

助手不是自动修复工具，而是一个面向已登录内部客服的 AI 诊断和只读查询助手。

### 只读查询安全基线（2026-07-11 起）

- Agent 查询计划采用版本化的受控 QueryPlan；其领域、动作、实体、日期范围、分页和工具数量均由服务端白名单校验，不能携带 SQL、表名、任意字段或排序表达式。
- 订单金额、单价、已收金额、优惠、退款金额及餐费余额均不属于 Agent 数据契约，不能进入工具结果、模型上下文、日志或前端卡片。
- 主系统仅向 Agent 下发当前客服可调用的固定工具名，用于预过滤无权限编排；主系统内部查询接口仍以签名访问上下文再次执行入口权限和业务权限校验，工具名白名单不承载授权。系统管理员登录态仅携带 `admin` 权限，视为具备全部 Agent 只读查询工具及内部查询权限。
- Agent 访问上下文携带“全部数据范围”标记或部门 ID 集合。非全量范围按“允许部门 -> 部门内系统用户 -> `customer_profile.create_by` -> 客户 ID”解析；客户、订单、排餐、核销、退餐和运营统计均在读取或聚合前按该客户集合收缩。套餐、公共菜单和菜品主档不包含客户数据，不参与该客户范围过滤。当前映射依赖客户档案创建人准确维护；如后续引入门店归属或客户负责人字段，应替换为显式业务归属字段并同步本说明。
- 排餐查询的客户、单日、开放日期上下界、餐次和客户排餐记录 ID 均为可选条件；空条件表示当前客服数据范围内的全部历史。查询通过 SQL 倒序分页控制返回量，默认第 1 页、单页最多 50 条，不会将无界历史一次性加载到内存。按客户排餐记录 ID 查询时仍校验客户归属。
- “他是否曾经排过餐”等多轮问法继承上一轮客户，直接查询历史并返回是否存在、总记录数和最近一条排餐；无日期表示全部历史，不默认解释为今天。高置信模型已识别该上下文追问时，零置信度且没有实质歧义证据的通用规则澄清不得覆盖模型结果。
- 智能客服查询和排餐诊断均只返回证据、规则说明与人工核对建议；本期不生成动作草稿、不展示确认入口，也不调用任何动作确认接口。
- 客服询问“签了什么套餐/套餐详情”时，Agent 先读取该客户受控概览中的已签约父套餐 ID，再限量查询父子套餐规格（最多 5 个）；不接受自然语言中任意套餐 ID、字段或排序条件，结果通过结构化卡片展示。
- 核销和退餐范围问法支持“本周”“本月”“最近 N 天”，其中 N 被限制在 1 至 31；范围会作为 `QueryPlan.filters.startDate/endDate` 传递至主系统，历史记录仍按最多 50 条返回。
- 客服可按客户、日期和午晚餐查询候选菜预览。预览只读取当日排期菜、固定米饭、进行中订单的父套餐、客户排除菜与三级过敏标签，最多显示 20 条并标识过滤原因；它不创建排餐，也不能作为配送或排餐生成成功的结论。
- 早餐与午晚餐余额必须由主系统业务服务计算；午餐和晚餐共享午晚餐餐数池，已删除的核销日志不参与统计。
- 单轮业务查询最多调用 6 个工具，并按每个工具登记的最大结果数预留累计 100 条数据预算；超出预算不会继续调用内部接口，响应会标记为部分结果并提示人工缩小查询范围。

## 2. 第一阶段边界

第一阶段只做:

- 输入客户、日期、餐次
- 查询诊断所需业务上下文
- 加载 `agent-service/rules/` 中的 rule registry
- 由 AI 基于规则和数据输出原因、证据和建议
- 前端展示“AI 诊断结果”

第一阶段不做:

- 不自动修改订单
- 不自动生成排餐
- 不直接新增客户
- 不让 AI 直接访问数据库
- 不对外开放给客户自助聊天

## 3. 诊断原则

AI 诊断必须遵循:

- 只能基于当前提供的业务上下文和 rule registry 分析
- 每个原因都应尽量带证据
- 每个原因应能追溯到字段、规则或规则版本
- 诊断结果是建议，不是最终结论
- 客服需要结合证据人工确认

## 4. 规则维护方式

`agent-service/rules/` 是 AI 诊断规则真相源，不再依赖普通文档作为规则依据。

规则文件需要维护:

- `ruleId`
- `reasonCode`
- `version`
- `triggerConditions`
- `requiredTools`
- `requiredData`
- `decisionHints`
- `evidenceFields`
- `nextActions`
- `severity`
- `owner`

当前规则加载会自动校验:

- `ruleId` 不允许重复
- `reasonCode` 不允许为空
- `requiredTools` 必须命中已注册工具
- `evidenceFields` 不允许为空
- `nextActions` 不允许为空
- `owner` 不允许为空

如果诊断相关代码发生变化但规则文件没有变化，部署脚本会要求提交信息包含 `rules-no-change`，否则拒绝部署。

## 5. 兜底结果

当 AI 不可用或输出不可信时，系统返回兜底结果。

兜底结果特点:

- `fallback=true`
- 不给出确定业务原因
- 提示客服按客户、订单、排餐记录、候选菜配置进行人工核对
- 不影响主业务系统使用

## 6. 后续扩展

后续可以扩展:

- 多轮补充问题
- 订单异常诊断
- 核销异常诊断
- 客服知识问答
- AI 辅助新增客户

其中“AI 辅助新增客户”必须采用:

1. AI 生成草稿
2. 服务端校验草稿
3. 前端展示差异和风险
4. 人工确认
5. 主系统正式提交

AI 不允许直接写数据库。

## 6.1 会话语义与集合追问

客服业务查询使用受控的组合语义帧表达目标对象、范围、度量、维度、操作和输出形态。模型只能提出需要的上下文类型，服务端再从已登记的 `ENTITY`、`ENTITY_SET`、`METRIC_RESULT`、`FILTER_SET`、`TASK` 句柄中按类型兼容、时效和会话显著度解析具体引用。

例如“活跃客户数”完成后会登记可重新计算的 `AGENT_ACTIVE_CUSTOMER_V1` 客户集合，不保存客户 ID 列表或工具原始结果。后续追问余额明细时仅能通过该句柄调用固定的 50 条受控明细能力；没有匹配句柄或存在多个近似候选时必须澄清，不能猜测对象。

“他们分别还剩多少餐”“这些客户每个人还有几餐”等追问，应解析为上一轮客户集合的餐数余额展开，直接返回各客户早餐、午晚餐和合计剩余餐数；已有唯一且有效的客户集合句柄时不得退回单客户编号追问。

会话还可持久化最多五个受控任务摘要，用于跨实例恢复待补槽位和相关插问；任务栈不保存用户原文、模型推理或工具原始响应。

## 7. 二期会话增强

二期开始，会话不再只是“一问一答”的临时缓存，而是显式维护会话阶段和最近诊断摘要。

当前已落地的会话规则:

- 会话默认阶段为 `COLLECTING_SLOTS`
- 当客户、日期、餐次齐全后进入诊断流程，诊断完成后阶段更新为 `DIAGNOSED`
- 用户发送“重新排查”时，保留当前槽位并重新发起诊断
- 用户发送“清空会话”时，清空槽位和最近诊断结果
- 追问场景优先复用上一轮诊断结果，返回证据摘要和下一步人工核对建议

当前会话上下文保留策略:

- 最近 10 条会话轮次
- 最近 3 次诊断摘要

该策略用于支持客服围绕同一客户和同一餐次持续追问，但仍保持会话内存窗口可控。

## 8. 二期槽位提取策略

二期的槽位提取不再只返回“有没有提到客户/日期/餐次”，还会给出每个槽位的来源和置信度。

当前规则:

- 明确命中客户编号、客户 ID、绝对日期、相对日期、餐次别名时，置信度为 `HIGH`
- 仅从上一轮会话继承的槽位，置信度为 `MEDIUM`
- 识别到“客户 123”这类歧义表达，或一条消息里出现多个未明确修正关系的候选值时，置信度为 `LOW`

当前来源标识:

- `EXPLICIT_INPUT`：本轮明确输入
- `CONTEXT_INHERIT`：沿用上一轮上下文
- `CORRECTION_OVERRIDE`：本轮用“换成”“不是…是…”等表达覆盖旧值
- `AMBIGUOUS_INPUT`：本轮识别到候选值，但仍需客服确认

追问优先级:

1. 缺失客户
2. 缺失日期
3. 缺失餐次
4. 低置信度槽位确认

## 9. 二期提示词策略化

二期开始，排餐诊断提示词不再完全硬编码在 Java 类里，而是拆为代码 + YAML policy 共同维护。

当前策略文件职责:

- `prompt-policy.yaml` 维护场景、角色、输出必填字段
- `forbiddenClaims` 维护禁止模型声称的写操作结果
- `toolPolicy` 维护工具调用预算和结论前必查工具
- `evidencePolicy` 维护证据最小要求

当前运行规则:

- PromptBuilder 每次构建工具模式提示词时都会加载并校验 policy
- policy 校验失败视为服务配置错误，不能静默退回旧提示词
- 提示词会显式带出 policy version，便于后续排查模型输出口径差异

## 10. 二期工具调用稳定性

二期开始，`agent-service` 对工具调用不再只做“注册给模型”，还会在服务端做预算、缓存和 trace 收集。

当前策略:

- 单次诊断默认最多 8 次实际工具调用
- 同一工具 + 相同入参摘要的重复调用直接复用上次结果
- 关键工具 `getCustomerProfile`、`listCustomerOrders`、`getMealPlan` 失败时，诊断直接降级为 fallback
- 超出预算时，诊断直接降级为 fallback

当前可观测项:

- 每次工具调用只记录 `inputDigest`、结果条数、耗时、错误类型，不落原始敏感入参/出参
- 每轮模型调用会记录 round、是否触发工具、工具名称摘要、耗时
- 最终诊断结果会附带 `diagnosisTrace` 和 `toolCallSummary`，供后续前端展示和问题排查使用

## 11. 二期结构化输出与建议模板

二期开始，AI 诊断结果的可展示字段不再只要求“有 summary 和 reasons”，而是要求一组更稳定的结构化字段。

当前重点字段:

- 结果级：`summary`、`confidence`、`fallbackReason`、`nextActions`、`diagnosisTrace`、`toolCallSummary`
- 原因级：`code`、`title`、`level`、`confidence`、`ruleIds`、`suggestion`、`nextActions`、`evidence`

当前校验规则:

- 不允许出现“已修复”“已修改数据库”“已创建客户”等写操作结论
- reason 必须带 `ruleIds`、`nextActions`、`evidence`
- 非 fallback 结果不允许引用不存在的 `ruleId`
- 非 fallback 结果的 `reason.code` 必须与规则 `reasonCode` 一致
- 非 fallback 结果的 `evidence.label` 必须命中关联规则的 `evidenceFields`
- 非 fallback 结果至少包含一个规则定义的真实业务证据字段
- fallback 结果必须带 `fallbackReason`

建议模板策略:

- AI 优先返回 `reason.code` 和证据
- 系统按 `reason.code` 读取建议模板，补齐 `suggestion` 和 `nextActions`
- 首批常见原因都在 `suggestion-template.yaml` 中固化，减少客服动作文案漂移

## 13. 一期评测集与规则治理

一期可信度建设新增 `agent-service/src/test/resources/evaluation/meal-plan-diagnosis-cases.yaml` 作为排餐未生成诊断评测集。

当前评测集包含 101 条结构化案例，字段包括用户输入、结构化槽位、期望原因码、期望规则 ID、期望证据字段、期望建议动作、是否允许 fallback 和人工备注。

当前覆盖分类:

- 客户停送或排除日期导致未排餐
- 客户不存在或编号错误
- 订单缺失、未生效或已结束
- 剩余餐数不足
- 排餐模式不匹配
- 候选菜不足或过敏忌口过滤为空
- 排餐生成失败
- 数据缺失或规则缺口，需要人工核对

评测执行当前包含两层:

- `MealPlanDiagnosisEvaluationCasesTest`：校验 YAML 案例数量、关键字段和覆盖分类
- `MealPlanDiagnosisEvaluationTest`：解析 YAML 案例，通过真实 `MealPlanDiagnosisOrchestrator` 加载 rule registry，并委托 mock `DiagnosisAiClient` 构造带 mock 工具证据的诊断输出；输出经过真实 `DiagnosisResultValidator` 后断言原因码、规则 ID、证据字段、建议动作、fallback 口径和编排器上下文回填

当前评测不依赖真实数据库和真实模型，但已经覆盖规则加载、AI client 委托、上下文字段回填和结果校验。后续如果要接入更接近真实模型的回归，应保持 YAML 案例源不变，只替换评测执行器里的 mock `DiagnosisAiClient`。

## 12. 二期前端工作台

二期前端不再只展示消息气泡和一张结果卡片，而是调整为客服排查工作台。

当前页面分区:

- 左侧：会话消息、快捷回复、输入区
- 右侧：当前槽位、会话阶段、最近一次诊断摘要
- 结果区：原因列表、建议动作、工具调用摘要、诊断链路、兜底原因

当前交互:

- 继续支持快捷补充客户、日期、餐次
- 支持“重新排查”“清空会话”
- 支持“换成午餐 / 晚餐”类局部覆盖输入
- 工具调用摘要和诊断链路默认折叠，需要时再展开

## 14. 三期工具扩展

三期开始，智能排查工具从一阶段的基础上下文查询扩展为更细的只读诊断工具，便于 AI 在下结论前按规则补齐证据。

当前新增工具:

- `getCustomerExcludeDates`：查询客户停送、排除日期、过敏忌口和特殊要求，返回诊断所需的非敏感字段。
- `getOrderMealBalance`：查询订单有效期、餐次类型、早餐餐数池和午晚餐餐数池的核销数、剩余数。
- `getPackageSpec`：查询父套餐、子套餐关系和餐品规格，返回套餐规格缺失项摘要。
- `getDishCandidateDetail`：查询指定排餐日期下套餐维度的候选菜诊断摘要，当前基于 `mealPlanService.statByDate` 输出候选数量。
- `listVerificationLogs`：查询订单或客户订单集合下的核销日志，可按日期范围、餐次过滤。
- `listMealRefunds`：查询订单或客户订单集合下的退餐日志。
- `getMealPlanGenerationSnapshot`：查询排餐生成快照、成功失败数量和失败客户摘要。

当前二期新增/细化原因码:

- `ORDER_EXPIRED`：订单已过有效期。
- `ORDER_MEAL_TYPE_MISMATCH`：订单餐次类型不覆盖目标餐次。
- `PACKAGE_SPEC_MISSING`：父套餐或子套餐规格缺失。
- `REFUND_OR_STOP_MEAL_HIT`：命中退餐、停餐或退款记录。
- `VERIFICATION_CONSUMED_COUNT`：核销记录已消耗对应餐数池。

## 15. 四期会话持久化与多会话

四期开始，智能排查会话不再只依赖 `agent-service` 的内存态，而是由主系统落库维护完整会话轨迹。

当前四期会话规则:

- 主系统新增 `agent_chat_session` 和 `agent_chat_message` 两张表，按 `sessionId` 串联会话摘要、消息、动作审计和客服反馈。
- 后台聊天接口 `/api/agent/meal-plan/chat` 保持不变，但请求体新增 `clientMessageId`，用于防止客服重复点击发送造成重复诊断。
- 当 `sessionId` 为空时，主系统先创建会话，再调用 `agent-service`。
- 会话列表按当前客服账号隔离查询，支持按客户编号、标题、最近摘要和时间筛选。
- 会话详情返回完整消息、最近一次诊断结果、当前槽位、最近动作确认和反馈记录，支持刷新页面后恢复排查上下文。
- 已归档会话保留历史追溯信息，但当前实现不允许继续写入新的聊天消息，客服需要新建会话继续排查。

当前标题与摘要策略:

- 显式创建会话时可指定标题。
- 未指定标题时，优先使用 `客户编号 + 日期 + 餐次` 作为标题；仍无法确定时回退到首条用户消息截断结果。
- 会话摘要优先保存最近一次诊断 `summary`，若本轮尚未触发诊断则回退保存助手最新回复。

## 16. 四期服务韧性与健康检查

四期第二批开始，主系统调用 `agent-service` 不再只做单次 HTTP 请求，而是补充了失败分类、短重试和明确兜底来源。

当前策略:

- 仅 `AGENT_SERVICE_TIMEOUT`、`AGENT_SERVICE_UNAVAILABLE`、`AGENT_SERVICE_5XX` 允许短重试。
- `AGENT_SERVICE_BAD_RESPONSE` 和 `AGENT_SERVICE_4XX` 不重试，直接返回明确兜底提示。
- 主系统生成的兜底结果统一带 `fallbackSource=ELADMIN_CLIENT`，并记录 `failureType`，便于运营统计区分是超时、服务不可达、结构异常还是服务端 5xx。
- 聊天接口 fallback 时，除了助手提示外，还会补一份可落库的 `diagnosisResult`，避免会话消息和运营指标丢失失败分类。

当前健康检查:

- `agent-service` 新增 `GET /api/agent/health`
- 该接口只检查规则是否成功加载、规则版本摘要是否可读、模型客户端是否已装配、工具数据客户端配置是否齐全
- 健康检查不主动调用模型，也不主动调用主系统工具接口，避免引入成本和额外延迟

当前运营看板扩展:

- 工作台右侧运营看板除原因码分布外，还展示 `failureTypeDistribution` 和 `fallbackSourceDistribution`
- 当前主系统调用 `agent-service` 失败的兜底来源统一为 `ELADMIN_CLIENT`
- 失败类型当前重点区分超时、不可用、返回异常、4xx 和 5xx，便于客服和管理员区分是服务问题还是输入问题

## 17. 四期动作草稿过期校验

四期第三批开始，动作草稿确认前会重新读取关键业务数据，避免客服基于过期诊断结果执行写操作。

当前已接入的过期校验:

- `RESUME_CUSTOMER_DELIVERY`
  - 重新读取客户档案 `excluded_dates`
  - 如果目标日期餐次已经不在排除配置中，返回 `STALE_DRAFT`
- `ADJUST_ORDER_EFFECTIVE_DATE`
  - 重新读取订单开始日期、结束日期和状态
  - 如果订单有效期已变化，或订单已不是进行中状态，返回 `STALE_DRAFT`
- `RECALCULATE_ORDER_BALANCE`
  - 对比草稿携带的当前核销餐数、剩余餐数、余额、状态快照
  - 若这些关键字段已变化，返回 `STALE_DRAFT`
- `REGENERATE_MEAL_PLAN`
  - 重新检查目标日期餐次是否已有有效排餐
  - 若排餐已经生成，返回 `STALE_DRAFT`

当前响应口径:

- `status=STALE_DRAFT`
- `message=业务数据已变化，请重新排查后再确认动作。`
- `failureReason` 返回更具体的变化说明，例如“诊断草稿生成后订单有效期已变化”

审计要求:

- 所有 `STALE_DRAFT` 都会写入 `agent_action_audit`
- 审计会额外记录 `draft_digest`、`stale_check_result`、`stale_check_detail`
- `MEAL_PLAN_ALREADY_EXISTS_BUT_CUSTOMER_MISSING`：排餐任务存在但客户未进入明细。

工具调用约束:

- 所有新增工具均为只读工具，不允许写库、补排餐、改订单或改客户档案。
- `agent-service` 通过内部 token 调用 `eladmin-system` 的 `/api/internal/agent/*` 接口。
- 内部接口日志只记录 requestId、查询主键、日期、结果条数和耗时，不记录 token、手机号和完整原始响应。
- 工具返回字段以诊断证据为目标，避免把客服排查不需要的敏感字段继续传递给模型。

业务查询评测集已扩展到 205 条真实不同问法，覆盖单客户、跨模块组合、运营统计、模糊追问、安全拒绝和未支持能力；运营统计与歧义样例会真实执行规则提取、受控分析和 QueryPlan 规划。前端诊断结果区提供客户档案、客户订单和排餐详情跳转入口，跳转后目标页面按客户、日期或餐次进行筛选。

## 20. 五期客户信息查询能力

### 20.1 背景

五期新增一条独立的"客户信息查询"能力，让 Agent 能直接回答客户维度的确定性业务数据问题，不再强制要求排餐日期和餐次。

目标示例：

```
B3303 这个客户还剩多少餐数
B3303 核销了多少餐
B3303 有哪些进行中订单
```

### 20.2 设计原则

1. 客户信息查询不硬塞进现有排餐诊断链路。
2. 排餐诊断仍要求客户、日期、餐次。
3. 客户信息查询只强制要求客户编号或客户 ID，日期和餐次作为可选过滤条件。
4. 餐数、核销、订单状态属于确定性统计，由服务端聚合计算，不依赖模型自由推理。
5. Agent 仍保持只读边界，不直接修改客户、订单、核销或排餐数据。
6. 所有返回口径必须能追溯到订单表和核销日志表。

### 20.3 意图与槽位差异

混合意图提取器对所有只读业务问题统一输出顶层 `BUSINESS_QUERY`。客户、订单、排餐、核销、退餐、套餐、菜品、规则和运营统计的细粒度旧意图仅保存为受控兼容路由线索，不能由用户输入直接指定；后续由 `BusinessQuestionAnalysis` 和 `QueryPlan` 确定领域、指标、维度和工具。`DIAGNOSE`、`FOLLOW_UP`、`RETRY`、`RESET`、`OUT_OF_SCOPE` 仍是独立顶层意图。

| 意图 | 必填槽位 | 可选槽位 |
|------|----------|----------|
| `DIAGNOSE`（排餐诊断） | 客户、排餐日期、餐次 | - |
| `BUSINESS_QUERY`（只读业务查询） | 由领域和 QueryPlan 决定；客户类查询要求客户 | 日期、餐次、订单、指标、维度 |
| 旧 `CUSTOMER_*_QUERY` 等细粒度意图 | 仅兼容路由使用，不对模型或前端开放 | - |

### 20.4 识别优先级

1. reset / retry 优先。
2. 客户餐数、核销、订单查询其次（关键词匹配）。
3. 排餐诊断表达最后。

具体关键词：

- **餐数余额类**：还剩多少、剩多少、剩余多少、剩余餐数、餐数余额、还能吃几餐
- **核销类**：核销了多少、已核销、核销记录、最近核销、用了多少餐
- **订单类**：有哪些订单、订单情况、进行中订单、订单状态

### 20.5 后端数据口径

#### 剩余餐数口径

客户维度汇总所有有效订单（status = 1 且 remaining_count > 0）：

```
剩余早餐数 = 早餐总数 - BREAKFAST 已核销数
剩余午晚餐数 = 午晚餐总数 - LUNCH 已核销数 - DINNER 已核销数
合计剩余餐数 = 剩余早餐数 + 剩余午晚餐数
```

其中已核销数来自 `meal_verification_log.meal_type`，已删除核销日志不计入有效核销。

#### 核销统计口径

累计核销按未删除核销日志统计：

```
已核销早餐 = SUM(meal_type = BREAKFAST 的 verification_count)
已核销午餐 = SUM(meal_type = LUNCH 的 verification_count)
已核销晚餐 = SUM(meal_type = DINNER 的 verification_count)
累计已核销 = 已核销早餐 + 已核销午餐 + 已核销晚餐
```

#### Agent 业务规则白名单

客服规则问答只读取主系统 `config/agent-business-rules.json` 中的版本化条目：`MEAL_BALANCE`、`ORDER_EFFECTIVE`、`MEAL_PLAN_MATCH`、`DIETARY_FILTER`、`VERIFICATION_REFUND_EFFECT`。目录在启动时校验稳定 ID、主题唯一性、版本、责任模块、依据业务文档和展示内容；校验失败会阻止目录注册。规则分别说明餐数池、订单有效性、排餐模式与餐次匹配、排除日期与过敏忌口过滤、核销及退餐影响；不读取任意文件路径，也不由模型根据常识补充规则。

### 20.6 响应类型

聊天响应新增 `responseType` 和 `insightResult` 字段：

| responseType | 说明 |
|---|---|
| `CUSTOMER_MEAL_SUMMARY` | 客户餐数余额 |
| `CUSTOMER_VERIFICATION_SUMMARY` | 客户核销统计 |
| `CUSTOMER_ORDER_SUMMARY` | 客户订单列表 |
| `SLOT_REQUIRED` | 槽位缺失提示 |
| `MEAL_PLAN_DIAGNOSIS` | 排餐诊断结果（原行为） |

### 20.7 回复模板

**剩余餐数：**
```
B3303（张三）当前有效订单共 2 笔。剩余早餐 3 餐，剩余午晚餐 18 餐，合计剩余 21 餐。已核销早餐 7 餐，午餐 12 餐，晚餐 10 餐。数据按未删除核销日志实时汇总。
```

**核销统计：**
```
B3303 累计已核销 29 餐，其中早餐 7 餐、午餐 12 餐、晚餐 10 餐。最近 10 条核销记录已列在下方。
```

**客户不存在：**
```
未找到客户编号 B3303，请确认编号是否正确。
```

**无有效订单：**
```
B3303 当前没有有效进行中订单，因此没有可继续核销的剩余餐数。历史订单和核销记录已列在下方供核对。
```

### 20.8 前端展示

根据 `responseType` 渲染不同卡片：

1. `CUSTOMER_MEAL_SUMMARY`：展示餐数余额卡（汇总统计 + 订单明细表）
2. `CUSTOMER_VERIFICATION_SUMMARY`：展示核销统计卡（累计核销 + 最近核销记录）
3. `CUSTOMER_ORDER_SUMMARY`：展示订单列表卡
4. `MEAL_PLAN_DIAGNOSIS`：保持原诊断结果展示

### 20.9 安全边界

- 所有查询接口继续使用 `X-Agent-Internal-Token` 鉴权
- 只允许内部 Agent 调用，不对普通前端开放
- 日志只记录客户编号、客户 ID、结果条数、耗时，不记录手机号等敏感信息
- 客户手机号、地址等敏感字段不要默认传给模型，也不要写入 Agent 链路日志

## 18. 三期运营看板统计口径

运营看板通过 `AgentOperationStatsService` 汇总诊断指标、动作确认审计和客服反馈。

当前统计口径:

- `diagnosisCount`：当前查询条件命中的诊断指标记录数
- `fallbackRate`：`fallback=true` 的诊断数 / 诊断总数
- `toolFailureRate`：工具失败次数 / 工具调用次数
- `averageDiagnosisCostMs`：当前查询条件命中的诊断平均耗时
- `reasonCodeDistribution`：按诊断指标中的预测原因码统计分布
- `feedbackAcceptedRate`：客服反馈中 `ACCEPTED` 数 / 反馈总数
- `actualReasonDistribution`：按客服反馈真实原因码统计分布
- `highFrequencyUnknownReasons`：当前重点统计 `AI_RESULT_INVALID`、`DATA_INCOMPLETE_NEED_RECHECK`
- `actionDraftConfirmationRate`：当前查询条件命中的诊断动作草稿确认数 / 动作草稿总数；确认数只统计同一批诊断 `requestId` 关联的 `agent_action_audit` 成功记录，避免跨日期或跨餐次审计记录污染当前看板

查询条件:

- `recordDateStart`、`recordDateEnd`：按诊断日期过滤诊断指标和反馈记录
- `mealType`：按餐次过滤诊断指标和反馈记录
- 动作确认审计表没有独立餐次字段，统计时通过诊断指标的 `requestId` 与审计记录对齐

当前反馈流程:

- 前端从诊断结果 `reasons[].code` 提取预测原因码
- 客服点击采纳、不采纳或部分正确后打开反馈弹窗
- 客服可选择或手工录入真实原因码，并补充备注
- 后端保存到 `agent_diagnosis_feedback` 表
- 后端提供反馈查询和统计接口，用于后续运营看板

当前接口:

- `POST /api/agent/feedback`：提交反馈，权限 `agentDiagnosis:feedback`
- `GET /api/agent/feedback`：分页查询反馈，权限 `agentDiagnosis:list`
- `GET /api/agent/feedback/stats`：统计反馈采纳率和真实原因分布，权限 `agentDiagnosis:list`

当前反馈表:

- 表名：`agent_diagnosis_feedback`
- 建表脚本：`eladmin/sql/agent_diagnosis_feedback.sql`
- 关键字段：`request_id`、`session_id`、`customer_id`、`record_date`、`meal_type`、`predicted_reason_codes`、`accepted`、`actual_reason_code`、`comment`、`operator`、`create_time`

统计口径:

- `accepted=ACCEPTED` 计入采纳数
- `accepted=REJECTED` 计入不采纳数
- `accepted=PARTIAL` 计入部分正确数
- 采纳率 = 采纳数 / 反馈总数
- 真实原因分布按 `actual_reason_code` 分组

反馈数据后续用于识别高频错误、未知原因和规则缺口，并反向补充评测集与规则库。

## 19. 三期规则缺口维护

客服反馈不采纳、部分正确，或录入了当前规则库未知的真实原因码时，系统会把反馈沉淀为规则缺口，进入规则维护池。

当前规则缺口来源:

- `accepted=REJECTED`：诊断结论错误，沉淀为 `WRONG_REASON`
- `accepted=PARTIAL`：诊断结论部分正确，沉淀为 `PARTIAL_REASON`
- `actualReasonCode` 不在当前稳定原因码集合内：沉淀为 `UNKNOWN_REASON`

缺口聚合规则:

- 同一 `actualReasonCode` 在 `OPEN` 或 `IN_PROGRESS` 状态下只保留一条维护记录
- 重复命中时累加 `occurrence_count`，并刷新 `update_time`
- 客服未填写真实原因码时按 `UNKNOWN` 聚合

当前接口:

- `GET /api/agent/rule-gaps`：分页查询规则缺口，权限 `agentDiagnosis:list`
- `PUT /api/agent/rule-gaps/{id}/status`：更新缺口状态、处理人和备注，权限 `agentDiagnosis:rule`

当前规则缺口表:

- 表名：`agent_rule_gap`
- 建表脚本：`eladmin/sql/agent_rule_gap.sql`
- 关键字段：`gap_type`、`source_type`、`source_id`、`request_id`、`customer_id`、`record_date`、`meal_type`、`predicted_reason_codes`、`actual_reason_code`、`accepted`、`gap_description`、`occurrence_count`、`status`、`owner`、`operator`、`create_time`、`update_time`

规则维护闭环:

- 运营人员优先按 `occurrence_count` 和 `update_time` 处理高频缺口
- 规则负责人确认后将状态流转为 `IN_PROGRESS`，后端强制要求填写 `owner`
- 规则补充、评测集补充和发布验证完成后流转为 `RESOLVED`，后端强制要求填写处理备注，备注应包含规则 YAML、提示词策略、评测用例或发布验证证据
- 确认不是规则问题或无需处理时流转为 `IGNORED`
- 关闭前应同步补充规则 YAML、提示词策略、评测集案例或业务说明，避免相同问题再次只靠人工经验处理
- 非法状态、缺少处理人或缺少解决证据时返回 `VALIDATION_FAILED`，不会更新缺口记录

## 20. 智能客服业务查询会话恢复

智能客服只读查询在助手消息中持久化受控的业务查询卡片快照，确保主系统刷新、Agent 服务重启或多实例切换后，客服仍能看到与当时回答一致的事实、查询时间和查询计划。

- 快照列：`agent_chat_message.business_result_json`，升级脚本为 `eladmin/sql/agent_chat_message_business_query_migration.sql`。
- 快照仅包含 `responseType`、脱敏 `insightResult`、`facts`、`warnings`、`partial`、`queriedAt` 和 `queryPlan`。
- 不保存内部工具原始响应、提示词全文、手机号明文、完整配送地址、特殊要求全文或任何订单/退款金额字段。
- 历史会话恢复直接使用快照渲染结构化卡片，不重新发起业务查询；再次提问时才按当次客服权限重新查询。

## 21. 智能客服业务查询审计

每次 `BUSINESS_QUERY*` 只读响应都会写入 `agent_business_query_audit`，用于按客服、会话、请求、客户和工具定位问题。

- 建表脚本：`eladmin/sql/agent_business_query_audit.sql`。
- 审计默认保留 180 天；每天 03:15 按 `create_time` 清理过期记录。可通过 `AGENT_AUDIT_RETENTION_DAYS` 和 `AGENT_AUDIT_RETENTION_CRON` 配置，保留天数小于 1 时显式禁用清理。
- 清理任务仅删除 `agent_business_query_audit` 的过期记录，不删除会话、聊天消息、反馈、动作审计或任何客户、订单、排餐业务数据。
- 审计字段包含客服账号、会话/请求 ID、查询领域/动作、客户/订单标识、工具名称摘要、结果条数、部分成功标记、稳定失败类型和耗时；同时记录受控 `analysisSource`、`analysisConfidence`、`clarificationRequired`、指标/维度代码、未支持原因和回答校验结果。
- 审计不保存原始提示词、模型原始输出、工具原始响应、手机号、完整地址、金额或自由 SQL。
- 部分回答会优先记录工具返回的稳定告警码；运营统计据此聚合权限/身份拒绝次数（`TOOL_PERMISSION_DENIED`、`AGENT_QUERY_ACCESS_DENIED`、`AGENT_QUERY_UNAUTHORIZED`），避免被 `ANSWERED` 聊天状态掩盖。
- 审计统计接口同时计算直接回答率、关键澄清率、澄清后成功回答率、回答安全校验拒绝率、平均/P95 耗时、领域/工具/指标分布和未支持问题原因 Top N；诊断反馈采纳率继续由既有反馈统计接口提供。
- 严禁写入手机号、完整地址、特殊要求全文、工具原始入参/出参、模型提示词及任何订单或退款金额。
- 审计数据默认保留期限由生产环境数据治理策略配置；上线前需由运维确认实际保留周期。

## 22. 统一业务语义、时间与跨实例续接

业务问答采用统一受控语义协议：LLM 负责选择领域、登记指标、时间表达、交互模式和实质歧义；服务端负责确定性槽位、日期落地、QueryPlan 编译、权限校验、工具执行与结果校验。

- 指标语义来自版本化 `AgentMetricCatalog`，Prompt 只包含指标枚举、业务定义、支持维度和默认时间，不暴露工具名、结果字段、URL 或数据库实现。
- 相对时间只允许 `CURRENT_DAY`、`PREVIOUS_DAY`、`NEXT_DAY`、`CURRENT_WEEK` 等枚举；具体日期由 `BusinessTemporalResolver` 按 `Asia/Shanghai` 解析，测试使用固定 `Clock`。
- 指标目录同时登记固定展示名、结果单位、结果字段和只读工具。聊天服务不再根据用户原始中文二次选择运营指标或结果字段。
- 模型不可用、非法输出、低于置信度阈值或命中实质规则保护冲突时进入高精度规则兜底，并在 `semanticTraceSummary` 中记录 `semanticSource` 和最长 32 位的稳定 `fallbackReason`；规则保护冲突使用 `MODEL_RULE_GUARDRAIL_CONFLICT`，主系统兼容转换旧码 `MODEL_CONFLICTS_WITH_RULE_GUARDRAIL`，正常澄清不属于 fallback。
- 会话表通过 `pending_business_query_json` 和 `last_business_query_context_json` 保存受控上下文。主系统每轮下发，Agent 响应每轮写回，内存状态仅为性能优化。
- Pending 只保存受控枚举、实体标识、日期、餐次、缺失字段、限长摘要和过期时间，不保存用户原文、Prompt、模型原始响应、工具原始结果、手机号、地址或金额。
- 纯日期、餐次或编号回复优先补 Pending；新完整问题清除旧 Pending 后重新分析。重置、查询成功和超时同样清除 Pending。
