# 订单管理

## 业务定位

订单管理是系统的交易核心，记录客户购买的餐数套餐、金额、核销进度和餐费余额。一个客户可以有多笔订单（增购时新建订单），且可以同时存在多笔进行中的订单，但同一天同一餐次不能冲突（详见"订单冲突校验"）。订单状态驱动整个业务流程：进行中（排餐/核销）→ 已完成 / 已取消（终止）。

---

## 核心概念

### 订单生命周期

```
已取消(0) ──┐
            │
进行中(1) ──┼──→ 已完成(2)
            │      ↑
            │      └── 核销完所有餐数或到期后手动标记
            │
            ├──→ 已退餐(3)
            │      ↑
            │      └── 退餐操作（取消订单+取消排餐）
            │
            └── 客户增购时：新建订单（老订单待状态变更）
```

状态流转规则：
- `status=0`（已取消）：手动取消，不可逆
- `status=1`（进行中）：正常服务状态，可核销、可排餐
- `status=2`（已完成）：服务结束，可由进行中手动标记完成
- `status=3`（已退餐）：退餐操作后自动变更，不可逆

### 餐数体系

订单中餐数分为两套独立账户：

| 餐次 | 字段 | 说明 |
|------|------|------|
| 早餐 | `breakfast_count` | 独立计算，与午晚餐分开 |
| 午餐+晚餐 | `lunch_dinner_count` | 共享一个餐数池 |

对应核销时：`meal_verification_log.meal_type` 为 `BREAKFAST` 时计入早餐核销，为 `LUNCH`/`DINNER` 时计入午晚餐核销。

### 菜品数量配置（Phase4 新增）

订单中包含每餐各类菜品的数量配置，用于排餐时的菜品分配：

| 字段 | 类型 | 说明 |
|------|------|------|
| `main_dish_count` | INT | 每餐主菜/荤菜数量 |
| `side_dish_count` | INT | 每餐副菜数量 |
| `veg_count` | INT | 每餐素菜数量 |
| `rice_count` | INT | 每餐米饭数量，默认1 |
| `rice_type` | VARCHAR | 米饭类型，默认白米饭 |
| `soup_count` | INT | 每餐汤数量 |

**规则**：超出每日固定提供的1份部分计入补菜数量（`supplementaryCount = max(0, 需求数 - 1)`）。米饭类型（`riceType`）可选"白米饭"、"普通杂粮米饭"、"杂粮1:1米饭"、"三色糙米"等，排餐时按类型优先匹配。

### 金额体系

| 字段 | 说明 |
|------|------|
| `total_amount` | 总价（原价） |
| `final_amount` | 成交价（实际付款） |
| `deposit_amount` | 已付定金 |
| `verified_amount` | 已核销金额 |
| `meal_balance` | 餐费余额 = `final_amount - verified_amount` |

`meal_balance` 由后端自动计算，不在前端传入。

### 排餐模式（schedule_mode）

| 模式 | code | 说明 |
|------|------|------|
| 每天送 | `DAILY` | 每天送餐 |
| 工作日送 | `WEEKDAY` | 工作日送餐 |
| 周末送 | `WEEKEND` | 周末送餐 |
| 指定日期送 | `SCHEDULE` | `delivery_dates` 指定具体日期列表 |

`delivery_dates` 支持两种格式：

**旧格式（仅日期）**：
```json
["2026-04-01", "2026-04-03"]
```

**新格式（日期+餐次）**：
```json
[
  {"date": "2026-04-01", "mealTypes": ["LUNCH", "DINNER"]},
  {"date": "2026-04-02", "mealTypes": ["LUNCH"]}
]
```

### 餐次类型（meal_type）

| 类型 | code | 说明 |
|------|------|------|
| 全餐次 | `ALL` | 早+午+晚餐（默认） |
| 午餐+晚餐 | `LUNCH_DINNER` | 午餐和晚餐（不包含早餐） |
| 午餐 | `LUNCH` | 仅午餐 |
| 晚餐 | `DINNER` | 仅晚餐 |

### 开始餐次（start_meal_type）

订单开始日期可以继续细化到具体餐次，控制**开始日期当天**从哪一餐开始生效：

| 类型 | code | 说明 |
|------|------|------|
| 早餐开始 | `BREAKFAST` | 从当天早餐开始（默认） |
| 午餐开始 | `LUNCH` | 当天早餐不排，从午餐开始 |
| 晚餐开始 | `DINNER` | 当天早餐、午餐不排，从晚餐开始 |

约束规则：

- `meal_type = ALL` 时，`start_meal_type` 可选 `BREAKFAST / LUNCH / DINNER`
- `meal_type = LUNCH_DINNER` 时，`start_meal_type` 可选 `LUNCH / DINNER`
- `meal_type = LUNCH` 时，`start_meal_type` 只能是 `LUNCH`
- `meal_type = DINNER` 时，`start_meal_type` 只能是 `DINNER`

非数据库字段（查询时填充）：
- `customer_name`：客户姓名
- `phone`：客户手机号
- `total_count`：合计餐数 = `breakfast_count + lunch_dinner_count`
- `scheduled_count`：已排餐餐数 = 当前订单全部有效排餐记录总数（`meal_plan_customer.deleted=0` 且 `meal_plan.deleted=0`）
- `estimated_remaining_count`：预计剩余餐数 = `remaining_count - 今日已排餐但未核销餐数`，最小值为 0
- `parent_package_name` / `child_package_name`：套餐名称
- `allergy_tags`：客户过敏标签（来自 `customer_profile`）
- `special_requirements`：客户特殊要求（来自 `customer_profile`）

前端列表页显示规则：
- **含汤**：`soup_count >= 1` 时显示"含汤"，否则显示"不含汤"
- **特殊要求**：显示 `special_requirements` 字段内容，为空时显示"-"
- **自定义菜单**：有图片时显示缩略图（点击可预览大图），无图片时显示"-"
- **已排餐**：显示当前订单全部有效排餐记录总数，不区分是否已核销
- **预计剩余餐数**：使用当前订单 `remaining_count` 扣减今日已生成、成功、未核销的排餐记录数；今天已核销的排餐已体现在 `remaining_count` 中，不重复扣减。

### 自定义菜单图片说明

- `custom_menu_image` 字段存储图片访问路径（如 `/file/image/20260529112233_abcd.jpg`），不存储物理路径
- 图片上传复用现有 `/api/localStorage/pictures` 接口，上传目录由 `file.path` 配置决定
- 替换或删除图片只清空订单字段，不删除物理文件，避免误删已复用文件
- 该图片作为订单维度的参考附件，不参与排餐生成、过敏过滤或换菜规则计算

## 核心业务流程

### 1. 创建订单（新增）

销售为已有客户增购餐数时创建新订单。系统**不自动关闭**老订单，需手动将老订单状态改为"已完成"。

**创建流程：**

1. **订单冲突校验**（基于 `startDate` + `mealType` 互斥，`status=1` 的订单参与校验）
   - **ALL 类型**：同 customerId + 同 startDate + status=1 不能有任何其他订单（ALL/LUNCH_DINNER/LUNCH/DINNER 均互斥）
   - **LUNCH_DINNER 类型**：不能与 ALL、LUNCH_DINNER、LUNCH、DINNER 共存
   - **LUNCH 或 DINNER 类型**：① 不能有 LUNCH_DINNER ② 最多 2 个不同餐次 ③ 不能有同餐次
   - startDate 为空时跳过冲突校验

2. **参数校验**
   - `final_amount ≤ total_amount`
   - `verified_count ≤ total_count`（`breakfast_count + lunch_dinner_count`）
   - `verified_amount ≤ final_amount`
   - `start_meal_type` 必须与 `meal_type` 兼容
   - 餐数默认值：早餐=0，午餐晚餐=0
   - 开始餐次默认值：`ALL → BREAKFAST`，`LUNCH/LUNCH_DINNER → LUNCH`，`DINNER → DINNER`
   - 米饭数量默认值：1
   - 米饭类型默认值：`白米饭`
   - 结束日期校验已注释（当前不做 endDate < startDate 检查）

3. **自动计算字段**
   - `meal_balance = final_amount - verified_amount`
   - `remaining_count = breakfast_count + lunch_dinner_count - verified_count`（存储在订单表中，核销时原子递减）

4. **试餐成单关联**
   - 新增客户首单和订单新增/编辑均支持选择“是否试餐成单”
   - `trial_converted=1` 时必须选择 `trial_order_id`
   - 可关联订单必须来自父套餐名称包含“试餐”的订单
   - `trial_converted=0` 时后端会清空 `trial_order_id`

5. **生成订单编号**
   - 格式：`ORD{yyyyMMdd}{NNN}`（当日序号三位）
   - 支持并发重试（最多3次），遇到重复键自动重试

### 2. 退餐（取消订单）

退餐是指客户需要取消订单的场景，相当于**取消订单 + 取消排餐**。

**业务规则：**

- 只有 `status=1`（进行中）的订单可以退餐
- 退餐后订单状态变为 `status=3`（已退餐）
- 退款金额 = 剩余餐数 × 单价（**只退未核销部分**）
- 已核销的餐数不退款（已享受服务）
- 核销日志保留，但标记为 `is_refunded=1`

**退款金额计算：**

```
退款金额 = 剩余早餐数 × 早餐单价 + 剩余午晚餐数 × 午晚餐单价

其中：
剩余早餐数 = 订单早餐数 - 已核销早餐数
剩余午晚餐数 = 订单午晚餐数 - 已核销午晚餐数
```

**退餐流程：**

1. 校验订单状态是否为进行中
2. 查询订单核销日志，统计已核销的各餐次数量
3. 计算剩余餐数和退款金额
4. 更新订单状态为已退餐（`status=3`）
5. 标记所有核销日志为已退餐（`is_refunded=1`）
6. 记录退餐日志（`meal_refund_log`）

**退餐后订单字段变更：**

| 字段 | 变更 |
|------|------|
| `status` | `1` → `3`（已退餐） |

退餐相关信息（退款金额、原因、操作人）存储在退餐日志表 `meal_refund_log` 中。

### 2. 核销餐数（与核销管理模块联动）

`meal_verification_log` 记录每次核销，订单以下字段由核销操作触发原子更新：

```
订单 verified_count   += 1
订单 verified_amount  += 单价
订单 meal_balance     -= 单价
订单 remaining_count  -= 1
```

当 `remaining_count`（合计剩余餐数）减至 0 时，系统在核销事务内**自动将订单状态更新为已完成**（`status=2`）。此逻辑保证餐数耗尽和订单完结的一致性。

### 3. 订单金额结算

餐费余额（`meal_balance`）是追踪客户未核销金额的核心指标：
- `meal_balance = final_amount - verified_amount`
- 当客户增购时，`final_amount` 增大，`meal_balance` 相应变化
- 订单取消时，`meal_balance` 理论上应为 0（若已有核销则需退款处理）

### 4. 订单级换菜规则

订单支持配置换菜规则（`customer_order_replace_rule`），在排餐计划生成时自动应用，将指定原菜替换为目标菜。

**数据模型：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `order_id` | BIGINT | 关联订单ID |
| `source_dish_id` | BIGINT | 原菜品ID |
| `source_dish_name` | VARCHAR | 原菜品名称（快照） |
| `source_dish_type` | VARCHAR | 原菜品类型（快照） |
| `target_dish_id` | BIGINT | 目标菜品ID |
| `target_dish_name` | VARCHAR | 目标菜品名称（快照） |
| `target_dish_type` | VARCHAR | 目标菜品类型（快照） |
| `enabled` | TINYINT | 是否启用 |
| `remark` | VARCHAR | 备注 |
| `deleted` | TINYINT | 软删除标记 |

**业务规则：**

- 同一订单内 `source_dish_id` 不能重复（一个原菜只能对应一个目标菜）
- 原菜和目标菜不能相同
- 换菜规则在订单创建/编辑时保存，采用"全量替换"策略：编辑时先软删除旧规则再重新插入
- 允许跨 `dish_type` 替换（如主菜替换为素菜）
- 换菜规则优先级最高，在排餐计划生成时**跳过过敏/忌口过滤**
- 替换后记录 `original_dish_type`（原菜类型），用于前端展示替换前后的类型变化
- 若目标菜被禁用或删除，排餐时跳过该规则（保留原菜选择）
- 规则以快照方式保存菜品名称和类型，菜品主档变更不影响已保存的规则
- 订单查询和详情接口会返回关联的换菜规则列表

### 5. 订单与客户编号

客户编号（`customer_code`）在**创建客户档案时**由父套餐的编号池生成，存储在 `customer_profile`。订单表中 `customer_code` 为冗余字段，用于查询展示，不参与业务流程。

编辑订单时，只有在修改了 `parent_package_id` 的情况下，系统才会按新父套餐编号池刷新**当前订单**的 `customer_code`，并同步更新 `customer_profile.customer_code`。该刷新不会影响该客户的其他历史订单。

## 与其他模块的关系

```
订单管理（customer_order）
  │
  ├── 客户管理（customer_profile）
  │     ├── 客户档案 ID → 关联客户基本信息
  │     └── allergy_tags → 核销/排餐时参考
  │
  ├── 套餐管理（parent_package / sub_package）
  │     └── parent_package_id / child_package_id → 确定餐数规格
  │
  ├── 核销管理（meal_verification_log）
  │     ├── 核销记录累加更新订单 verified_count / verified_amount
  │     └── 影响 remaining_count / meal_balance
  │
  └── 排餐管理（meal_plan / meal_plan_customer）
        ├── 订单 status=1 时客户计入排餐范围
        ├── start_date / end_date / delivery_dates 决定排餐日期
        ├── main_dish_count / side_dish_count 等决定菜品分配
        └── customer_order_replace_rule → 排餐时自动应用换菜规则

订单换菜规则（customer_order_replace_rule）
  ├── 关联 customer_order.id → 订单维度配置
  ├── 关联 dish.id（source/target）→ 菜品快照
  └── 排餐计划生成时读取 → 替换选中菜品
```

**重要**：删除订单不影响核销记录（`meal_verification_log` 独立保留）；删除排餐记录也不影响核销数据。

## 业务规则汇总

| 规则 | 说明 |
|------|------|
| 餐数独立计算 | 早餐和午晚餐分开，不混用 |
| 餐费余额 | `final_amount - verified_amount`，自动计算 |
| 剩余餐数 | `breakfast_count + lunch_dinner_count - verified_count`，存储在订单表中，核销时原子递减 |
| 订单编号 | `ORD{yyyyMMdd}{NNN}`，当日三位序号 |
| 编号预览 | `GET /api/customerProfile/generateCode?parentPackageId=X` |
| 进行中订单限制 | 有 `status=1` 订单的客户不允许删除档案 |
| 增购不关老单 | 增购时新建订单，老订单状态不变（待手动完结） |
| 订单冲突校验 | 基于 `startDate` + `mealType` 互斥，`status=1` 的订单参与校验（详见冲突校验章节） |
| 退餐规则 | 只有进行中订单可退餐，退款 = 剩余未核销早餐数×早餐单价 + 剩余未核销午晚餐数×午晚餐单价 |
| 退餐状态变更 | 退餐后订单 status → 3（已退餐），核销日志标记 is_refunded=1 |
| 餐数耗尽自动完单 | remaining_count 减至 0 时，核销事务内自动将订单 status → 2（已完成） |
| 核销回退 | 删除核销日志时自动回退：verified_count-1, remaining_count+1, verified_amount-价格；若已完成自动回退为进行中 |
