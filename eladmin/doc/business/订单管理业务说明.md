# 订单管理

## 业务定位

订单管理是系统的交易核心，记录客户购买的餐数套餐、金额、核销进度和餐费余额。一个客户可以有多笔订单（增购时新建订单），但同一时间只有一笔进行中的订单有效。订单状态驱动整个业务流程：进行中（排餐/核销）→ 已完成 / 已取消（终止）。

---

## 核心概念

### 订单生命周期

```
已取消(0) ──┐
            │
进行中(1) ──┼──→ 已完成(2)
            │      ↑
            │      └── 核销完所有餐数或到期后手动标记
            │
            └── 客户增购时：新建订单（老订单待状态变更）
```

状态流转规则：
- `status=0`（已取消）：手动取消，不可逆
- `status=1`（进行中）：正常服务状态，可核销、可排餐
- `status=2`（已完成）：服务结束，可由进行中手动标记完成

### 餐数体系

订单中餐数分为两套独立账户：

| 餐次 | 字段 | 说明 |
|------|------|------|
| 早餐 | `breakfast_count` | 独立计算，与午晚餐分开 |
| 午餐+晚餐 | `lunch_dinner_count` | 共享一个餐数池 |

对应核销时：`meal_verification_log.meal_type` 为 `BREAKFAST` 时计入早餐核销，为 `LUNCH`/`DINNER` 时计入午晚餐核销。

### 金额体系

| 字段 | 说明 |
|------|------|
| `total_amount` | 总价（原价） |
| `final_amount` | 成交价（实际付款） |
| `deposit_amount` | 已付定金 |
| `verified_amount` | 已核销金额 |
| `meal_balance` | 餐费余额 = `final_amount - verified_amount` |

`meal_balance` 由后端自动计算，不在前端传入。

### 排餐模式（schedule_mode）

| 模式 | code | 说明 |
|------|------|------|
| 每天送 | `DAILY` | 每天送餐 |
| 工作日送 | `WEEKDAY` | 工作日送餐 |
| 周末送 | `WEEKEND` | 周末送餐 |
| 指定日期送 | `SCHEDULE` | `delivery_dates` 指定具体日期列表 |

`delivery_dates` 为 JSON 数组格式：`["2026-04-01", "2026-04-03"]`。

### 餐次类型（meal_type）

| 类型 | code | 说明 |
|------|------|------|
| 全餐次 | `ALL` | 早+午+晚餐（默认） |
| 午餐 | `LUNCH` | 仅午餐 |
| 晚餐 | `DINNER` | 仅晚餐 |

## 数据模型

| 字段 | 类型 | 说明 |

| `customer_id` | BIGINT | 客户ID |

| `final_amount` | DECIMAL | 成交金额 |
| `breakfast_count` | INT | 早餐合计餐数 |
| `lunch_dinner_count` | INT | 午餐+晚餐合计餐数 |
| `breakfast_price` | DECIMAL | 早餐单价 |
| `lunch_dinner_price` | DECIMAL | 午餐晚餐单价 |
| `verified_count` | INT | 已核销餐数（合计） |
| `verified_amount` | DECIMAL | 已核销金额 |
| `meal_balance` | DECIMAL | 餐费余额（自动计算） |
| `remaining_count` | INT | 剩余餐数（自动计算） |
| `deal_time` | DATETIME | 成交时间 |
| `first_delivery_time` | DATETIME | 第一次送餐时间 |
| `start_date` | DATE | 订单开始日期 |
| `end_date` | DATE | 订单结束日期 |
| `status` | TINYINT | 状态：0=已取消，1=进行中，2=已完成 |
| `meal_type` | VARCHAR | 餐次类型：ALL / LUNCH / DINNER |
| `schedule_mode` | VARCHAR | 排餐模式 |

| `customer_source` | VARCHAR | 销售渠道 |
| `remark` | VARCHAR | 备注 |

非数据库字段（查询时填充）：
- `customer_name`：客户姓名
- `phone`：客户手机号
- `total_count`：合计餐数 = `breakfast_count + lunch_dinner_count`
- `parent_package_name` / `child_package_name`：套餐名称
- `allergy_tags`：客户过敏标签（来自 `customer_profile`）

## 核心业务流程

### 1. 创建订单（新增）

销售为已有客户增购餐数时创建新订单。系统**不自动关闭**老订单，需手动将老订单状态改为"已完成"。

业务规则校验：
- `final_amount ≤ total_amount`
- `verified_count ≤ total_count`（`breakfast_count + lunch_dinner_count`）
- `verified_amount ≤ final_amount`
- `end_date ≥ start_date`

### 2. 核销餐数（与核销管理模块联动）

`meal_verification_log` 记录每次核销，订单的 `verified_count` 和 `verified_amount` 由核销操作触发更新：

```
订单 verified_count += 本次核销数量
订单 verified_amount += 本次核销数量 × 对应单价
订单 meal_balance = final_amount - verified_amount
```

当 `remaining_count`（合计剩余餐数）变为 0 时，订单不会自动变为"已完成"，需人工操作。

### 3. 订单金额结算

餐费余额（`meal_balance`）是追踪客户未核销金额的核心指标：
- `meal_balance = final_amount - verified_amount`
- 当客户增购时，`final_amount` 增大，`meal_balance` 相应变化
- 订单取消时，`meal_balance` 理论上应为 0（若已有核销则需退款处理）

### 4. 订单与客户编号

客户编号（`customer_code`）在**创建客户档案时**由父套餐的编号池生成，存储在 `customer_profile`。订单表中 `customer_code` 为冗余字段，用于查询展示，不参与业务流程。

## 与其他模块的关系

```
订单管理（customer_order）
  │
  ├── 客户管理（customer_profile）
  │     ├── 客户档案 ID → 关联客户基本信息
  │     └── allergy_tags → 核销/排餐时参考
  │
  ├── 套餐管理（parent_package / sub_package）
  │     └── parent_package_id / child_package_id → 确定餐数规格
  │
  ├── 核销管理（meal_verification_log）
  │     ├── 核销记录累加更新订单 verified_count / verified_amount
  │     └── 影响 remaining_count / meal_balance
  │
  └── 排餐管理（dish_schedule_record）
        ├── 订单 status=1 时客户计入排餐范围
        └── start_date / end_date / delivery_dates 决定排餐日期
```

**重要**：删除订单不影响核销记录（`meal_verification_log` 独立保留）；删除排餐记录也不影响核销数据。

## 业务规则汇总

| 规则 | 说明 |
|------|------|
| 餐数独立计算 | 早餐和午晚餐分开，不混用 |
| 餐费余额 | `final_amount - verified_amount`，自动计算 |
| 剩余餐数 | `total_count - verified_count`，自动计算 |
| 订单编号 | `ORD{yyyyMMdd}{NNN}`，当日三位序号 |
| 编号预览 | `GET /api/customerProfile/generateCode?parentPackageId=X` |
| 进行中订单限制 | 有 `status=1` 订单的客户不允许删除档案 |
| 增购不关老单 | 增购时新建订单，老订单状态不变（待手动完结） |
