# 配菜管理

## 业务定位

配菜管理是系统的核心基础模块，负责管理菜品主档、配料字典、以及排餐计划生成。菜品是面向客户的最终交付物，配菜管理的质量直接决定客户体验。

---

## 核心概念

### 菜品类型（DishType）

一顿标准餐食由以下类型组合构成：

| 类型 | code | 说明 |
|------|------|------|
| 主菜 | `MAIN` | 核心荤菜，每餐 1 个 |
| 副菜 | `SIDE` | 辅助菜品，每餐 1 个 |
| 汤 | `SOUP` | 汤品，可选 |
| 素菜 | `VEGETABLE` | 蔬菜，可选 |
| 米饭 | `RICE` | 主食，每餐 1 个 |

一顿完整月子餐通常为：主菜 + 副菜 + 汤 + 素菜 + 米饭（共 5 个菜品槽位）。

### 餐次（MealType）

| 类型 | 说明 |
|------|------|
| `BREAKFAST` | 早餐（独立计算餐数） |
| `LUNCH` | 午餐（计入午餐晚餐共用池） |
| `DINNER` | 晚餐（计入午餐晚餐共用池） |

### 排期格式（Schedule）

格式为 `W-D`，W = 周序号（1-4），D = 星期（1-7，周一=1）。

```
["1-1", "2-1", "3-1", "4-1"]  →  每周一出现
["3-5"]                       →  第3周周五出现
["1-1", "1-2", "1-3", "1-4", "1-5"]  →  第1周工作日出现
```

### 菜品与套餐的关系（两套体系，注意区分）

**第一套：餐品线代码（`dish.meal_packages`，`MealPackageEnum`）**

用于菜品维度的分类，是枚举常量，独立于数据库表：

| code | 含义 | 说明 |
|------|------|------|
| `yuezi` | 月子餐 | 产后调理 |
| `yunqi` | 孕期餐 | 孕期营养 |
| `xiaoyuezi` | 小月子 | 流产后调理 |
| `yingyang` | 营养餐 | 通用营养 |
| `fenmian` | 分娩餐 | 分娩前后 |

一个菜品可以属于多个餐品线（如 `"mealPackages": ["yuezi", "yingyang"]`）。

**第二套：父子套餐结构（`parent_package` / `sub_package`）**

用于客户订单维度，与菜品管理模块独立。详见套餐管理业务说明。

**两者的区别**：餐品线代码（枚举）是菜品属性，决定某道菜出现在哪个产品线；父子套餐是客户购买的商品规格，影响订单计费和餐数分配。两者没有直接关联，但都叫"套餐"，容易混淆。

## 数据模型

### dish — 菜品主档

| 字段 | 类型 | 说明 |
|------|------|------|
| `name` | VARCHAR | 菜品名称 |
| `cooking_method` | TEXT | 做法流程 |
| `ingredients` | TEXT | 配料（自由文本，冗余备用） |
| `image_url` | VARCHAR | 图片路径 |
| `dish_type` | VARCHAR | 菜品类型：MAIN / SIDE / SOUP / VEGETABLE / RICE |
| `meal_types` | JSON Array | 适用餐次：`["LUNCH"]` / `["DINNER"]` / `["LUNCH","DINNER"]` |
| `meal_packages` | JSON Array | 所属餐品线代码：`["yuezi"]` / `["yunqi","yingyang"]` |
| `schedule` | JSON Array | 排期，格式 `["1-1","2-3"]` |
| `sort` | INT | 排序值，越小越靠前 |
| `enabled` | TINYINT | 是否启用 |

非数据库字段：
- `ingredientList`：配料详情列表（来自 `dish_ingredient_relation` JOIN）
- `mealPackageDetails`：套餐详情（来自 `parent_package` 表，用于表格展示，与 meal_packages JSON 无直接关系）

### dish_ingredient — 配料字典

菜品食材的主档字典，可复用。

| 字段 | 类型 | 说明 |
|------|------|------|
| `name` | VARCHAR | 配料名称，唯一 |
| `category` | VARCHAR | 分类：MEAT / VEGETABLE / SEAFOOD / TOFU / SPICE / OTHER |
| `unit` | VARCHAR | 单位：克(g)、毫升(ml)、个 |
| `calories` | INT | 每单位热量（卡路里） |

### dish_ingredient_relation — 菜品配料关联

菜品与配料的多对多关系表，记录用量和备注。

| 字段 | 类型 | 说明 |
|------|------|------|
| `dish_id` | INT | 菜品ID |
| `ingredient_id` | INT | 配料ID |
| `ingredient_name` | VARCHAR | 配料名称（冗余存储） |
| `unit` | VARCHAR | 单位（冗余存储） |
| `quantity` | DOUBLE | 用量 |
| `remark` | TEXT | 备注，如"少盐"、"切丁" |

## 核心业务流程

### 1. 菜品维护

新建菜品 → 填写基本信息 → 关联配料 → 设置排期和适用餐次/餐品线 → 保存。

编辑时配料列表通过 `dish_ingredient_relation` 表管理，增删改都在该表进行。

### 2. 排餐计划生成

触发条件：定时任务或管理员手动调用 `/api/dishes/schedule/{date}?mealType=ALL`

流程：
1. 根据日期计算周数（W）和星期（D）
2. 查找 `schedule` 匹配 `W-D` 的菜品
3. 按餐品线（`meal_packages`）分组
4. 若传了 `customerId`，只为指定客户排餐；否则为所有生效客户排餐
5. 根据客户忌口（`customer_dietary_restrictions`）过滤/替换菜品
6. 保存排餐记录到 `dish_schedule_record` 和 `customer_menu_record`

**忌口替换逻辑**：如果菜品 `ingredients` 字段包含客户的任一忌口词，该菜品被替换为同类别的其他可用菜品，并记录 `isReplaced=true` 和替换原因。

### 3. 客户可用菜品查询

`GET /api/dishes/available?customerId=X&mealType=LUNCH&week=2&day=3`

根据客户ID查忌口列表，对菜品 `ingredients` 字段做关键词匹配，返回不过敏的菜品。传了排期参数时额外按 `schedule` 筛选。

### 4. 首页排餐统计

`GET /api/dishes/schedule/stats/{date}` 返回指定日期各套餐线的客户数、替换数、各菜品类型的菜品名称和替换统计，用于销售看板。

## 与其他模块的关系

```
客户档案（customer_profile）
  ├── 客户饮食限制（customer_dietary_restrictions）
  │     → 配菜时过滤/替换菜品
  ├── 客户订单（customer_order）
  │     → 关联套餐，决定客户应享受的餐品线
  └── 客户关键词（customer_keywords）
        → 配菜参考偏好

配菜管理（dish / dish_ingredient）
  ├── 排餐计划（dish_schedule_record / customer_menu_record）
  │     → 基于菜品和排期生成
  └── 核销管理（meal_verification_log）
        → 客户实际核销的餐次，不影响菜品数据
```

**注意**：核销管理（MealVerificationLog）和排餐计划是两个独立流程：
- 排餐计划：系统预先编排的菜品
- 核销管理：客户实际取餐的记录

两者通过 `customer_id` + `日期` 关联，但删除排餐记录不会影响核销数据。
