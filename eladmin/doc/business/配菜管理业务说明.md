# 配菜管理

## 业务定位

配菜管理是系统的核心基础模块，负责菜品主档、配料字典（含三层分类）、以及"周-星期-餐次-坑位"维度的排期模板。菜品是面向客户的最终交付物，配菜管理的质量直接决定客户体验和排餐结果。

---

## 核心概念

### 菜品类型（DishType）

一顿标准餐食由以下类型组合构成：

| 类型 | code | 说明 |
|------|------|------|
| 主菜 | `MAIN` | 核心荤菜，每餐 1 个 |
| 副菜 | `SIDE` | 辅助菜品，每餐 1 个 |
| 汤 | `SOUP` | 汤品，可选 |
| 素菜 | `VEGETABLE` | 蔬菜，可选 |
| 米饭 | `RICE` | 主食占位，仅在排餐明细中出现 |
| 米饭类型 | `RICE_TYPE` | 主档侧的米饭档案（白米饭、糙米饭等），订单选米饭、排餐选米饭都基于此类型 |

`RICE_TYPE` 与 `RICE` 的关系：
- 主档（`dish` 表）和订单（`CustomerOrder.riceType`）使用 `RICE_TYPE` 区分白米饭、糙米饭等不同米饭档案；
- 排餐执行时不区分米饭类型（先按订单的 `riceType` 匹配，匹配不上回退白米饭），但写入 `meal_plan_customer_item.dish_type` 时统一归类为 `RICE`；
- `RICE_TYPE` 是全局通用菜品，不绑定任何父套餐，所有套餐都共享同一池米饭。

一顿完整月子餐通常为：主菜 + 副菜 + 汤 + 素菜 + 米饭（共 5 个菜品槽位）。

### 餐次（MealType）

| 类型 | 说明 |
|------|------|
| `BREAKFAST` | 早餐（独立计算餐数，不进入排餐计划） |
| `LUNCH` | 午餐（计入午餐晚餐共用池） |
| `DINNER` | 晚餐（计入午餐晚餐共用池） |

### 排期模型（已迁移到 meal_schedule_plan 表）

历史上排期保存在 `dish.schedule` JSON 字段（格式 `["W-D", ...]`）。**该字段已废弃**，所有排期查询和排餐生成统一以 `meal_schedule_plan` 表为准：

| 字段 | 含义 |
|------|------|
| `week_num` | 周序号 1-4 |
| `day_of_week` | 星期 1-7（周一=1） |
| `meal_time` | 餐次 LUNCH / DINNER |
| `dish_category` | 坑位类型 MAIN / SIDE / SOUP / VEGETABLE |
| `dish_id` | 关联的菜品ID |

排期 key（`W-D`）由 `ScheduleKeyUtil` 工具类统一计算，规则见排餐管理业务说明。

### 菜品与套餐的关系（两套体系，注意区分）

**第一套：餐品线代码（`dish.meal_packages`，`MealPackageEnum`）**

用于菜品维度的分类，是枚举常量，独立于数据库表：

| code | 含义 | 说明 |
|------|------|------|
| `yuezi` | 月子餐 | 产后调理 |
| `yunqi` | 孕期餐 | 孕期营养 |
| `xiaoyuezi` | 小月子 | 流产后调理 |
| `yingyang` | 营养餐 | 通用营养 |
| `fenmian` | 分娩餐 | 分娩前后 |

一个菜品可以属于多个餐品线（如 `"mealPackages": ["yuezi", "yingyang"]`）。

**第二套：父子套餐结构（`parent_package` / `sub_package`）**

用于客户订单维度，与菜品管理模块独立。详见套餐管理业务说明。

**两者的区别**：餐品线代码（枚举）是菜品属性，决定某道菜出现在哪个产品线；父子套餐是客户购买的商品规格，影响订单计费和餐数分配。两者没有直接关联，但都叫"套餐"，容易混淆。

### 配料三层分类（2026-04-24 新增）

配料从单层 `category` 字符串升级为「一级分类 → 二级分类 → 配料」三层结构：

```
肉类（一级）
  ├── 禽肉（二级）
  │     ├── 鸡肉
  │     └── 鸭肉
  ├── 畜肉（二级）
  │     ├── 猪肉
  │     └── 牛肉
  └── 内脏（二级）

蔬菜（一级）
  ├── 瓜类（二级）
  ├── 叶菜类（二级）
  └── ...
```

| 层级 | 来源 | 说明 |
|------|------|------|
| 一级分类 | `dish_ingredient_category` 表，`level=1`，`parent_id=NULL` | 7 大类：蔬菜/肉类/水产/豆制品/调料/主食/其他 |
| 二级分类 | `dish_ingredient_category` 表，`level=2`，`parent_id=一级ID` | 每个一级下有若干二级，未确定的归入"未分类" |
| 配料 | `dish_ingredient` 表，通过 `category_id` 关联到二级分类 | 旧字段 `category`（MEAT/VEGETABLE/...）保留兼容，迁移脚本会按映射写入新 `category_id` |

三层分类的核心用途是**排餐过敏过滤**——客户的过敏标签可以是配料名（"鸡肉"）、二级分类名（"禽肉"）、或一级分类名（"肉类"）任一层级，详见排餐管理业务说明的过敏过滤章节。

## 数据模型

### dish — 菜品主档

| 字段 | 类型 | 说明 |
|------|------|------|
| `name` | VARCHAR | 菜品名称 |
| `cooking_method` | TEXT | 做法流程 |
| `ingredients` | TEXT | 配料（自由文本，冗余备用） |
| `image_url` | VARCHAR | 图片路径 |
| `dish_type` | VARCHAR | 菜品类型：MAIN / SIDE / SOUP / VEGETABLE / RICE / RICE_TYPE |
| `meal_types` | JSON Array | 适用餐次：`["LUNCH"]` / `["DINNER"]` / `["LUNCH","DINNER"]` |
| `meal_packages` | JSON Array | 所属餐品线代码：`["yuezi"]` / `["yunqi","yingyang"]`；`RICE_TYPE` 米饭通常为空，作为全局通用菜品 |
| `schedule` | JSON Array | **已废弃字段**，新代码不再读写，排期统一查 `meal_schedule_plan` |
| `sort` | INT | 排序值，越小越靠前 |
| `enabled` | TINYINT | 是否启用 |

非数据库字段：
- `ingredientList`：配料详情列表（来自 `dish_ingredient_relation` JOIN）
- `mealPackageDetails`：套餐详情（来自 `parent_package` 表，用于表格展示，与 meal_packages JSON 无直接关系）

### dish_ingredient — 配料字典

| 字段 | 类型 | 说明 |
|------|------|------|
| `name` | VARCHAR | 配料名称，唯一 |
| `category_id` | INT | 二级分类ID，关联 `dish_ingredient_category.id` |
| `category` | VARCHAR | **旧字段保留兼容**：MEAT / VEGETABLE / SEAFOOD / TOFU / SPICE / OTHER |
| `unit` | VARCHAR | 单位：克(g)、毫升(ml)、个 |
| `calories` | INT | 每单位热量（卡路里） |

非数据库字段（查询时 JOIN 拼接）：
- `categoryName`：二级分类名（如"瓜类"）
- `parentCategoryId` / `parentCategoryName`：一级分类信息（如"蔬菜"）
- `categoryPathName`：分类路径（如"蔬菜/瓜类"），用于表格展示

### dish_ingredient_category — 配料分类表（2026-04-24 新增）

| 字段 | 类型 | 说明 |
|------|------|------|
| `parent_id` | INT | 父分类ID，一级分类为 NULL |
| `name` | VARCHAR | 分类名称 |
| `level` | TINYINT | 1 = 一级分类，2 = 二级分类 |
| `sort` | INT | 排序值 |
| `enabled` | TINYINT | 是否启用 |

约束：`UNIQUE KEY uk_parent_name (parent_id, name)`——同一父分类下分类名不能重复（数据库层面强制唯一）。

删除规则：
- 一级分类下若仍有二级分类 → 拒绝删除；
- 二级分类下若仍有配料关联 → 拒绝删除。

### dish_ingredient_relation — 菜品配料关联

菜品与配料的多对多关系表，记录用量和备注。

| 字段 | 类型 | 说明 |
|------|------|------|
| `dish_id` | INT | 菜品ID |
| `ingredient_id` | INT | 配料ID |
| `ingredient_name` | VARCHAR | 配料名称（冗余存储） |
| `unit` | VARCHAR | 单位（冗余存储） |
| `quantity` | DOUBLE | 用量 |
| `remark` | TEXT | 备注，如"少盐"、"切丁" |

### meal_schedule_plan — 排期模板表

| 字段 | 类型 | 说明 |
|------|------|------|
| `week_num` | INT | 周序号 1-4 |
| `day_of_week` | INT | 星期 1-7 |
| `meal_time` | VARCHAR | 餐次 LUNCH / DINNER |
| `dish_category` | VARCHAR | 坑位类型 MAIN / SIDE / SOUP / VEGETABLE |
| `dish_id` | INT | 关联菜品ID（仅 `RICE_TYPE` 不进此表） |
| `enabled` | TINYINT | 是否启用 |

一条记录 = 一个「周-星期-餐次-坑位」的候选菜。同一坑位通常配置多条以支持随机选菜。

### 公共排期菜单与客户候选菜边界（2026-07-13）

- 公共排期菜单是只读展示口径：只读取指定日期、`LUNCH` / `DINNER` 的启用 `meal_schedule_plan` 记录及关联菜品，并按餐次分组返回。
- 客户候选菜是排餐辅助口径：在排期菜基础上，还会合并 `RICE_TYPE` 全局米饭池，并按套餐、忌口和三级过敏规则过滤。
- 因此公共菜单为空或未配置时，不能以全局米饭填充；全局米饭池也不能被视作公共菜单的一部分。
- 客户实际吃什么应查询已生成的 `meal_plan_customer_item`，不能用公共菜单或候选菜替代。

## 核心业务流程

### 1. 菜品维护

新建菜品 → 填写基本信息 → 关联配料（含分类路径） → 设置适用餐次/餐品线 → 保存。

排期不在菜品表单内维护，而是在「排餐矩阵页」按周-星期-餐次-坑位编辑 `meal_schedule_plan`。

编辑配料列表通过 `dish_ingredient_relation` 表管理；新增配料时若输入未存在的二级分类名，前端可让分类选择器自动创建。

### 2. 菜品列表查询（含排期日期过滤）

`GET /api/dishes` 支持以下条件：

| 参数 | 说明 |
|------|------|
| name | 名称模糊匹配 |
| dishType | 菜品类型 |
| mealType | 餐次（JSON_CONTAINS 匹配 `meal_types`） |
| mealPackage | 餐品线（JSON_CONTAINS 匹配 `meal_packages`） |
| `scheduleDate` | 排期日期 yyyy-MM-dd（2026-04-25 新增） |
| enabled | 启用状态 |

**`scheduleDate` 过滤实现**：在 Service 层将日期换算成 `(weekNum, dayOfWeek)`（复用 `ScheduleKeyUtil`），先查 `meal_schedule_plan` 拿到当日 `dish_id` 列表，再用 `id IN (...)` 拼到主查询。空集合时不附加 IN 条件，由前端字段控制是否过滤。

### 3. 菜品主档列表页（2026-04-25 新增）

矩阵页（`/meal/dish`）头部新增"主档列表"按钮，跳转到独立的菜品主档列表页 `/meal/dish-list`：
- 路由：`router/routers.js` 中作为隐藏路由注册（`hidden: true`，`noCache: true`）；
- 复用 `DishForm` 组件做新增/编辑，支持分页、状态切换、批量删除；
- 列表搜索栏含上节列出的全部条件，包含排期日期。

### 4. 客户可用菜品查询

`GET /api/dishes/available?customerId=X&mealType=LUNCH&week=2&day=3`

读取客户忌口列表，对菜品 `ingredients` 字段做关键词匹配，返回不过敏的菜品。传了排期参数时按 `dish.schedule` 旧逻辑筛（此接口仍依赖 JSON 字段，未迁移到 `meal_schedule_plan`）。

> 注意：排餐主流程已切到 `meal_schedule_plan`，但该"客户可用菜品"接口和首页统计接口仍读 `dish.schedule`。如要彻底废弃 `dish.schedule`，这两处需要同步改造。

### 5. 首页排餐统计

`GET /api/dishes/schedule/stats/{date}` 返回指定日期各套餐线的客户数、替换数、各菜品类型的菜品名称和替换统计，用于销售看板。同样仍基于 `dish.schedule` JSON 字段读取，待统一迁移。

## 与其他模块的关系

```
客户档案（customer_profile）
  ├── 客户饮食限制（customer_dietary_restrictions）
  │     → 配菜时过滤/替换菜品
  ├── 客户订单（customer_order）
  │     → 关联套餐、riceType，决定客户应享受的餐品线和米饭档案
  └── 客户关键词（customer_keywords）
        → 配菜参考偏好

配菜管理
  ├── dish               菜品主档（schedule 字段已废弃）
  ├── dish_ingredient    配料字典（含 category_id）
  ├── dish_ingredient_category  配料一级/二级分类
  ├── dish_ingredient_relation  菜品配料多对多
  └── meal_schedule_plan 排期模板（周-星期-餐次-坑位 → dish_id）
        ↓
排餐计划（meal_plan / meal_plan_customer / meal_plan_customer_item）
  └── 核销管理（meal_verification_log）
        → 客户实际核销的餐次，不影响菜品数据
```

**注意**：核销管理（MealVerificationLog）和排餐计划是两个独立流程：
- 排餐计划：系统预先编排的菜品
- 核销管理：客户实际取餐的记录

两者通过 `customer_id` + `日期` 关联，但删除排餐记录不会影响核销数据。
