# 客户话术建档解析草稿规则

## 目标

客服粘贴客户沟通文本后，系统解析出“客户档案 + 首单”的草稿。草稿必须由客服确认后再保存，最终保存仍调用现有新增客户接口，沿用客户编号、套餐、订单和排餐校验。

## 基本原则

- 解析服务只生成草稿，不直接写入数据库。
- 明确字段优先，模糊描述只进入提示或备注。
- 系统已有枚举和字典必须使用规范值。
- 不能把“不吃甜食”“水果不超过200g”等泛化忌口硬解析为具体菜品。
- 无法唯一匹配的数据必须提示客服确认。

## 推荐输入模板

模板尽量沿用客服现有话术结构，只要求少数字段改成系统可识别的规范写法。尤其是 `套餐`、每餐菜品配置、过敏食物/排除菜品要按规则填写。

```text
联系人：陈女士
电话：17760193876
地址：成都市天府新区天府合印5栋1单元
客户编号：Y1001
套餐：营养餐
来源：小红书
午餐开始配送时间：11点整
晚餐开始配送时间：17点整
【目前情况】
目前身体感受：有孕反，容易胀气
血糖血压情况：正常
缺铁、缺钙：缺钙
是否有贫血：否
尊医嘱：
【饮食禁忌】
🈲过敏食物：牛奶,海鲜
⚠️不能吃的: 米换成糙米，不吃面食、甜食、水果（每天不超过200g）反季节水果不吃，生肉不吃，冰箱里的不碰，不吃肥肉，不吃皮
餐别：孕期营养餐不含汤（两荤一素）
菜品配置：2主1副1素0汤
合计：30餐不含汤
配送日期：默认等通知配送
特殊要求：不能要任何冻货
排除菜品：红烧肉,回锅肉
过敏等：
```

`客户编号` 可不传。不传时，由系统根据首单父套餐编号池自动生成。

## 字段映射

| 话术字段 | 系统字段 | 规则 |
|---|---|---|
| 客户编号 | `customerCode` | 可选。传入后按父套餐编号池校验前缀、数字部分、范围、占用；数字部分不限制固定位数。 |
| 联系人 / 客户 / 姓名 | `customerName` | 必填。同步作为默认地址联系人。 |
| 电话 / 手机号 | `phone` | 必填。同步作为默认地址电话。 |
| 地址 / 配送地址 | `addresses[0].addressDetail` | 必填。地址类型固定为 `DEFAULT`。 |
| 套餐 / 父套餐 | `orderInfo.parentPackageId` | 必填。优先按系统父套餐名称或编码匹配，例如“营养餐”。若未填写或未匹配成功，可从 `餐别` 长文本中兜底匹配已启用父套餐名称或编码。 |
| 来源 / 客户来源 / 销售渠道 | `orderInfo.customerSource` | 可选。应匹配系统 `customer_source` 字典值。括号/方括号内的候选提示会被忽略，按括号外真实选择匹配；括号外为空时默认 `其他`。 |
| 餐数 / 合计 | `orderInfo.lunchDinnerCount` | 每日一餐默认计入午晚餐池。 |
| 早餐数 | `orderInfo.breakfastCount` | 可选，未传为 0。 |
| 餐次 | `orderInfo.mealType` / `orderInfo.startMealType` | 午餐=`LUNCH`，晚餐=`DINNER`，午餐+晚餐=`LUNCH_DINNER`，包含早餐时使用 `ALL`。 |
| 菜品配置 | `orderInfo.mainDishCount` / `sideDishCount` / `vegCount` / `soupCount` | 推荐一行填写，如 `2主1副1素0汤` 或 `2 主 1 副 1 素 0 汤`。 |
| 汤数 / 是否含汤 | `orderInfo.soupCount` | 菜品配置未写汤数时，未传、不含汤、无汤、0汤均为 0。 |
| 米饭类型 | `orderInfo.riceType` | 可选，未传默认白米饭。 |
| 排餐模式 | `orderInfo.scheduleMode` | 使用现有值：`SCHEDULE`、`DAILY`、`WEEKDAY`、`WEEKEND`。 |
| 配送日期 | `orderInfo.deliveryDates` / `orderInfo.startDate` / `orderInfo.scheduleMode` / `orderInfo.mealType` / `orderInfo.startMealType` | “默认等通知配送”表示指定日期送且日期为空。也支持在同一行混写开始日期、开始餐次和配送频率，例如 `6 月 1 日、晚餐开始、默认每天配送晚餐`。 |
| 开始日期 / 订单开始日期 / 开始配送日期 | `orderInfo.startDate` | 可选，未传默认新增订单当天。 |
| 过敏食物 | `allergyTags` | 逗号分隔，存配料名或分类名。整段为 `无`、`没有`、`无过敏` 等空值表达时回填空数组。 |
| 排除菜品 | `excludedDishIds` | 逗号分隔，必须匹配系统菜品名。 |
| 不能吃的 / 特殊要求 | `specialRequirements` | “不能吃的”整段记录到特殊要求，不拆成过敏标签或排除菜品。整段为 `无`、`没有` 等空值表达时忽略。 |
| 目前身体感受 / 血糖血压情况 / 缺铁缺钙 / 是否贫血 / 尊医嘱 | `medicalRequirements` | 合并为医嘱和身体情况说明。 |

## 默认值规则

| 字段 | 默认值 |
|---|---|
| `customerCode` | 不传，由后端按父套餐编号池生成。 |
| `orderInfo.startDate` | 新增订单当天。 |
| `orderInfo.scheduleMode` | `SCHEDULE`。 |
| `orderInfo.deliveryDates` | 空。 |
| `orderInfo.breakfastCount` | 0。 |
| `orderInfo.lunchDinnerCount` | 从“餐数/合计”提取；无法提取时提示。 |
| `orderInfo.startMealType` | 午餐或午晚餐默认 `LUNCH`，晚餐默认 `DINNER`。 |
| `orderInfo.mealType` | 默认 `LUNCH_DINNER`。 |
| `orderInfo.riceCount` | 1。 |
| `orderInfo.riceType` | `白米饭`。 |
| `orderInfo.soupCount` | 0。 |
| 金额字段 | 0，客服可后续补录。 |

## 排餐模式规范

| 客服输入 | 系统值 |
|---|---|
| 指定日期送 | `SCHEDULE` |
| 默认等通知配送 | `SCHEDULE`，`deliveryDates = null` |
| 每天送 / 每天配送 / 每日送 / 每日配送 | `DAILY` |
| 工作日送 | `WEEKDAY` |
| 周末送 | `WEEKEND` |

`配送日期` 字段兼容运营混合话术：

| 配送日期示例 | 解析结果 |
|---|---|
| `6 月 1 日、晚餐开始、默认每天配送晚餐` | `startDate = 2026-06-01`，`scheduleMode = DAILY`，`mealType = DINNER`，`startMealType = DINNER` |
| `2026-06-01、午餐开始、每日配送午餐` | `startDate = 2026-06-01`，`scheduleMode = DAILY`，`mealType = LUNCH`，`startMealType = LUNCH` |

## 订餐描述规范

`订餐描述` 推荐按“天数 + 配送频率 + 餐次”填写，例如 `14天每天早中晚`、`11天工作日午餐`。解析时先识别天数和排餐模式，再把餐次别名归一化为早餐、午餐、晚餐。

| 订餐描述示例 | `breakfastCount` | `lunchDinnerCount` | `mealType` | `startMealType` |
|---|---:|---:|---|---|
| `14天每天早中晚` / `14天每天早午晚` | 14 | 28 | `ALL` | `BREAKFAST` |
| `14天每天早餐午餐晚餐` | 14 | 28 | `ALL` | `BREAKFAST` |
| `14天每天早中` / `14天每天早午` | 14 | 14 | `ALL` | `BREAKFAST` |
| `14天每天早晚` | 14 | 14 | `ALL` | `BREAKFAST` |
| `14天每天早餐` / `14天每天早饭` | 14 | 0 | `ALL` | `BREAKFAST` |
| `14天每天中晚` / `14天每天午晚` | 0 | 28 | `LUNCH_DINNER` | `LUNCH` |
| `14天每天中餐` / `14天每天午餐` | 0 | 14 | `LUNCH` | `LUNCH` |
| `14天每天晚餐` / `14天每天晚饭` | 0 | 14 | `DINNER` | `DINNER` |

餐次别名规则：

| 别名 | 餐次 |
|---|---|
| `早` / `早餐` / `早饭` | 早餐 |
| `中` / `午` / `中餐` / `午餐` / `中饭` / `午饭` | 午餐 |
| `晚` / `晚餐` / `晚饭` | 晚餐 |

## 套餐和菜品配置规范

`套餐` 优先使用系统父套餐名称或编码，例如：

```text
套餐：营养餐
```

客服原始话术里的 `餐别` 可以保留。解析时会从 `餐别` 长文本中兜底查找已启用父套餐名称或编码，例如：

```text
餐别：小月子餐、两荤一素一汤
```

如果系统存在父套餐 `小月子餐`，则可回填 `orderInfo.parentPackageId`；`两荤一素一汤` 仍由 `菜品配置` 字段决定。

若 `餐别` 只包含模糊业务描述而没有系统父套餐名称或编码，则不会作为父套餐。比如：

```text
餐别：孕期营养餐不含汤（两荤一素）
```

原因是“孕期”“不含汤”“两荤一素”分别可能属于业务描述、汤数、菜品数量配置，不一定是系统父套餐。

在原模板基础上补充这些规范字段即可：

```text
套餐：营养餐
菜品配置：2主1副1素0汤
合计：30餐不含汤
```

## 过敏食物和排除菜品

`过敏食物` 用于客户过敏标签，按逗号分隔：

```text
过敏食物：牛奶,海鲜,花生
```

以下空值表达不会写入过敏标签：

```text
过敏食物：无
过敏食物：没有
过敏食物：无过敏
```

`排除菜品` 用于具体菜品过滤，按逗号分隔：

```text
排除菜品：红烧肉,回锅肉
```

`不能吃的` 整段记录到特殊要求，不解析为菜品 ID，也不解析为过敏标签：

```text
不能吃的：不吃面食、甜食、水果每天不超过200g、反季节水果不吃、生肉不吃
```

如果整段为 `无`、`没有`、`暂无` 等空值表达，则不写入 `specialRequirements`。

## 阻塞保存的问题

出现以下问题时，解析结果 `valid = false`，前端允许应用草稿但必须提示客服修正后保存：

- 联系人为空。
- 电话为空或手机号格式不正确。
- 地址为空。
- 套餐为空、不是系统父套餐、或混入“不含汤/两荤一素”等规格描述。
- 餐数为空或小于等于 0。
- 菜品配置无法从文本解析且表单仍为 0 时，需要客服确认。
- 指定客户编号不符合父套餐编号池规则，例如前缀不匹配、前缀后不是纯数字、数字值超出编号池范围或编号已占用。

## 非阻塞提醒

以下问题只提示，不阻塞保存：

- 来源不在字典中。
- 米饭类型未匹配到系统米饭档案。
- 排除菜品名称未匹配到菜品主档。
- 配送时间只有“11点整/17点整”，当前系统没有结构化字段，只能放入特殊要求。
- “不能吃的”已整体记录到特殊要求，客服仍需确认是否另有明确过敏食物或排除菜品。
