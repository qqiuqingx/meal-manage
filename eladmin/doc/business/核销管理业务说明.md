# 核销管理

## 业务定位

核销管理是送餐交付的核心环节，记录客户每次取餐的事实，作为餐费结算和订单状态推进的触发器。一次核销对应一次送餐成功，同时扣减订单剩余餐数和餐费余额。当餐数耗尽时，核销操作会**自动将订单状态更新为已完成**。

---

## 核心概念

### 核销粒度

核销的操作对象是**客户排餐记录**（`meal_plan_customer`），而非菜品明细（`meal_plan_customer_item`）。核销一次，扣一餐（`verification_count=1`）。

| 餐次 | 扣减字段 | 单价来源 |
|------|---------|---------|
| 午餐 `LUNCH` | `remaining_count`（合计） | `customer_order.lunch_dinner_price` |
| 晚餐 `DINNER` | `remaining_count`（合计） | `customer_order.lunch_dinner_price` |
| 早餐 `BREAKFAST` | 参与排餐计划（生成 `meal_plan_customer` 记录，但不生成菜品明细），通过排餐记录核销 | `customer_order.breakfast_price` |

### 核销日志（meal_verification_log）

核销是不可逆操作，每一笔核销都会生成一条日志记录，包含核销前后的状态快照，用于对账和追溯：

```
核销前剩余餐数 → remaining_before
核销后剩余餐数 → remaining_after
核销前已核销总数 → verified_total_before
核销后已核销总数 → verified_total_after
```

### 自动完成订单

核销后若 `remaining_count` 减至 0，系统在核销事务内自动将订单状态更新为 `status=2`（已完成）。此逻辑保证餐数耗尽和订单完结的一致性。

---

## 数据模型

### meal_verification_log — 核销日志表

| 字段 | 类型 | 说明 |
|------|------|------|
| `meal_plan_customer_id` | BIGINT | 客户排餐记录ID |
| `meal_plan_id` | BIGINT | 排餐计划ID |
| `customer_id` | BIGINT | 客户ID |
| `order_id` | BIGINT | 关联订单ID |
| `record_date` | DATE | 排餐日期 |
| `meal_type` | VARCHAR | 餐次：LUNCH / DINNER |
| `verification_count` | INT | 核销餐数（默认 1） |
| `remaining_before` | INT | 核销前剩余餐数 |
| `remaining_after` | INT | 核销后剩余餐数 |
| `verified_total_before` | INT | 核销前已核销总数 |
| `verified_total_after` | INT | 核销后已核销总数 |
| `operator` | VARCHAR | 操作人 |
| `operate_time` | DATETIME | 操作时间 |
| `remark` | VARCHAR | 备注 |

### meal_plan_customer — 核销状态字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `is_verified` | TINYINT | 是否已核销：0=未核销，1=已核销 |
| `verification_time` | DATETIME | 核销时间 |
| `verification_operator` | VARCHAR | 核销操作人 |
| `is_refunded` | TINYINT | 是否已退餐：0=否，1=是 |
| `refund_time` | DATETIME | 退餐时间 |

## 核心业务流程

### 1. 执行核销

`POST /api/meal-verification/verify`，传入 `customerPlanIds`（客户排餐记录 ID 列表）。

**完整校验链路（顺序执行，失败立即返回）：**

```
① 客户排餐记录是否存在？
  └─ 不存在 → "客户排餐记录不存在"
② 是否已核销（is_verified=1）？
  └─ 已核销 → "该客户已完成核销，请勿重复操作"
③ 关联订单是否存在？
  └─ 不存在 → "关联订单不存在"
④ 订单剩余餐数是否 > 0？
  └─ 不足 → "订单剩余餐数不足，当前剩余: X"
⑤ 订单单价是否有效（lunchDinnerPrice / breakfastPrice > 0）？
  └─ 无效 → "订单单价配置异常，无法核销"
```

**通过全部校验后，原子性执行以下操作：**

```
⑥ 计算核销单价：
   - meal_type=LUNCH/DINNER → lunch_dinner_price
   - meal_type=BREAKFAST    → breakfast_price

⑦ 更新 customer_order：
   verified_count   += 1
   verified_amount  += 单价
   meal_balance     -= 单价
   remaining_count  -= 1

⑧ 若 remaining_count == 0：
   status → 2（已完成），自动完结订单

⑨ 更新 meal_plan_customer：
   is_verified            = 1
   verification_time      = 当前时间
   verification_operator  = 当前操作人

⑩ 写入 meal_verification_log（记录状态快照）
```

### 2. 批量核销

支持一次传入多个 `customerPlanIds`，逐个执行上述链路。部分失败时返回成功/失败明细：

```json
{
  "successCount": 2,
  "failCount": 1,
  "failReasons": [
    "客户ID 3: 订单剩余餐数不足，当前剩余: 0"
  ]
}
```

### 3. 核销回退（删除核销日志）

`DELETE /api/meal-verification/logs/{id}`，软删除核销日志并回退所有关联数据：

```
① 软删除核销日志（deleted=1，记录删除时间和操作人）
② 回退排餐记录：is_verified → 0，清空 verification_time 和 verification_operator
③ 回退订单：
   - verified_count  -= 1
   - verified_amount -= 单价
   - meal_balance    += 单价
   - remaining_count += 1
   - 若订单已是"已完成"（status=2）→ 自动回退为"进行中"（status=1）
```

### 4. 核销日志查询

| 查询维度 | 接口 |
|---------|------|
| 按核销日志 ID 查详情 | `GET /api/meal-verification/logs/{id}` |
| 按订单 ID 查所有核销记录 | `GET /api/meal-verification/logs/order/{orderId}` |
| 分页综合查询 | `GET /api/meal-verification/logs` |

分页查询支持筛选：客户姓名、订单 ID、餐次、日期范围、操作时间范围、操作人。

## 业务规则汇总

| 规则 | 说明 |
|------|------|
| 禁止重复核销 | `is_verified=1` 时拒绝再次核销 |
| 核销即扣餐数 | 每次核销 `remaining_count -= 1` |
| 核销即扣余额 | 每次核销 `meal_balance -= 单价` |
| 餐数耗尽自动完单 | `remaining_count == 0` 时事务内将订单 `status → 2` |
| 仅核销进行中订单 | `status=1` 才允许核销；`status=0/2` 的订单拒绝 |
| 早餐参与排餐但无菜品明细 | `meal_type=BREAKFAST` 在排餐计划中生成 `meal_plan_customer` 记录，但不生成 `meal_plan_customer_item` 菜品明细 |
| 核销不影响排餐记录 | 删除排餐记录不影响核销日志 |
| 核销日志可回退 | 删除核销日志时触发回退：软删除日志 + 回退排餐核销状态 + 回退订单核销数据；若订单已自动完成则回退为进行中 |
| 退餐标记核销日志 | 退餐时核销日志 `is_refunded → 1`，但日志保留 |

## 金额计算示例

假设某订单配置：
- 午餐晚餐数：`lunch_dinner_count = 20`
- 午餐晚餐单价：`lunch_dinner_price = 45.00 元`
- 成交价：`final_amount = 5000.00 元`

核销 5 次午餐后：

```
verified_count  = 5
verified_amount = 5 × 45 = 225.00 元
meal_balance    = 5000 - 225 = 4775.00 元
remaining_count = 20 - 5 = 15
```

第 20 次核销后：`remaining_count == 0`，订单**自动变为已完成**（`status=2`）。

## 与其他模块的关系

```
核销管理（meal_verification_log）
  │
  ├── 订单管理（customer_order）
  │     ├── 扣减 verified_count / verified_amount / meal_balance / remaining_count
  │     ├── 校验 remaining_count > 0
  │     └── 核销至 0 时自动将 status → 2
  │
  ├── 排餐管理（meal_plan_customer）
  │     ├── 操作对象：meal_plan_customer.id
  │     ├── 标记 is_verified / verification_time / verification_operator
  │     └── 核销日志通过 meal_plan_customer_id 关联
  │
  ├── 配菜管理（dish）
  │     └── 无关：核销不涉及菜品数据
  │
  └── 客户管理（customer_profile）
        └── 无关：核销不涉及客户档案变更
```

## 智能客服只读查询口径

- Agent 仅查询未删除的核销日志，可按客户、订单、日期范围和餐次限制；默认最近 10 条，最大 50 条，缺少客户和订单时拒绝无界历史查询。
- Agent 退餐记录返回退早餐数、退午晚餐数和已核销不退餐数，不返回退款金额。
- 回答餐数变化时必须明确：`BREAKFAST` 扣早餐池，`LUNCH` 与 `DINNER` 均扣午晚餐共享池。
- `DAILY_UNVERIFIED_CUSTOMER_COUNT` 表示指定业务日已排餐但尚未核销的客户数；`DAILY_VERIFIED_CUSTOMER_COUNT` 表示当日已核销客户数，两者不能仅因问题出现“排餐”而误选为已排餐指标。
- `VERIFICATION_COUNT` 统计有效核销次数，默认要求用户提供明确日期或日期范围；“本周”由服务端按 `Asia/Shanghai` 解析为本周一至当前业务日。
