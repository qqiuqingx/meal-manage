# 客户管理

## 业务定位

客户管理是系统的前台业务入口，管理客户档案、地址、签约套餐、孕周信息等。一个客户档案对应一个活客户，客户档案关联多笔订单（增购时新建订单）。客户管理是连接套餐管理、配菜管理、核销管理的枢纽模块。

---

## 核心概念

### 客户档案（customer_profile）

主档，记录客户基本信息。每个客户有唯一编号（`customer_code`），支持手动输入或系统自动分配。编号格式：`{poolPrefix}{序号}`，如 `A11001`、`A11002`。

档案字段分为两类：
- **固定字段**：姓名、手机号、孕周、过敏标签、医嘱要求、排除菜品、排除日期
- **计算字段**（非数据库存储，查询时实时计算）：剩余早餐数、剩余午餐晚餐数

### 套餐分类（customer_package_category）

与套餐管理模块（`parent_package`）是**两套独立的分类体系**，设计目的不同：

| 维度 | 模块 | 表 | 用途 |
|------|------|-----|------|
| 套餐分类 | 客户档案页 | `customer_package_category` | 客户档案页联动选择，父子两级树形结构 |
| 父子套餐 | 套餐管理 | `parent_package` / `sub_package` | 订单核心，决定餐数规格和编号池 |

`customer_package_category` 在客户档案页用于选择"月子餐 → 两荤一素"这样的分类路径，最终通过该分类关联到具体父套餐生成客户编号。`parent_package` 是底层商品配置，包含编号池、菜品线等运营数据。

### 地址管理

每个客户支持三个地址槽位：

| 地址类型 | code | 用途 |
|---------|------|------|
| 默认地址 | `DEFAULT` | 主要送餐地址 |
| 工作日地址 | `WORKDAY` | 工作日送餐地址 |
| 周末地址 | `WEEKEND` | 周末送餐地址 |

业务规则：三个槽位中**至少有一个非空**。地址随客户档案一并创建/编辑，整体覆盖。

### 过敏标签（allergy_tags）

JSON 数组格式，存储客户的过敏食物列表，如 `["牛奶", "海鲜"]`。在配菜管理中用于过滤/替换菜品。

### 排除菜品（excluded_dish_ids）

JSON 数组格式，存储客户明确要求不要在餐单中出现的菜品ID列表。在排餐时被标记为过敏过滤菜品。

### 排除日期（exclude_dates）

JSON 数组格式，支持按日期+餐次排除配送。格式：
```json
[
  {"date": "2026-04-15", "mealTypes": ["LUNCH", "DINNER"]},
  {"date": "2026-04-20", "mealTypes": ["DINNER"]}
]
```

## 数据模型

### customer_profile — 客户主档

| 字段 | 类型 | 说明 |
|------|------|------|
| `customer_code` | VARCHAR | 客户编号，唯一，由系统自动生成 |
| `customer_name` | VARCHAR | 客户姓名 |
| `phone` | VARCHAR | 手机号 |
| `gestational_week` | INT | 孕周（正整数） |
| `allergy_tags` | JSON Array | 过敏食物标签 |
| `excluded_dish_ids` | JSON Array | 排除菜品ID列表 |
| `exclude_dates` | JSON Array | 排除日期列表（带餐次） |
| `medical_requirements` | TEXT | 医嘱要求（如"少盐少油"） |
| `special_requirements` | TEXT | 特殊要求 |
| `production_date` | DATE | 生产日期 |
| `remark` | TEXT | 备注 |

### customer_profile_address — 客户地址

| 字段 | 类型 | 说明 |
|------|------|------|
| `customer_id` | BIGINT | 客户档案ID |
| `address_type` | VARCHAR | 地址类型：DEFAULT / WORKDAY / WEEKEND |
| `address_detail` | VARCHAR | 详细地址 |
| `contact_name` | VARCHAR | 联系人姓名 |
| `contact_phone` | VARCHAR | 联系人电话 |

### customer_profile_package — 签约记录

客户档案与父套餐的签约关联表（区别于 `customer_order` 的订单记录）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `customer_id` | BIGINT | 客户档案ID |
| `parent_package_id` | BIGINT | 父套餐ID |
| `start_date` | DATE | 签约开始日期 |
| `end_date` | DATE | 签约结束日期 |

### customer_package_category — 套餐分类

| 字段 | 类型 | 说明 |
|------|------|------|
| `category_name` | VARCHAR | 分类名称 |
| `category_code` | VARCHAR | 分类编码，全局唯一 |
| `parent_id` | BIGINT | 父级分类ID（顶级为 NULL） |
| `level` | INT | 层级：1=父级，2=子级 |
| `sort` | INT | 排序值 |
| `enabled` | BOOLEAN | 是否启用 |
| `code_prefix` | VARCHAR | 编号前缀（仅父级使用，单个大写字母） |

## 核心业务流程

### 1. 创建客户（销售录入）

```
POST /api/customerProfile
{
  "customerName": "张三",
  "phone": "138xxxx",
  "gestationalWeek": 32,
  "allergyTags": ["牛奶"],
  "excludedDishIds": [1, 5, 12],
  "excludeDates": [{"date": "2026-04-15", "mealTypes": ["LUNCH"]}],
  "medicalRequirements": "少盐少油",
  "specialRequirements": "米饭加量",
  "addresses": [{ "addressType": "DEFAULT", "addressDetail": "北京市..." }],
  "orderInfo": {
    "parentPackageId": 1,
    "childPackageId": 3,
    "breakfastCount": 10,
    "lunchDinnerCount": 20,
    "startDate": "2026-04-01",
    "startMealType": "DINNER",
    "endDate": "2026-06-30",
    "mainDishCount": 2,
    "sideDishCount": 1,
    "vegCount": 1,
    "riceCount": 1,
    "soupCount": 1
  }
}
```

**后端自动执行以下步骤：**

1. **参数校验**
   - 客户姓名、手机号（非空、格式校验）、地址（非空、类型校验）
   - 首单信息校验（父套餐存在且启用、餐数非负；`startMealType` 与 `mealType` 兼容；主菜/副菜/素菜/汤数量不能为负数；米饭数量不能为负数且不能超过10）
   - 排除日期格式校验

2. **生成客户编号**
   - 优先使用前端传入的手动编号（需校验格式、前缀、范围、唯一性）
   - 无手动编号时调用编号池服务自动分配（`poolPrefix + 序号`）
   - 编号池支持并发，使用 `SELECT FOR UPDATE` 保证唯一性

3. **创建客户档案**
   - 写入 `customer_profile` 表

4. **创建地址记录**
   - 批量写入 `customer_profile_address` 表

5. **创建首笔订单**
   - 自动生成订单编号 `ORD{yyyyMMdd}{NNN}`
   - 状态默认为"进行中"（status=1）
   - 写入 `customer_order` 表

**编号生成机制**：

| 模式 | 说明 |
|------|------|
| 手动编号 | 前端传入，需校验：① 以 `poolPrefix` 开头 ② 数字部分在 `[poolStart, poolEnd]` 范围内 ③ 唯一 |
| 自动编号 | 调用编号池服务，根据 `poolPrefix`、`poolStart`、`poolEnd` 分配下一个可用号 |

### 2. 编辑客户档案

仅修改客户基本资料 + 地址。**不操作套餐/订单**，不走套餐变更流程。

### 3. 批量删除客户

仅允许删除**无进行中订单**的客户：
- 检查 `customer_order` 表 `status=1` 的记录
- 有进行中订单 → 拒绝删除，返回被拦截的客户姓名
- `status=0`（已取消）或 `status=2`（已完成）不影响删除

级联删除：`customer_profile_address`（地址）、`customer_profile_package`（签约记录）。**不删除** `customer_order`（订单独立管理）。

### 4. 剩余餐数计算

不存储在数据库，查询时实时计算，**汇总该客户所有有效订单**（`status=1` 且 `remaining_count > 0`）：

```
总早餐数 = Σ 所有有效订单的 breakfast_count
总午晚餐数 = Σ 所有有效订单的 lunch_dinner_count
总已核销早餐 = Σ 各订单已核销的 BREAKFAST 数
总已核销午晚餐 = Σ 各订单已核销的 LUNCH 数 + DINNER 数

剩余早餐数 = max(0, 总早餐数 - 总已核销早餐)
剩余午晚数 = max(0, 总午晚餐数 - 总已核销午晚餐)
```

数据来源：
- 订单表：`customer_order` 的 `breakfast_count`、`lunch_dinner_count`
- 核销记录：`meal_verification_log` 按 `order_id` + `meal_type` 分组统计（排除已退餐的记录）

### 5. 客户用餐统计页展示口径

客户用餐统计页参考运营 Excel 的第一页展示方式，按“客户 + 餐池明细行”返回数据，而不是简单按客户一行展示。

**按月查询口径：**

- 页面按月份查询，参数格式为 `yyyy-MM`
- 例如选择 `2026-04` 时，统计开始送餐时间早于 `2026-05-01` 的订单客户，即 `start_date < 2026-05-01`
- 其他月份同理，例如选择 `2026-05` 时，统计 `start_date < 2026-06-01` 的订单客户
- 订单必须满足 `status = 1` 且 `remaining_count > 0`
- 不按订单结束时间过滤

**拆行规则：**

- 只有早餐：展示 1 行，`送餐情况 = 早餐`
- 只有午晚餐：展示 1 行，`送餐情况 = 排餐模式 + 午/晚餐类型`
- 同时有早餐和午晚餐：展示 2 行，早餐行在前，午晚餐行在后

**数量展示说明：**

- 客户用餐统计页的表格行数和分页总数按“餐池明细行”统计，不等同于订单管理页的订单条数
- 同一笔订单如果同时包含早餐和午晚餐，会在本页面拆成 2 行，因此本页面总数可能大于订单管理页符合条件的订单数
- 例如 12 笔符合条件订单中有 5 笔同时包含早餐和午晚餐，则本页面展示为 `12 + 5 = 17` 条餐池明细行

**典型示例：**

- 若客户同时有 `breakfast_count > 0` 和 `lunch_dinner_count > 0`
- 页面会拆成：
  - 早餐行：展示早餐餐数、早餐剩余数
  - 午晚餐行：展示午晚餐数、午晚餐剩余数

**字段口径：**

- 编号：`customer_profile.customer_code`
- 电话：`customer_profile.phone`
- 地址：优先取 `DEFAULT` 地址，没有则依次取 `WORKDAY`、`WEEKEND`
- 备注信息：`customer_profile.remark`
- 特殊要求：`customer_profile.special_requirements` + `medical_requirements`
- 汤品：午晚餐订单任一订单 `soup_count >= 1` 时显示 `含汤`
- 购买时间：该餐池对应进行中订单的最早成交时间
- 开始时间：该餐池对应进行中订单的最早开始日期

**剩余餐数口径：**

- 早餐行：`Σ breakfast_count - Σ BREAKFAST 已核销数`
- 午晚餐行：`Σ lunch_dinner_count - Σ LUNCH 已核销数 - Σ DINNER 已核销数`

## 与其他模块的关系

```
客户管理（customer_profile）
  │
  ├── 套餐管理（parent_package / sub_package）
  │     ├── 通过订单关联，查到父套餐的 code_prefix → 生成客户编号
  │     └── 通过 customer_profile_package 查到签约套餐
  │
  ├── 配菜管理（dish / dish_ingredient）
  │     ├── allergy_tags → 配菜时过滤/替换菜品
  │     └── excluded_dish_ids → 标记为排除菜品
  │
  ├── 核销管理（meal_verification_log）
  │     └── 影响剩余餐数计算
  │
  └── 排餐管理（meal_plan / meal_plan_customer）
        ├── 客户档案 + 订单 → 确定生效客户 → 生成排餐计划
        └── exclude_dates → 过滤指定日期+餐次的排餐
```
