# 套餐管理

## 业务定位

套餐管理是系统的商品与订单核心模块，管理父套餐（产品线）和子套餐（规格）的两级结构，并为每个父套餐配置专属的编号池（客户编号前缀）。套餐决定了客户购买的餐数规格和价格体系。

---

## 核心概念

### 父子套餐两级结构

```
父套餐（parent_package）
  ├── 子套餐（sub_package）× N
  │     ├── 荤菜数量
  │     ├── 素菜数量
  │     ├── 是否含汤
  │     └── 是否含米饭
  └── 编号池配置
        ├── 前缀（pool_prefix）
        ├── 起始号（pool_start）
        └── 结束号（pool_end）
```

**父套餐**：代表一个产品线，如"月子餐"、"营养餐"。包含套餐代码、名称、状态，以及**编号池配置**（用于给客户分配唯一编号）。

**子套餐**：代表父套餐下的具体规格，如"7天月子餐"、"28天月子餐"。描述每顿的菜品数量配置（荤/素/汤/米饭），供下单时选择。

### 套餐与餐品线的关系（易混淆点）

这是系统中两个最容易混淆的概念：

| 维度 | 体系 | 说明 |
|------|------|------|
| 餐品线 | `MealPackageEnum` 枚举（代码常量） | 菜品属性，决定哪些菜品出现，**与数据库无关** |
| 父子套餐 | `parent_package` / `sub_package` 表 | 订单商品规格，决定客户买什么规格的餐 |

举例：
- 客户购买了"月子餐 → 7天月子餐"（父+子套餐，关联 `parent_package` 表）
- 配菜系统根据客户的餐品线（`yuezi`）从菜品库中选取 `meal_packages` 包含 `yuezi` 的菜品

两者通过**客户档案**间接关联：通过订单中的父套餐查到客户档案中的餐品线代码（`customer_profile.meal_package` → `MealPackageEnum.code`），再影响排餐时的菜品筛选。

### 编号池（Number Pool）

每个父套餐可独立配置编号池：

| 字段 | 说明 | 示例 |
|------|------|------|
| `prefix` | 单字母大写前缀（单个字母） | `A` |
| `pool_prefix` | 编号池前缀（可多个字符） | `A1` |
| `pool_start` | 起始号 | `1001` |
| `pool_end` | 结束号 | `1199` |

客户编号格式：`pool_prefix` + 序号 → 如 `A11001`、`A11002`。

编号池的作用：
- 避免不同套餐的客户编号冲突（如 A 套餐用 A1xxx，B 套餐用 B1xxx）
- 支持并发拿号（基于数据库唯一约束 + 重试）
- 用完自动扩号或报错

详见 `number_pool_migration.sql` 及 `numberpool` 模块。

## 数据模型

### parent_package — 父套餐

| 字段 | 类型 | 说明 |
|------|------|------|
| `package_code` | VARCHAR | 套餐代码（业务编码） |
| `prefix` | VARCHAR | 单字母前缀 |
| `pool_prefix` | VARCHAR | 编号池前缀 |
| `pool_start` | INT | 编号池起始号 |
| `pool_end` | INT | 编号池结束号 |
| `package_name` | VARCHAR | 套餐名称 |
| `status` | BOOLEAN | 状态：true=启用，false=停用 |
| `remark` | VARCHAR | 备注 |

### sub_package — 子套餐

| 字段 | 类型 | 说明 |
|------|------|------|
| `sub_package_code` | VARCHAR | 子套餐代码 |
| `sub_package_name` | VARCHAR | 子套餐名称 |
| `meat_count` | INT | 荤菜数量（一顿几道荤菜） |
| `veg_count` | INT | 素菜数量（一顿几道素菜） |
| `include_soup` | BOOLEAN | 是否含汤 |
| `include_rice` | BOOLEAN | 是否含米饭 |
| `status` | BOOLEAN | 状态 |
| `remark` | VARCHAR | 备注 |

### parent_package_sub — 父子关联

| 字段 | 类型 | 说明 |
|------|------|------|
| `parent_package_id` | BIGINT | 父套餐ID |
| `sub_package_id` | BIGINT | 子套餐ID |

## API 结构

控制器：`/api/package`

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/package` | GET | 分页查询父套餐 |
| `/api/package/tree` | GET | 获取父子套餐树（父子套餐联动展示） |
| `/api/package/parent/{id}` | GET | 查询单个父套餐详情 |
| `/api/package` | POST | 新增父套餐（含关联子套餐） |
| `/api/package` | PUT | 编辑父套餐（含关联子套餐） |
| `/api/package/status/{id}` | PUT | 修改父套餐状态 |
| `/api/package/{id}` | DELETE | 删除父套餐 |
| `/api/package/sub/{id}` | GET | 查询单个子套餐 |
| `/api/package/sub` | POST | 新增子套餐（独立于父套餐新建流程） |
| `/api/package/sub` | PUT | 编辑子套餐 |
| `/api/package/sub/status/{id}` | PUT | 修改子套餐状态 |
| `/api/package/sub/{id}` | DELETE | 删除子套餐 |

父子套餐树返回结构示例：

```json
{
  "id": 1,
  "packageCode": "YZ",
  "packageName": "月子餐",
  "status": 1,
  "prefix": "A",
  "poolPrefix": "A1",
  "poolStart": 1001,
  "poolEnd": 1199,
  "children": [
    {
      "id": 1,
      "subPackageCode": "YZ7",
      "subPackageName": "7天月子餐",
      "meatCount": 1,
      "vegCount": 1,
      "includeSoup": true,
      "includeRice": true,
      "status": 1
    }
  ]
}
```

## 核心业务流程

### 1. 套餐配置（管理员）

新建父套餐 → 配置编号池 → 批量创建子套餐规格 → 保存。

子套餐可以在创建父套餐时一起创建，也可以在套餐列表中独立新增/编辑。

### 2. 客户下单（销售）

客户选父套餐 → 选子套餐规格 → 填写信息 → 系统从编号池拿一个客户编号 → 创建 `customer_profile` + `customer_order` 记录。

编号池拿号逻辑：并发安全，基于唯一约束冲突检测，重试获取下一个可用号。

删除校验规则：
- 删除父套餐时，仅按父套餐ID检查 `customer_order.parent_package_id` 是否存在订单引用；存在引用时禁止删除。
- 删除子套餐时，仅按子套餐ID检查 `customer_order.child_package_id` 是否存在订单引用；存在引用时禁止删除。
- 父套餐ID与子套餐ID属于不同字段，删除时不能交叉使用，也不能用 OR 混合判断。

### 3. 配餐执行（运营）

根据订单中的父套餐 → 查客户档案的餐品线代码 → 按餐品线筛选菜品进行排餐 → 记录排餐计划。

## 与其他模块的关系

```
套餐管理（parent_package / sub_package）
  │
  ├── 客户订单（customer_order）
  │     ├── 关联 parent_package_id → 确定客户购买的产品线
  │     └── 关联 sub_package_id → 确定客户的餐数规格
  │
  ├── 客户档案（customer_profile）
  │     ├── 通过订单关联，查到客户的餐品线代码（MealPackageEnum）
  │     └── 通过订单查到编号池，分配客户编号
  │
  └── 配菜管理（dish）
        └── 客户的餐品线代码（yuezi/yunqi/...）→ 筛选 dish.meal_packages 匹配的菜品
```
