# 排餐管理

## 业务定位

排餐管理根据每日生效订单，生成客户的菜品配送计划。是连接订单（客户买了什么）和配菜（实际送什么）的执行层。每天由运营人员手动触发，或可配置定时任务自动生成。

---

## 核心概念

### 三层表结构

```
meal_plan（排餐计划主表）
  └── meal_plan_customer（排餐客户表）
        └── meal_plan_customer_item（客户菜品明细表）
```

| 表 | 作用 | 核心字段 |
|----|------|----------|
| `meal_plan` | 一次排餐事件 | record_date, meal_type, total_count, success_count, fail_count, status |
| `meal_plan_customer` | 一个客户当次餐的排餐结果 | customer_id, order_id, status, fail_reason, is_verified, verification_time, supplementary_*_count |
| `meal_plan_customer_item` | 该客户当次餐的菜品明细 | dish_type, dish_id, dish_name, is_replaced, original_dish_id, replace_reason |

### scheduleKey（排期匹配键）

由 `ScheduleKeyUtil` 工具类统一计算：

| 日期范围 | 周序号 | 计算规则 |
|---------|--------|---------|
| 1-7 日 | 第 1 周 | |
| 8-14 日 | 第 2 周 | |
| 15-21 日 | 第 3 周 | |
| 22-28 日 | 第 4 周 | |
| 29-31 日 | 映射为第 1 周 | 29日→第1周, 30日→第1周, 31日→第1周 |

格式：`W-D`（W=周序号 1-4，D=星期 1-7）

```
2026-04-01（周三）→ "1-3"
2026-04-08（周二）→ "2-2"
2026-04-25（周六）→ "4-6"
2026-04-29（周三）→ "1-3"  （月末映射回第1周）
```

`ScheduleKeyUtil.calcWeek` / `calcDay` / `isWeekday` / `isWeekend` 也用于订单 `schedule_mode` 判断和菜品列表的排期日期搜索，是跨模块复用的纯函数。

生成接口允许额外传入 `menuWeekNum + menuDayOfWeek` 覆盖菜单定位：

- `recordDate` 始终是实际排餐/配送日期
- `menuWeekNum + menuDayOfWeek` 仅用于查询 `meal_schedule_plan`
- 两个字段要么都不传，要么都传
- 不传时按 `recordDate` 通过 `ScheduleKeyUtil` 推导

### 餐次（meal_type）

| 类型 | code | 说明 |
|------|------|------|
| 早餐 | `BREAKFAST` | 参与排餐计划（生成 `meal_plan_customer` 记录），但不生成菜品明细 |
| 午餐 | `LUNCH` | |
| 晚餐 | `DINNER` | |

### 排餐状态（status）

| 状态 | code | 说明 |
|------|------|------|
| 生成中 | `GENERATING` | 事务进行中 |
| 成功 | `SUCCESS` | 所有客户排餐成功 |
| 失败 | `FAILED` | 部分或全部客户排餐失败 |

### 客户排餐状态（status）

| 状态 | 值 | 说明 |
|------|------|------|
| 失败 | `0` | 该客户本次排餐失败（记录在 `fail_reason`） |
| 成功 | `1` | 排餐成功 |

## 核心业务流程

### 1. 生成排餐计划

触发：`POST /api/meal-plan/generate`，参数为 `recordDate` + `mealType`，可选 `customerId`、`menuWeekNum`、`menuDayOfWeek`。

其中：

- `recordDate` 是实际排餐/配送日期
- `mealType` 支持 `BREAKFAST` / `LUNCH` / `DINNER`
- `menuWeekNum + menuDayOfWeek` 仅用于查询 `meal_schedule_plan`
- `menuWeekNum`、`menuDayOfWeek` 要么都不传，要么都传；不传时按 `recordDate` 推导菜单周次和菜单星期

**前置准备：并发锁控制**

对同日期同餐次使用 `ReentrantLock` 加锁，防止重复并发生成。事务提交后自动释放锁。

**步骤一：有效订单过滤**

对每个 `status=1`（进行中）的订单，检查：

```
remaining_count > 0
start_date <= recordDate <= end_date
meal_type 包含当前餐次（BREAKFAST/LUNCH/DINNER/ALL）
若 recordDate = start_date，则还需满足 start_meal_type <= 当前餐次：
  - BREAKFAST → 当天早餐/午餐/晚餐均可排
  - LUNCH     → 当天仅午餐/晚餐可排
  - DINNER    → 当天仅晚餐可排
schedule_mode 满足当前日期：
  - DAILY     → 每天
  - WEEKDAY   → 周一至周五
  - WEEKEND   → 周六、周日
  - SCHEDULE  → delivery_dates JSON 包含当天
    - 新格式: [{"date": "2026-04-01", "mealTypes": ["LUNCH", "DINNER"]}]
    - 旧格式: ["2026-04-01", "2026-04-02"]
    - 新格式支持按日期指定餐次
```

**步骤二：已排餐数量上限检查**

批量统计各订单当前餐次的有效排餐记录数（`countScheduledByOrderIds`），统计口径为 `meal_plan_customer.deleted=0` 且关联 `meal_plan.deleted=0`。若该数量 >= 订单对应餐次的餐数上限则跳过：
- BREAKFAST：上限为 `order.breakfastCount`
- LUNCH/DINNER：上限为 `order.lunchDinnerCount`

**步骤三：客户排除日期过滤**

检查客户档案中的排除日期配置 `exclude_dates`，如果该日期+餐次被排除，则跳过该订单。

**步骤四：候选菜加载（已迁移到 meal_schedule_plan）**

候选菜查询使用“菜单定位”而不是配送日期本身：

- 未传 `menuWeekNum`、`menuDayOfWeek` 时，按 `recordDate` 推导 `week_num + day_of_week`
- 传入 `menuWeekNum`、`menuDayOfWeek` 时，实际排餐日期仍是 `recordDate`，但普通菜品候选池改为按指定 `week_num + day_of_week` 查询
- 两个字段仅影响 `meal_schedule_plan` 查询，不影响订单生效判断、客户排除日期判断、排餐主记录日期等业务语义

候选菜由两部分合并而成，按 `dish.id` 去重：

1. **排期菜品**：从 `meal_schedule_plan` 表查 `week_num + day_of_week + meal_time` 匹配的记录，JOIN `dish` 取菜品详情。SQL 中用 `msp.dish_category` 覆盖 `d.dish_type`，保证候选池里每条菜的 `dishType` 与排期坑位一致（同一道菜在不同坑位可以扮演不同角色）。
2. **米饭菜品**：从 `dish` 主档查 `dish_type = RICE_TYPE` 且 `enabled = 1` 的记录。**米饭不依赖 `meal_schedule_plan`**，是全局通用菜池。

旧逻辑通过 `dish.schedule` JSON 字段过滤的方式已废弃。

**步骤五：按父套餐归类候选菜池（buildCandidateDishPool）**

对每个出现在订单里的父套餐，构造一个独立的 `dishType → List<Dish>` 候选 Map：

- 普通菜品（`MAIN/SIDE/SOUP/VEGETABLE`）：必须满足 `meal_packages` 包含当前父套餐编码或 ID 才纳入；
- `RICE_TYPE` 米饭：跳过套餐匹配检查，所有父套餐都共享同一池米饭；
- 进入候选池后，`RICE_TYPE` 被规整化为 `RICE`（`normalizeCandidateDishType`），下游选菜逻辑只看到 `RICE` 类型。

候选池按 `sort` + `id` 稳定排序，保证同一输入下选菜结果可预期。

**步骤六：过敏过滤（升级为三级匹配，2026-04-24）**

不再用 `customer.allergy_tags` 与 `dish.ingredients` 字符串做模糊匹配，而是：

1. 加载菜品 → 配料名集合：通过 `dish_ingredient_relation` JOIN `dish_ingredient` 获取每道菜对应的配料名集合；
2. 加载分类 → 配料映射：从 `dish_ingredient_category` 一次性加载「分类名 → 该分类下所有配料名集合」（一级和二级分类合并到同一个 Map，分类名重复时合并集合）；
3. 对每个客户过敏标签依次匹配：
   - **优先级 1**：与配料名精确相等 → 命中（如客户过敏"鸡肉"，菜含"鸡肉"）；
   - **优先级 2**：与某个分类名相等 → 展开该分类下所有配料名集合，菜的配料命中其中任意一个 → 命中（如客户过敏"禽肉"二级分类，菜含"鸡肉"）；
   - **优先级 3**：分类匹配同样适用一级分类（如客户过敏"肉类"一级分类，菜含"猪肉"也命中）。

匹配命中的过敏标签会回写到 `meal_plan_customer_item.allergy_reasons`（多个用逗号分隔），用于前端展示过敏原因。

**步骤七：按菜品数量配置选菜**

菜品数量配置从 `CustomerOrder` 直接读取（废弃 SubPackage 字段），每日每类固定提供 1 份：

| 菜品类型 | 每日固定提供 | 说明 |
|---------|-------------|------|
| 主菜 | 1 份 | 从 MAIN 类型选 |
| 副菜 | 1 份 | 从 SIDE 类型选 |
| 素菜 | 1 份 | 从 VEGETABLE 类型选 |
| 汤 | 0-1 份 | 从 SOUP 类型选（订单要求时才选） |
| 米饭 | 0-1 份 | 见下方"米饭选菜逻辑" |

**米饭选菜逻辑（pickRiceDish）：**

1. 候选池中筛 `dish_type = RICE`（已规整化的 `RICE_TYPE`）；
2. 若订单 `riceType` 有具体值且不等于 `默认`，先在候选中按 `name` 精确匹配；
3. 若订单 `riceType = 默认`，按未指定处理，直接从全部米饭候选中选一个可用的；
4. 匹配不到指定类型 → 回退到名称等于 `白米饭` 的默认米饭；
5. 仍未命中 → 回退到全部米饭候选中选一个可用的；
6. 选定的菜品在写入明细时 `dish_type` 统一记为 `RICE`。
7. 若订单 `riceType != 默认`，写入 `meal_plan_customer_item` 时统一标记 `is_replaced=1`，`replace_reason=客户选择替换`，`original_dish_id/original_dish_name` 固定记录排序后首个菜单米饭候选（即当天菜单基准米饭），不记录最终回退选中的米饭。

**补菜数量计算**：
```
补菜数量 = max(0, 订单配置数 - 每日固定1份)
```

| 订单配置数 | 每日固定 | 补菜数量 |
|-----------|---------|---------|
| 1 | 1 | 0 |
| 2 | 1 | 1 |
| 3 | 1 | 2 |

补菜数量记录在 `meal_plan_customer` 表的 `supplementary_*_count` 字段，供前端显示"加菜"提示。

**步骤八：订单级换菜规则应用**

在选菜完成后、保存菜品明细前，检查当前订单是否配置了换菜规则（`customer_order_replace_rule`）：

1. 批量加载所有有效订单的换菜规则，构建 `Map<orderId, Map<sourceDishId, rule>>`
2. 遍历已选菜品，若匹配到换菜规则：
   - 检查目标菜是否仍然启用且未被删除，若目标菜无效则跳过该规则（保留原菜）
   - 用目标菜替换原菜，记录 `original_dish_id`、`original_dish_name`、`original_dish_type`
   - 标记 `is_replaced=1`，`replace_reason=ORDER_RULE`
   - **跳过过敏/忌口过滤**：被替换的菜品不参与后续过敏匹配
3. 换菜规则优先级最高，覆盖之前的选菜结果（包括过敏替换的结果）

**步骤九：回退逻辑（暂未启用）**

当日菜品因过敏全部过滤时，暂不启用次选类型替换和次日菜品回退。

### 2. 幂等生成

同一天同一餐次重复调用时，采用**软删除旧记录 + 重新生成**策略：
1. 软删除 `meal_plan_customer_item`（`deleted=true`）
2. 软删除 `meal_plan_customer`
3. 软删除 `meal_plan`
4. 插入新记录

**指定客户生成模式**：如果传入 `customerId`，只删除该客户的排餐明细，保留主记录和其他客户的记录。

### 3. 地址自动选择

`GET /api/meal-plan/{mealPlanId}/customer-addresses` 根据排餐日期自动选择客户地址：

| 排餐日期 | 选择规则 |
|---------|---------|
| 周六、周日 | `WEEKEND` 地址 |
| 周一至周五 | `WORKDAY` 地址 |
| 指定类型地址不存在 | fallback 到 `DEFAULT` 地址 |
| 三个都没有 | 返回 null |

### 4. 删除排餐记录

- 按日期+餐次删除：`DELETE /api/meal-plan?recordDate=X&mealType=Y`
- 按 ID 批量删除：`DELETE /api/meal-plan/batch`，级联删除客户和菜品明细
- 按客户排餐 ID 删除：`DELETE /api/meal-plan/customers`

**重要**：删除排餐记录是软删除（`deleted=true`），且**不影响核销数据**（`meal_verification_log` 独立保留）。

## 数据模型

### meal_plan — 排餐计划主表

| 字段 | 类型 | 说明 |
|------|------|------|
| `record_date` | DATE | 实际排餐/配送日期 |
| `meal_type` | VARCHAR | 餐次：BREAKFAST / LUNCH / DINNER |
| `total_count` | INT | 参与排餐客户总数 |
| `success_count` | INT | 成功数 |
| `fail_count` | INT | 失败数 |
| `status` | VARCHAR | GENERATING / SUCCESS / FAILED |
| `generate_time` | DATETIME | 生成时间 |
| `deleted` | TINYINT | 软删除标记 |

### meal_plan_customer — 客户排餐表

| 字段 | 类型 | 说明 |
|------|------|------|
| `meal_plan_id` | BIGINT | 关联 meal_plan.id |
| `customer_id` | BIGINT | 客户ID |
| `order_id` | BIGINT | 关联订单ID |
| `parent_package_id` | BIGINT | 父套餐ID（冗余） |
| `child_package_id` | BIGINT | 子套餐ID（冗余） |
| `status` | TINYINT | 0=失败，1=成功 |
| `fail_reason` | VARCHAR | 失败原因 |
| `is_verified` | TINYINT | 是否已核销（0未核销，1已核销） |
| `verification_time` | DATETIME | 核销时间 |
| `verification_operator` | VARCHAR | 核销操作人 |
| `meat_required_count` | INT | 主菜需求数（从订单读取） |
| `veg_required_count` | INT | 素菜需求数（从订单读取） |
| `include_soup` | TINYINT | 是否含汤（0否，1是） |
| `include_rice` | TINYINT | 是否含米饭（0否，1是） |
| `supplementary_main_count` | INT | 补主菜数量 = max(0, 主菜需求数 - 1) |
| `supplementary_side_count` | INT | 补副菜数量 = max(0, 副菜需求数 - 1) |
| `supplementary_veg_count` | INT | 补素菜数量 = max(0, 素菜需求数 - 1) |
| `supplementary_rice_count` | INT | 补米饭数量 = max(0, 米饭需求数 - 1) |
| `supplementary_soup_count` | INT | 补汤数量 = max(0, 汤需求数 - 1) |

| `deleted` | TINYINT | 软删除标记 |

### meal_plan_customer_item — 客户菜品明细表

| 字段 | 类型 | 说明 |
|------|------|------|
| `meal_plan_customer_id` | BIGINT | 关联 meal_plan_customer.id |
| `dish_type` | VARCHAR | 菜品类型：MAIN / SIDE / VEGETABLE / SOUP / RICE；订单专用米饭档案使用 `RICE_TYPE`，排餐明细仍落 `RICE` |
| `dish_id` | BIGINT | 菜品ID |
| `dish_name` | VARCHAR | 菜品名称（冗余） |
| `seq` | INT | 顺序（1-99为正常菜品，100+为过敏菜品） |
| `is_replaced` | TINYINT | 是否被替换 |
| `original_dish_id` | BIGINT | 原菜品ID（替换时有值） |
| `original_dish_name` | VARCHAR | 原菜品名称 |
| `replace_reason` | VARCHAR | 替换原因：ALLERGY / EXCLUDED / 客户选择替换 / ORDER_RULE |
| `original_dish_type` | VARCHAR | 原菜品类型（被替换前的类型，跨类型替换时有值） |
| `is_allergy_filtered` | TINYINT | 是否因过敏被过滤 |
| `allergy_reasons` | VARCHAR | 触发的过敏标签（原始标签，可以是配料名/二级分类名/一级分类名），多个逗号分隔 |
| `deleted` | TINYINT | 软删除标记 |

## 与其他模块的关系

```
排餐管理（meal_plan）
  │
  ├── 订单管理（customer_order）
  │     ├── 查 status=1 + remaining_count>0 + 日期范围
  │     ├── 查 meal_type / schedule_mode / delivery_dates
  │     ├── 查 main_dish_count / side_dish_count / veg_count / rice_count / soup_count → 决定菜品槽位
  │     ├── 查 rice_type → 排餐选米饭时按类型优先匹配
  │     └── customer_order_replace_rule → 排餐时自动应用订单级换菜规则（ORDER_RULE）
  │
  ├── 套餐管理（sub_package）
  │     └── 查父套餐配置（废弃字段，菜品数量从订单直接读取）
  │
  ├── 配菜管理
  │     ├── meal_schedule_plan → 周-星期-餐次-坑位 取候选菜（普通坑位）
  │     ├── dish (RICE_TYPE)   → 全局米饭池（不依赖 meal_schedule_plan）
  │     ├── dish_ingredient_relation → 加载每道候选菜的配料名集合
  │     └── dish_ingredient_category → 三级过敏词展开（一级/二级分类 → 配料名集合）
  │
  ├── 客户管理（customer_profile）
  │     ├── 查 allergy_tags → 三级过敏过滤（配料名 / 二级分类 / 一级分类）
  │     └── 查 exclude_dates → 排除日期过滤
  │
  └── 核销管理（meal_verification_log）
        ├── 核销操作标记 is_verified / verification_time
        └── 排餐记录删除不影响核销日志
```

## 首次成功排餐标记

- **标记维度**：`order_id + meal_type`
- 仅成功排餐（`meal_plan_customer.status = 1`）参与首次判定
- 失败排餐不占用首次资格
- 左侧编号区展示时，首次客户排在前面，组内保持原顺序
- **标记页面范围**：排餐计划管理页、打印页
- **判定逻辑**：后端 SQL 使用 `NOT EXISTS` 模式，查询同一 `order_id + meal_type` 下是否有更早的成功记录
